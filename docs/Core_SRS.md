## 0. 설계 원칙

이 문서에서 물류 작업 시스템을 이하 Core로 표기한다. 이 챕터는 Core 시스템 운영의 핵심 원칙을 정의한다.

### 0.1 Core는 공정 내 Unit 이동 경로를 추적하는 시스템이다

Core의 실제 관리 범위는 물류 AMMR, WIP, CNC 작업대 간 Unit 이동으로 한정된다. 제품 생산 이력, 품질 추적, 작업실적 등은 Core가 자동으로 추적하는 영역이 아니며 GM/SM 등 외부 시스템의 영역이다. 단, 작업실적은 사용자가 Dashboard에서 수동 입력하면 Core가 기록한다.

**※ 이 원칙이 필요한 이유**

문서를 처음 보는 사람이 Unit ID, 공정 이력, GM 연동 등을 보고 제품 생산 관리 시스템으로 오해할 수 있다. Core가 관리하는 것은 어떤 Unit이 어느 공정 구간을 거쳤는지에 대한 이동 경로이며, 각 공정에서 어떤 작업이 이루어졌는지, 제품의 품질이나 실적은 어떠한지는 Core가 알지 못한다.

**※ 현재 한계가 생긴 배경**

사람과 협업하는 환경과 WIP이 없는 공정이 공존하는 환경상 Unit의 이동을 추적할 수 없는 한계가 있다. 공정 추적은 이동 자동화만으로 가능하나, 제품 추적은 작업 자동화 또는 실적 처리 자동화가 추가로 갖춰져야 한다. 이는 현재 Core의 범위 밖이다.

### 0.2 Core가 본 것만 기록한다

이 시스템의 가장 중요한 설계 원칙은 "Core가 직접 확인한 데이터만 기록한다"는 것이다. Slot Sensor 신호, AMMR Pickup·Dropoff 보고, QR Scan 등 Core가 실제로 감지한 정보만 기록하며, 감지하지 못한 구간은 이유와 무관하게 null로 처리한다.

**※ 이 원칙이 필요한 이유**

신뢰성: Core가 직접 확인한 데이터만 기록하면 잘못된 정보가 시스템에 쌓이지 않는다. 추정이나 간접 정보를 기록하다 보면 나중에 데이터 정합성 문제가 생긴다.

확장성: 나중에 Sensor나 장비가 추가되면 Core가 보는 범위가 늘어나고, 그때 자연스럽게 기록이 늘어나는 구조다.

유지보수: 원칙이 명확하면 나중에 다른 개발자나 담당자가 시스템을 이해할 때도 혼란이 없다.

**※ 적용 범위**

"Core가 본 것만 기록한다"는 원칙은 WIP을 거치지 않는 공정, 수동 처리 구간, 자동 추적 불가 공정 모두에 동일하게 적용된다. null은 오류가 아니라 Core가 해당 구간을 감지하지 못했음을 의미한다.

**※ Slot 점유와 Unit 식별값 확정 기준**

Slot 점유는 Slot Sensor로 확정한다. Slot에 적재된 Unit의 식별값을 확정하는 기준은 QR Scan이며, AMMR 이동 구간처럼 QR Scan이 불가능한 구간은 추정값으로 관리한다.

Unit 식별값은 출처에 따라 두 등급으로 운영된다 — **확정 등급**은 QR 인식·Dashboard 수동 입력 시점에 부여되며, **추정 등급**은 AMMR Pickup·Dropoff 동작 시점에 부여된다. 두 등급은 후속 운영에 동일 권위로 사용되며, 추정 등급 상태로도 정상 작업 진행이 가능하다. 추정값은 다른 QR Scan 시점에 실제값으로 보정된다.

WIP Slot과 AMMR 자체 Slot 모두 Slot Sensor와 Vision Sensor가 이중으로 감지한다. 단, AMMR의 Manipulator Vision Sensor는 QR 인식이 불가하므로 AMMR Slot Unit 식별값은 추정 등급으로 운영된다.

> **[예시] Core 원칙 적용 — 공정 기록 예시**
>
> 상황: U-045가 CNC → 세척 이동 중 현장 작업자가 WIP을 거치지 않고 직접 세척 구역으로 가져다 놓음.
>
> 공정 / 입고 시간 / 작업 시작 / 작업 종료 / 출고 시간 / 수량
>
> ─────────────────────────────────────────
>
> CNC 2공정 / 11:45 / 11:46 (CNC 공정 AMMR) / 13:20 (CNC 공정 AMMR) / 13:25 / 16개
>
> 세척 / null / null / null / null / null  ← (WIP 미경유)
>
> ─────────────────────────────────────────
>
> → null은 오류가 아니다. Core가 감지하지 못한 구간은 null로 표시하는 것이 정상이다.

### 0.3 안전망 요구

이 시스템은 다음 상황에서 해당 Slot 또는 작업의 신규 진행을 일시 차단하고, 원인이 해소되면 자동 해제할 수 있어야 한다.

- **외부 측 상태 미인지**: 외부 시스템 또는 AMMR과의 통신이 두절되어 외부 측 상태를 알 수 없는 동안 신규 작업을 강행하면, 통신이 끊긴 사이 외부에서 발생한 상태 변화를 반영하지 못해 잘못된 위치의 Pickup·Dropoff 등 사고가 발생할 수 있다. 통신이 끊긴 영역의 신규 작업은 자동 차단한다.
- **물리적 안전**: AMMR HW가 장애를 보고했거나 도착 시점에 목적지가 만재이거나 물리 충돌 상황이 감지된 경우, 추가 작업을 강행하면 사고 위험이 있다. 해당 영역의 신규 작업은 자동 차단한다.
- **데이터 정합성 보호**: Slot Sensor와 Unit 식별값이 일치하지 않는 Slot은 추정으로 임의 채우지 않고 사용 보류한다. Unit이 갈 수 없는 공정이나 수행할 수 없는 작업으로 들어온 경우 등 Core가 유효하지 않다고 판정하는 상황도 해당 Slot 또는 작업을 사용 보류한다.
- **운영자 인지**: 모든 보류 상태는 Dashboard에 표시하여 운영자가 확인할 수 있어야 한다.
- **자동 해제**: 원인이 해소되면 별도 사용자 명령 없이 자동으로 해제된다. 자동 해제가 불가능한 경우에는 운영자가 수동으로 해제한다.

이 요구는 운영 데이터 신뢰성과 물리 자산(WIP·CNC 작업대·AMMR·Unit) 안전을 위한 시스템 차원 안전망이다.

## 1. 공장 및 시스템 개요

이 챕터는 공장 시설과 Core 시스템 구성요소를 정의한다. 알루미늄 가공품을 생산하는 메탈 가공 공장의 사내 물류를 자동화하는 시스템이며, 모든 공정 WIP과 AMMR은 사내 내부망으로 연결된다.

**※ 모든 Slot(WIP·AMMR·CNC 작업대)은 1 Unit만 담는다.**

**※ 일반 공정 WIP·되담기 WIP은 통합 Slot로 운영하며, 입고·출고 구분은 QR Scan 시 함께 전달되는 구분값으로 판정한다. CNC 작업대는 SM 측 입고/출고 Slot Sensor 신호를 사용한다.**

### 1.1 시스템 구성요소

**※ 로봇 / 장비 / 외부 시스템**

| 이름                     | 역할                                                                       |
|--------------------------|----------------------------------------------------------------------------|
| GM 시스템 (외부)         | 제품 정보 / Recipe 제공 (REST API, 폴링 주기·Endpoint 등 **변동 가능**) |
| SM 시스템 (외부)         | 장비 상태(Idle 여부) 및 CNC 작업대 입고/출고 Slot Sensor 신호(On/Off) 제공.<br><br>Core가 직접 REST API 폴링하여 수신 (폴링 주기·Endpoint 등 **변동 가능**). |
| Core                     | 이 시스템 메인 서버. 모든 WIP·CNC 작업대·물류 AMMR 간 Unit 이동을 추적·관리하고, 외부 시스템(GM/SM) 연동·물류 AMMR 명령·WIP 중계 프로그램 통신·Dashboard 인터페이스를 담당한다. |
| 되담기 WIP의 이동 로봇   | 되담기 WIP 내부에서 통합 Slot ↔︎ 스테이션 간 Unit 이동 담당 (1회 1 Unit).<br><br>되담기 WIP 중계 프로그램 제어 하에 Slot Sensor 감지 시 해당 위치로 이동하여 QR 인식 담당. Vision Sensor로 QR을 인식한다. |
| 되담기 WIP의 되담기 로봇 | 스테이션에서 공정 Tray의 제품을 적층 Tray로 옮기는 실제 되담기 작업 수행. Vision Sensor로 적층 Tray의 제품 유무를 감지한다. |
| 물류 AMMR                | 공정 구간 간 Unit 외부 이동 (최대 6 Unit 적재). 현재 2대 운영.<br><br>※ AMMR이 현재 적재 중인 Unit 목록은 추정값이다. Pickup 동작 시 출발 WIP의 QR Scan 정보를 기반으로 추정되며, 후속 QR Scan 시점에 실제값으로 확정된다.<br><br>※ AMMR 자체 Slot 점유는 Slot Sensor와 Manipulator Vision Sensor가 이중으로 감지한다. 단, Manipulator Vision Sensor는 QR 인식이 불가하므로 Unit 식별값 갱신은 별도 QR Scan 시점까지 추정 등급으로 운영된다. |
| 충전 스테이션            | 물류 AMMR이 충전 시 도킹하는 물리 위치. 도킹·충전·이탈 등 물리적 수행 세부와 충전 중단 결정은 AMMR HW가 자체 처리한다. Core는 Charge Job 수행 결과 보고만 수신하며 충전 스테이션과 직접 통신하지 않는다. |
| CNC 공정 AMMR            | CNC 공정 작업 수행. 작업 시작·종료 보고는 별도 강제 사항이 아니며, 보고할 경우 QR Scan으로 보고한다. 총 4대 운영.<br><br>단, CNC 공정 AMMR이 없는 라인은 사람이 수동으로 처리. |

**※ WIP (Work In Process)**

| 이름                      | 역할                  |
|---------------------------|-----------------------|
| 일반 공정 WIP (통합 Slot) | 공정의 Unit 대기 Slot. 입고·출고는 QR Scan 구분값으로 판정 |
| 되담기 WIP (통합 Slot)    | 통합 Slot에 공정 Tray Unit·적층 Tray Unit이 함께 놓여 Pair가 갖춰지면 스테이션으로 이동 → 되담기 수행<br><br>완료된 적층 Tray Unit + 빈 공정 Tray Unit은 통합 Slot에 남아 물류 AMMR이 다음 공정으로 이동.<br><br>입고·출고는 QR Scan 구분값으로 판정.<br><br>되담기 허브 역할 + 출고 후 다음 공정 이동 (일반 WIP 구조적 겸행). 만재 Unit 임시 보관 중인 AMMR은 새 Job 목적지 후보에서 되담기 WIP 제외 |

**※ 사람**

| 생산관리 담당자 | Unit별 Recipe 관리 (공정 추가/삭제) / Tray 되담기 공정 수동 추가 / 시스템 전역 설정 변경 |

**※ 비고: WIP에 이미 입고된 Unit의 이동 배치(교체 작업 등)는 이 시스템에서 제외한다. 교체가 필요한 경우 사람이 직접 처리한다.**

## 3. Unit / Tray / 제품 구성

이 챕터는 제품·Tray·Unit의 구성 단위와 ID 정책을 정의한다.

### 3.1 구성 단위

**※ 구성 단위별 정의**

| 단위        | 구성                | 비고                       |
|-------------|---------------------|----------------------------|
| 제품 (Item) | 알루미늄 가공품 1개 | GM 시스템의 수량 단위      |
| Tray        | 제품 4개 안착       | 뚜껑 역할 Tray는 제품 없음 |
| Unit        | Tray N개 (최대 5단) | 최소 2단(뚜껑 1개 + 제품 Tray 최소 1단) ~ 최대 5단(뚜껑 1개 + 제품 Tray 4단). 최대 제품 16개. 수량에 따라 가변 구성 |

**※ 용어 보강**

"제품(Item)"은 개별 가공품을 가리킨다. "제품 정보"는 GM에서 받아오는 Metadata 묶음(제품 종류·수량·납기일·Recipe 등)을 의미한다.

예시: GM 100개 등록 → 5단 Unit 6개(96개) + 2단 Unit 1개(4개) = 총 7개 Unit

### 3.2 Tray 종류

**※ Tray 종류별 특성**

| 구분      | 이름      | QR 방식                    | 비고                                                             |
|-----------|-----------|----------------------------|------------------------------------------------------------------|
| 되담기 전 | 공정 Tray | QR 스티커 (QR + 제품 목록) | 사람이 육안으로 제품 확인 가능                                   |
| 되담기 후 | 적층 Tray | QR 각인만                  | 스티커 없음. QR로만 식별. 필요 시 Dashboard에서 식별 스티커 출력 |

**※ 공정 Tray Unit 구성**

| 단계 | 처리                                                                          |
|------|-------------------------------------------------------------------------------|
| 1    | GM에서 신규 투입코드 수신 → Dashboard에서 실제 입고 수량 확인                |
| 2    | 실제 입고된 수량 기준으로 공정 Tray Unit 구성 (수량 입력 → 시스템 제안 확인) |
| 3    | QR 스티커 출력 → 실물 공정 Tray에 부착                                       |
| 4    | 공정 투입 시작                                                                |

**※ 적층 Tray Unit 구성**

| 단계 | 처리                                            |
|------|-------------------------------------------------|
| 1    | 발주처에서 적층 Tray 납품 → 물류실 적치        |
| 2    | 작업자가 적층 Tray를 한 단씩 각인 QR로 스캔해 제품 정보를 매핑 |
| 3    | 적층 Tray를 임의 단수로 되담기 WIP에 투입 |
| 4    | 되담기 WIP이 짝 공정 Tray Unit에 맞춰 적층 Tray Unit 구성을 완성 |

### 3.3 Tray 되담기 규칙

**※ Tray 되담기란?**

공정 Tray에 담긴 제품을 적층 Tray로 옮기는 공정이다. Recipe에 명시된 필수 공정으로, 선택적으로 수행하는 것이 아니다. 공정 Tray는 QR 스티커가 붙어있어 사람이 내용물을 육안으로 확인할 수 있다. 반면 적층 Tray는 QR 각인만 있어 사람이 내용물을 직접 식별할 수 없으며, 시스템(Dashboard)을 통해서만 확인 가능하다.

> **[예시] 되담기 전/후 비교**
>
> 되담기 전 (공정 Tray): Tray에 제품 4개씩 안착, QR 스티커로 내용물 확인 가능.
>
> 되담기 후 (적층 Tray): 제품이 적층 Tray에 옮겨짐. QR 각인으로만 식별 (스티커 없음).

되담기는 되담기 WIP 전용 로봇이 담당한다. 단, 현장 작업자가 직접 수동 되담기를 처리한 경우, Dashboard에서 작업실적 수동 입력으로 공정 완료를 기록한다.

되담기는 Recipe에 'Tray 되담기' 공정으로 명시적으로 삽입되어 관리. GM Recipe 수신 후 담당자가 시스템에서 Unit별로 명시적으로 추가.

물류 AMMR이 되담기 WIP에 Unit을 넣는 경우는 Recipe에 'Tray 되담기' 공정이 명시된 경우에만 가능하다.

되담기 공정이 명시되지 않은 Unit이 되담기 WIP에 투입된 경우 정합성 이상 처리.

### 3.4 Unit ID 정책

**※ Unit 식별값 — 라벨과 QR**

Unit은 여러 Tray로 구성되며, 식별은 두 층위로 나뉜다.
- **라벨** (`투입코드_유닛번호` 형식, 예: `26SF03002-001_001`) — 어떤 투입코드의 몇 번 Unit인지, 즉 제품정보 수준의 정보를 담는 사람 읽기용 표기다.
- **QR 식별값** — Tray마다 유니크한 시스템 권위 식별값으로, 공정 Tray는 스티커 QR·적층 Tray는 각인 QR에 인코딩된다. 실물 Tray 수준의 유니크 식별자로, QR Scan 시점에 Core가 받는다.

라벨만으로는 제품정보까지만 알 수 있고, 실물 Tray를 유니크하게 특정하는 것은 QR 식별값이다. 이 QR 식별값으로 Tray가 어느 Unit에 속하는지가 결정되며, 확정 등급 Unit 식별값은 이를 기준으로 한다.

**※ Unit ID 관리 시점별 처리**

| 시점           | 처리                                                  |
|----------------|-------------------------------------------------------|
| 최초 사내 입고 | 담당자가 실제 입고 수량 기준으로 Dashboard에서 Unit 구성 → QR 스티커 출력 → 실물 부착 |
| 외주 복귀 시   | 이전 Unit ID 완전 폐기 → 실제 복귀 수량 기준으로 신규 Unit 구성 → 신규 ID 재발급 (새 QR 부착) |
| 적층 Tray 되담기 | 각 Tray를 각인 QR로 식별. 작업자가 제품 정보를 매핑하고, 되담기 WIP이 짝 공정 Tray Unit에 맞춰 Unit 구성을 완성 |
| 이력 추적      | Unit 단위로만 추적. 외주 구간은 외주 배치 단위로 연결 |

> **[예시] Unit 구성 흐름 — 제품 100개 중 20개가 먼저 입고된 경우**
>
> 상황: 투입코드에 제품 100개 등록. 1차로 20개가 먼저 사내에 입고됨.
>
> [1차 입고 - 20개]
>
> ① 담당자가 Dashboard → Unit 구성 탭에서 수량 "20" 입력
>
> ② 시스템이 Unit 구성 제안: 5단 Unit 1개(16개) + 2단 Unit 1개(4개) = 총 2개 Unit
>
> ③ 담당자 확인 후 QR 발행 → U-001, U-002 생성 → 실물에 스티커 부착
>
> ④ 공정 투입 시작
>
> [2차 입고 - 80개]
>
> ① 담당자가 Dashboard에서 수량 "80" 입력
>
> ② 시스템 제안: 5단 Unit 5개(80개) = 총 5개 Unit → QR 발행 → U-003~U-007 생성
>
> → 투입코드의 제품 전체(100개)에 대해 미리 Unit을 만들지 않고, 실제 입고된 수량 기준으로만 Unit을 구성한다.
>
> **[예시] 외주 복귀 후 Unit 구성 — 복귀 수량이 달라진 경우**
>
> 상황: U-001(16개), U-002(4개) 총 20개가 외주로 나갔다가, 복귀 시 18개만 돌아온 경우.
>
> ① 담당자가 Dashboard → 외주 관리 탭에서 복귀 수량 "18" 입력
>
> ② 시스템 제안: 5단 Unit 1개(16개) + 2단 Unit 1개(2개) = 총 2개 Unit
>
> ③ 기존 U-001, U-002 폐기 → 신규 U-089, U-090 생성
>
> ④ 외주 배치 이력에 출발 수량 20 / 복귀 수량 18 기록
>
> → 수량 불일치(20개 출발, 18개 복귀)가 발생해도 시스템은 있는 그대로 기록한다.

### 4.1 CNC 작업대 동작 방식

Core 입장에서 CNC 작업대 동작 방식은 CNC 공정 AMMR 유무와 무관하게 동일하다. AMMR이 없는 라인은 사람이 수동으로 작업을 수행한다.

**※ CNC 작업대 항목별 정의**

- **작업대 구성**: 장비별 입고 1 Slot / 출고 1 Slot
- **입고·출고 감지**: SM 폴링 → 작업대 Slot Sensor 신호로 물류 AMMR 입고·출고 시점 판단
- **작업 시작·종료 기록**: 외부 QR Scan으로 보고 시 기록 (보고 없으면 미기록)
- **Unit 확정**: QR Scan 시점에 확정
- **장비 Jig Setup**: SM 시스템에서 수신
- **Jig Setup 미완료 시**: 이전 공정에서 대기 → Jig Setup 수신 즉시 Move Job 생성
- **작업대 우선순위**: DB에서 정의. 같은 공정 내 여러 장비 중 우선순위 기준 배정

> **[예시] CNC 1공정 (QR 보고 없음) vs CNC 2공정 (QR 보고 있는 경우) 기록 비교**
>
> ▶ CNC 1공정 (QR 보고 없음):
>
> 작업대 입고: 10:05 (Slot Sensor) / 작업 시작: null / 작업 종료: null / 작업대 출고: 11:30 (Slot Sensor)
>
> ▶ CNC 2공정 (작업 시작·종료 QR 보고 있는 경우):
>
> 작업대 입고: 11:45 (Slot Sensor) / 작업 시작: 11:46 (QR Scan) / 작업 종료: 13:20 (QR Scan) / 작업대 출고: 13:25 (Slot Sensor)
>
> → 두 경우 모두 작업대 Slot Sensor는 물류 AMMR의 입고·출고 시점만 판단한다. 작업 시작·종료는 외부 QR 보고가 있을 때만 기록되며, 보고는 외부 측 자율로 강제 사항이 아니다.

## 6. 되담기 WIP 동작 로직

이 챕터는 되담기 WIP 중계 프로그램이 수행하는 입고 판단 케이스와 출고 Trigger를 정의한다.

### 6.1 입고 판단 케이스

**※ 입고 판단 케이스별 처리**

되담기 WIP 중계 프로그램은 Core 회신 Unit 정보(Tray 형태 포함)와 자기 Slot Sensor 상태를 결합해 Pair 판단을 자체 수행한다. 적층 Tray는 제품 정보가 매핑된 상태로 들어오며, 되담기 WIP이 제품 정보로 짝 공정 Tray Unit을 식별해 적층 Tray Unit 구성을 완성한 뒤 Pair 판단·되담기를 진행한다.

**Case 1** — 공정 Tray 입고 / 현재 Slot 상태: 적층 Tray 없음 (Pair 미완성)
- 처리: 되담기 불가 → Slot에서 Pair 대기. Pair가 갖춰질 때까지 해당 Slot 점유 상태 유지, 새 입고 작업 미생성.

**Case 2** — 공정 Tray 입고 / 현재 Slot 상태: 적층 Tray 있음 (Pair 완성)
- 처리: 되담기 진행 → 완료 후 출고 처리.

**Case 3** — 적층 Tray 입고 / 현재 Slot 상태: 공정 Tray 없음 (Pair 미완성)
- 처리: 되담기 불가 → Slot에서 Pair 대기. Pair가 갖춰질 때까지 해당 Slot 점유 상태 유지, 새 입고 작업 미생성.

**Case 4** — 적층 Tray 입고 / 현재 Slot 상태: 공정 Tray 있음 (Pair 완성)
- 처리: 되담기 진행 → 완료 후 출고 처리.

> **[예시] Slot Pair 대기 및 이동 로봇 대기 상황**
>
> 상황: 공정 Tray U-045가 되담기 WIP 통합 Slot에 들어왔는데, 아직 적층 Tray가 없는 상태 (Case 1)
>
> ① U-045 공정 Tray 입고 → 되담기 WIP 중계 프로그램이 "적층 Tray 없음" 확인 → Case 1 판정
>
> ② Slot은 U-045가 점유한 채로 대기 상태 유지
>
> ③ 이동 로봇은 적층 Tray가 들어올 때까지 대기
>
> ④ 적층 Tray 입고 → Pair 완성 → 이동 로봇이 스테이션으로 이동 → 되담기 진행
>
> Slot이 꽉 찬 상태에서 새 Unit이 들어오려 할 때:
>
> → 이동 로봇이 Vision Sensor로 통합 Slot 빈 자리 확인 → 없으면 스테이션에 다시 내려놓고 빈 자리 생길 때까지 대기.

### 6.2 출고 Trigger

**※ 출고 Trigger 조건**

| Trigger   | 조건           | 처리 |
|----------|----------------|------|
| Trigger A | 되담기 완료 후 | 이동 로봇이 완료된 적층 Tray Unit을 통합 Slot 빈 자리로 이동 → 물류 AMMR이 다음 공정으로 이동 |

**※ 되담기 WIP 통합 Slot은 되담기 완료 Unit 외에도 CNC 가공 결과 Unit이 다음 목적지 WIP 만재 시 임시 보관 위치로 사용된다.**

## 7. 물류 AMMR 운영 정책

이 챕터는 물류 AMMR의 외부 보고 상태값과 운영 정책을 정의한다.

### 7.1 AMMR 상태값

이 절은 AMMR이 외부 보고하는 운영 상태값(AMMR HW 상태·Slot Sensor·Manipulator Vision Sensor·위치·BMS)을 정의한다. Core가 일괄 보고 재전송을 요청하면 AMMR은 초기 연결 시와 같은 일괄 보고를 재발행한다.

**※ AMMR HW 상태**

AMMR HW는 초기 연결 시 1회 일괄 보고하고, 이후 상태 전이 시점마다 Event로 보고한다. Job 수행 중에는 수행 중인 Job 동작을, Job이 없을 때는 운영 상태를 보고한다.

- **Move**: Move Job 수행 중 (이동)
- **Pickup**: Pickup Job 수행 중
- **Dropoff**: Dropoff Job 수행 중
- **Charge**: Charge Job 수행 중 (충전존 이동·도킹)
- **대기**: 작업 배정 대기 중 (충전 완료 후 또는 작업 종료 후)
- **충전 중**: 충전존 도킹 후 충전 중
- **저전력**: HW 자체 임계 통과 보고 — Battery 20% 이하 진입 시. 이 상태 중 진행 중 단위 Job 종료 후 AMMR HW가 자율적으로 충전 스테이션으로 이동·도킹한다. 도킹 시점에 '충전 중'으로 전이하며, 80% 완충 시점에 '대기'로 전이.
- **장애**: 고장 또는 수리 중. Dashboard 경고 표시

**※ Slot Sensor**

AMMR HW는 초기 연결 시 6 Slot 일괄 보고하고, 이후 사람 개입 등 외부 원인으로 On↔Off 전이 시점에 해당 Slot 1개 보고한다.

| 항목            | 출처         | 설명                  |
|-----------------|--------------|-----------------------|
| Slot 점유 (6개) | AMMR HW 보고 | 각 Slot의 On/Off 상태 |

**※ 위치**

AMMR HW는 1초 주기로 위치 정보를 스트리밍으로 보고한다 (HW 설정 가변).

| 항목      | 출처         | 설명                             |
|-----------|--------------|----------------------------------|
| (x, y, a) | AMMR HW 보고 | 현재 위치 좌표 + 방향각 (연속값) |

**※ BMS**

BMS는 5초 주기로 다음 정보를 스트리밍으로 보고한다 (HW 설정 가변).

| 항목          | 출처     | 설명                                   |
|---------------|----------|----------------------------------------|
| AMMR ID       | BMS 보고 | 이 BMS가 부착된 AMMR의 식별값          |
| Battery ID     | BMS 보고 | Battery 팩의 식별값                     |
| Battery        | BMS 보고 | 현재 충전 상태 (%, 연속값)             |
| 전류          | BMS 보고 | 현재 전류 (연속값)                     |
| 온도          | BMS 보고 | 현재 온도 (연속값)                     |
| 전압          | BMS 보고 | 현재 전압 (연속값)                     |
| BMU 오류 여부 | BMS 보고 | Battery Management Unit 오류 발생 여부 |

**※ Manipulator Vision Sensor**

AMMR HW에 부착된 Manipulator Vision Sensor는 AMMR 자체 Slot의 Unit 존재 감지 및 Vision 기반 충돌 감지를 수행한다. QR 인식 기능은 없으므로 Unit 식별값 갱신에는 별도 QR Scan이 필요하다.

| **항목**            | **출처**                  | **설명**                                                |
|---------------------|---------------------------|---------------------------------------------------------|
| Slot Unit 존재 감지 | Manipulator Vision Sensor | 6 Slot 각각의 Unit 존재 여부 (Slot Sensor와 이중 감지) |
| Vision 충돌 감지    | Manipulator Vision Sensor | AMMR 작업 수행 중 Vision 기반 충돌 감지 시 보고         |

### 7.2 AMMR 태블릿 설정값

AMMR 운영에 필요한 다음 설정값은 AMMR 태블릿에서 운영자가 변경하며, AMMR 내부에 저장되어 재시작 후에도 유지된다. 이 설정값은 Core 측에 별도 저장되지 않는다.

**※ 설정 항목**

| 항목 | 초기값 | 비고 |
|------|--------|------|
| 충전 스테이션 번호 | - | AMMR이 충전 시 이동할 스테이션 번호 (자율 충전·Core Charge 지시 공통) |
| Battery 저전력 임계치 | 20% | §7.1 '저전력' 진입 판단 기준값 |
| Battery 대기 임계치 | 80% | §7.1 '저전력' → '대기' 전이 판단 기준값 |
| 시스템 Host IP | - | AMMR이 Core에 접속할 IP 주소 |
| 시스템 포트 | - | Core 접속 포트 |
| MQTT 접속 ID | - | Core MQTT Broker 접속 자격증명 |
| MQTT 접속 비밀번호 | - | Core MQTT Broker 접속 자격증명 |

**※ 시스템 접속 정보 변경 제약**

시스템 접속 정보(Host IP·포트·MQTT 자격증명)는 Core 연결이 끊긴 상태에서만 변경 가능하다. 운영 중 변경이 통신에 영향을 주는 것을 피하기 위함이다.

### 7.3 AMMR Job 배정 정책

**※ Pickup 우선순위**

물류 AMMR은 Pickup할 Unit이 여럿인 경우 우선순위가 높은 Unit을 먼저 Pickup한다.

**※ 신규 Job 배정 조건**

AMMR HW 상태가 '저전력' 또는 '충전 중'인 동안 Core는 신규 Job 배정을 차단한다. 단, 충전 중 Battery가 충분히 회복되면 AMMR HW의 충전 완료를 기다리지 않고 이 AMMR에 신규 Job을 배정하거나 적재 중 Unit의 진행 Job을 이어간다.

### 7.4 AMMR Job 지시 수신 확인 정책

Core는 Job 지시 후 AMMR이 보고하는 Job 지시 수신 확인 보고를 수신한다. 이 보고는 AMMR HW가 Job 지시를 수신한 직후 송신된다.

**※ 수신 확인 미수신 시 처리**

Core가 Job 지시 후 3초 안에 수신 확인을 받지 못하면 해당 AMMR을 단절로 간주하여 AMMR HW 단절과 동일하게 처리한다. 이 임계값은 운영 결과 및 AMMR 통신 지연 특성에 따라 조정 가능하다.

### 7.5 AMMR Job 수행 전 Slot 상태 확인 원칙

물류 AMMR은 Pickup·Dropoff 동작을 수행하기 직전에 대상 Slot의 상태를 재확인한다. 정합성 이상 감지 시 해당 Slot 정합성 이상 처리 + 보안 Log + Dashboard 경고가 발생한다.

**※ 재확인이 필요한 이유**

사람이 현장에서 자동 인지 경로(QR Scan·AMMR 동작 보고)를 거치지 않고 Slot의 Unit을 임의로 이동·교체·회수할 수 있다. Core는 이 경우의 사람 개입을 실시간 인지할 수 없으며, AMMR Job 직전 재확인이 이 한계의 안전 장치가 된다.

**※ 정합성 회복 절차**

Slot 정합성 이상은 자동 해제되지 않는다. 담당자가 Dashboard에서 정합성 회복 조치를 수행하면 정합성이 회복되어 자동 해제된다. 이 조치는 보안 Log 기록 대상이다.

**※ AMMR Slot 적재 정보 일괄 재로드**

AMMR 자체 Slot의 정합성 회복은 적재 정보 일괄 재로드로 수행한다. 담당자가 AMMR 태블릿에서 Slot별 Unit 식별값을 입력한 뒤 일괄 재로드하면, AMMR은 6개 Slot 전체의 Sensor 상태와 입력된 식별값을 Core에 한 번에 전달한다. Core는 Slot별로 상태를 확정·검사해 결과(정상 / 빈 Slot / 정합성 이상)와 해당 Unit 정보·Job 배정을 Slot별로 일괄 응답하며, 정합이 회복된 Slot은 이상이 자동 해제된다. 주로 통신 단절 중 사람이 AMMR Slot의 Unit을 임의로 다룬 뒤 재연결 시점에 상태를 다시 맞추는 데 쓴다. 이 조치는 보안 Log 기록 대상이다.

### 7.6 AMMR 적재 Unit 추정값 정책

AMMR이 현재 싣고 있는 Unit 목록은 추정값이며, Pickup 동작 시 출발 WIP의 QR Scan 정보를 기반으로 추정된다. Dropoff 시점에 도착 WIP의 QR Scan 정보로 보정되며, 보정이 없으면 후속 QR Scan 시점까지 추정 등급이 유지된다.

**※ 추정값 오보정 시나리오**

| 시나리오       | 발생 원인                                  | 처리                                             |
|----------------|--------------------------------------------|--------------------------------------------------|
| 적재 Unit 교체 | 사람이 AMMR 위의 Unit을 다른 Unit으로 바꿈 | 다음 QR Scan 시점에 정합성 회복 + 보안 Log 기록 |
| 적재 Unit 소실 | 사람이 AMMR 위의 Unit을 가져감             | AMMR Dropoff 시점에 정합성 이상 감지 → 해당 Slot 정합성 이상 처리 + 보안 Log + Dashboard 경고 |

**※ 추정값도 재판단 및 우선순위 계산에 활용된다. 추정값은 다음 QR Scan 시점에 실제값으로 보정되므로 시스템 정합성은 유지된다.**

### 7.7 AMMR 장애 대응 정책

이 절의 '장애'는 AMMR HW 상태 '장애' 또는 AMMR HW 단절로 운영 불가 묶음을 가리킨다.

**※ AMMR 장애 상황별 처리**

| 상황      | 처리                                                                 |
|-----------|----------------------------------------------------------------------|
| 장애 발생 | Dashboard 경고 / 해당 AMMR 운영 불가 표시                            |
| Unit 처리 | 사람이 직접 꺼내서 다음 or 이전 공정에 배치 → QR Scan으로 위치 기록 |
| 1대 운영  | 처리 속도 저하 경고 표시                                             |
| 0대 운영  | Dashboard 전체 정지 경고 / 사람이 수동 대응                          |

**※ AMMR Slot 정합성 이상이 발생한 경우 해당 Slot을 제외한 나머지 Slot으로는 정상 운영을 계속한다.**

**※ 운영자에 의한 명시적 연결 해제·재연결**

운영자가 AMMR 태블릿에서 시스템 연결을 명시적으로 해제·재연결하는 동작을 AMMR이 지원한다. 이 동작에 의한 단절·재연결은 통상 단절·재연결 흐름과 동일하게 처리된다.

## 8. 외부 시스템 통신

이 챕터는 Core가 연동하는 외부 시스템·물류 AMMR·Dashboard와의 통신을 정의한다. 각 외부 시스템별로 통신 방식, 데이터 종류, 동기화 정책, 오류 처리 원칙을 다룬다.

**※ 외부 시스템 통신 개요**

| **외부 시스템**             | **통신 방향**                         | **프로토콜**       |
|-----------------------------|---------------------------------------|--------------------|
| GM                          | Core → GM (폴링)                     | REST API           |
| SM                          | Core → SM (폴링)                     | REST API           |
| 일반 공정 WIP 중계 프로그램 | 중계 → Core (Event)                 | REST API           |
| 되담기 WIP 중계 프로그램    | 중계 → Core (Event)                 | REST API           |
| 물류 AMMR                   | AMMR ↔ Core (스트리밍·Event·명령) | MQTT               |
| Dashboard                   | Core ↔ Dashboard (Push·명령)        | SignalR + REST API |

Core의 외부 시스템 통신은 크게 세 가지 패턴으로 나뉜다. GM/SM은 Core가 직접 폴링하여 데이터를 가져오는 구조이고, WIP 중계 프로그램·물류 AMMR은 외부 측이 Event·스트리밍 발생 시 Core에 전달하는 구조이며, Dashboard는 Core가 운영 상태를 Push로 전달하고 사용자 명령은 Dashboard가 REST API로 송신하는 양방향 구조이다.

이 챕터의 중계 프로그램들과 물류 AMMR은 모두 HW 제어를 직접 담당하는 외부 Component이며, Core는 통신만 담당한다. 외부 Component의 내부 구현(Sensor 폴링 주기, 제어 로직, 물리적 수행 세부 등)은 Core 설계 범위 밖이다.

### 8.1 GM 시스템 통신

GM은 Core 외부에 존재하는 시스템으로, 생산관리시스템 역할을 한다. Core는 GM에서 작업 진행에 필요한 정보를 가져와야 한다.

**※ 외부 측 통신 제약**

GM은 REST API만 제공하며 Push를 지원하지 않는다.

**※ Core가 수신하는 데이터**

| **항목**  | **용도**                                 |
|-----------|------------------------------------------|
| 제품 정보 | Unit 구성 (수량·납기일·제품 속성 포함) |
| Recipe    | Unit의 공정 흐름 정의                    |

**※ GM 활성 제품 정책**

GM은 활성 상태의 제품 정보만 Core에 송신한다. 제품의 활성/비활성 결정 권위는 GM 측에 있으며, 사람이 GM 측에서 완료 처리한 시점에 비활성으로 전이된다.

GM 측에서 비활성 처리된 제품을 다시 활성으로 복원하는 경우는 발생하지 않는 것을 전제로 한다. 복원이 필요한 경우는 GM 측에서 신규 제품으로 재등록한다.

**※ 데이터 동기화 정책**

**※ 확정 시점은 QR 발행(=Unit 구성) 시점이다. 확정 전의 GM 데이터 변경은 자유롭게 반영하며, 확정 후에는 해당 Unit에 대해 GM 측 변경을 반영하지 않는다.**

| **데이터 항목**             | **확정 후** | **GM 측 변경 시 영향**                |
|-----------------------------|-------------|---------------------------------------|
| 제품 속성 (제품명·사양 등) | 변경 무시   | 발행 시점의 제품 속성으로 끝까지 진행 |
| 수량                        | 변경 무시   | 이미 발행된 Unit 수는 변경되지 않음   |
| 납기일                      | 변경 무시   | 발행 시점의 납기일로 우선순위 계산    |
| Recipe                      | 변경 무시   | 발행 시점의 Recipe로 끝까지 진행      |

**※ 협의 필요 사항**

위 단일 원칙은 Core 설계상의 가정이며, 현장 운영자와의 협의를 통해 검증이 필요하다.

| **협의 안건**         | **핵심 질문**                                                            |
|-----------------------|--------------------------------------------------------------------------|
| 수량 변경             | 이미 발행된 Unit은 그대로 진행해도 되는가?                               |
| Recipe 변경           | 구 Recipe로 끝까지 가도 품질상 문제 없는가?                              |
| 투입코드 삭제         | 이미 발행된 Unit은 그대로 진행 / 작업 중단 / 사용자 확인 중 어느 쪽인가? |
| 납기일 변경           | 이미 발행된 Unit의 우선순위가 변하지 않는 것이 충분한가?                 |
| 제품 속성 변경        | 이미 발행된 Unit은 어떻게 처리하는가?                                    |
| 활성/비활성 전이 시점 | 사람이 GM에서 완료 처리한 시점이 활성 → 비활성 전이 시점인가? 다른 운영 방식이 있는가? |

협의 결과에 따라 단일 원칙이 변경되거나 특정 항목에 예외 정책이 추가될 수 있다.

### 8.2 SM 시스템 통신

SM은 Core 외부에 존재하는 시스템으로, 장비/작업대 모니터링 시스템 역할을 한다.

**※ 외부 측 통신 제약**

SM은 REST API만 제공한다.

**※ Core가 수신하는 데이터**

| **항목**                | **용도**                                                     |
|-------------------------|--------------------------------------------------------------|
| 공정-작업대 Mapping 정보   | 어느 작업대가 어느 제품의 CNC 몇 공정을 담당하는지 식별      |
| 작업대 Slot Sensor 신호 | 작업대 Slot Sensor On/Off 값. 물류 AMMR 입고·출고 시점 판단 |

두 데이터가 조합되어야 의미가 있다. Sensor 신호만으로는 어느 공정인지 알 수 없고, Mapping 정보만으로는 현재 상태를 알 수 없다.

### 8.3 일반 공정 WIP 중계 프로그램 통신

일반 공정 WIP 중계 프로그램은 일반 공정 WIP의 HW 장비 및 QR 인식 수단과 Core 사이를 중계하는 외부 프로그램이다. Slot Sensor 신호는 HW 장비에서 받고, Unit 식별값 확정용 QR 정보는 QR 인식 수단에서 받는다.

**※ 외부 측 통신 제약**

REST API만 제공. WIP 중계가 Core를 호출하는 방향이다. 호출 시점은 (a) Event 발생 시 (b) HealthCheck 주기(예: 10초). HealthCheck 주기는 외부 측 보장 값으로 Core가 판단의 전제로 활용한다.

WIP 중계는 다음 두 경우의 부활 시점 첫 호출에 전체 Slot Sensor 상태를 동반 전송한다 — (가) WIP 중계 자기 측 단절 인지 후 복귀, (나) Core 측 무응답 지속 후 응답 재개.

**※ Core가 수신하는 Event**

| **Event**       | **내용**                                      |
|------------------|-----------------------------------------------|
| Slot Sensor 신호 | Slot Sensor On/Off 변화                       |
| QR 정보          | QR Scan으로 확정된 Unit 식별 정보 + Slot 위치 + 입고/출고 구분값 |

일반 공정 WIP 중계 프로그램은 Slot Sensor 신호를 감지하여 Core에 전달한다.

**※ Core가 송신하는 데이터** (Core → WIP 중계 프로그램)

| **항목**         | **Trigger**                                              |
|------------------|---------------------------------------------------------|
| Slot Unit 식별값 | 입고/출고 QR Scan 수신 시. WIP 중계 프로그램이 PLC 운영에 사용. |

### 8.4 되담기 WIP 중계 프로그램 통신

되담기 WIP 중계 프로그램은 일반 공정 WIP 중계 프로그램의 역할에 더해, 되담기 작업 완료 정보를 Core에 전달하는 역할을 담당한다.

**※ 외부 측 통신 제약**

REST API만 제공. 되담기 WIP 중계가 Core를 호출하는 방향이다. 호출 시점은 (a) Event 발생 시 (b) HealthCheck 주기(예: 10초). HealthCheck 주기는 외부 측 보장 값으로 Core가 판단의 전제로 활용한다.

되담기 WIP 중계는 다음 두 경우의 부활 시점 첫 호출에 전체 Slot Sensor 상태를 동반 전송한다 — (가) 되담기 WIP 중계 자기 측 단절 인지 후 복귀, (나) Core 측 무응답 지속 후 응답 재개.

**※ Core가 수신하는 Event**

| **Event**       | **내용**                                      | **비고**             |
|------------------|-----------------------------------------------|----------------------|
| Slot Sensor 신호 | Slot Sensor On/Off 변화                       | 일반 공정 WIP과 동일 |
| QR 정보          | QR Scan으로 확정된 Unit 식별 정보 + Slot 위치 + 입고/출고 구분값 | 일반 공정 WIP과 동일 |
| 되담기 완료      | Unit ID + 되담기 완료 여부                    | 되담기 WIP 고유      |

**※ Core가 송신하는 데이터** (Core → 되담기 WIP 중계 프로그램)

| **항목**         | **Trigger**                                              |
|------------------|---------------------------------------------------------|
| Slot Unit 식별값 | 입고/출고 QR Scan 수신 시. 되담기 WIP 중계 프로그램이 자체 Pair 판단에 사용. |

### 8.5 물류 AMMR 통신

물류 AMMR은 Core 외부에 존재하는 자율주행 로봇으로, Unit의 외부 이동을 담당한다. 현재 2대 운영.

**※ 외부 측 통신 제약**

AMMR HW와 Core는 MQTT 프로토콜로 통신한다. AMMR HW 단절은 MQTT Last Will로 Core가 인지한다.

**※ Core가 수신하는 데이터** (AMMR → Core)

AMMR이 보고하는 운영 상태값(HW 상태·위치·BMS·Slot Sensor), Job 지시 수신 확인 보고(Job 지시 직후), Job 종료 시점의 Job 수행 결과 보고(Move·Pickup·Dropoff·Charge)를 수신한다.

**※ Core가 송신하는 데이터** (Core → AMMR)

| 항목                                     | Trigger        |
|------------------------------------------|---------------|
| Job 지시 (Move·Pickup·Dropoff·Charge — 출발지·도착지 Slot 포함, 단 Charge는 위치 미포함) | Job 결정 시점 |
| AMMR Slot Sensor 변동 보고·Job 수행 결과 보고에 대한 응답 (해당 Slot의 Unit 식별값·Unit 제품 정보[제품 모델·버전·Tray 단 수·수량]·Job 배정·시스템 상태[정상·정합성 이상]) | 해당 보고 수신 시 (정상 운영 중 자동 갱신) |
| AMMR HW 상태 전이 보고에 대한 응답 (시스템이 판단한 AMMR 상태·작업 Slot 번호) | 해당 보고 수신 시 |
| AMMR 운영 상태값 일괄 보고에 대한 응답 (Slot별 결과와 해당 Unit 정보·Job 배정 6개 일괄) | 일괄 보고 수신 시 (초기 연결·재연결·재전송 요청 응답 포함) |
| AMMR Slot 적재 정보 일괄 재로드 응답 (Slot별 결과[정상·빈 Slot·정합성 이상]와 해당 Unit 정보·Job 배정) | AMMR 적재 정보 일괄 재로드 요청 수신 시 |
| 운영 상태값 일괄 보고 재전송 요청 | Core 측 통신 재연결·Core 재시작 복구 시점 |

AMMR 보고에 대한 위 응답들에는 시스템이 판단한 AMMR 상태·작업 Slot 번호가 함께 실린다. AMMR에 Mount된 태블릿은 위 응답 정보로 적재 상태·배정 상태·상단 정보 영역 화면을 표시한다. 태블릿은 AMMR과 함께 Core와 MQTT로 통신하는 단말이며, 별도 통신 채널을 사용하지 않는다.

**※ AMMR HW 자율 동작 영역**

도킹·충전·이탈 등 물리적 수행 세부와 충전 중단 결정은 AMMR HW가 자체 처리한다. Core는 Job 수행 결과 보고만 수신한다. 자율 주행(위치 측위·경로 결정·물리적 이동 수행 세부)도 AMMR HW 영역이다. AMMR HW 정지·리셋·재시작 등 HW 생명주기 제어도 AMMR 업체 자율주행부 영역이며, Core는 물류 Job(Move·Pickup·Dropoff·Charge)만 지시한다.

### 8.6 Dashboard 통신

Dashboard는 Core 외부에 존재하는 클라이언트 애플리케이션으로, 운영자가 시스템 상태 조회·수동 명령을 수행하는 단말기 역할을 한다.

**※ 외부 측 통신 제약**

Dashboard는 두 채널을 사용한다 — Core가 운영 상태를 SignalR로 실시간 Push하고, Dashboard가 사용자 명령을 REST API로 송신한다.

**※ Core가 송신하는 데이터** (Core → Dashboard, SignalR Push)

| 항목                                                                         | Trigger             |
|------------------------------------------------------------------------------|--------------------|
| 운영 상태 변화 (Slot 점유·Unit 식별값·정합성 이상·AMMR 상태·Job 진행 등) | 상태 변화 발생 시  |
| 경고 (병목·장애·정합성 이상 등, 병목은 주의·위험·긴급 단계별 구분)       | 경고 발생·해제 시 |

**※ Dashboard가 송신하는 데이터** (Dashboard → Core, REST API)

| 항목 | 용도                    |
|------|-------------------------|
| 사용자 명령 (설정 변경·Recipe 관리·Unit 식별값 수동 입력·QR 발행·외주 관리 등) | 사용자 명령 처리 Trigger |

## 9. Event 시스템

### 9.1 외부 사건 목록

CNC 작업대는 Slot Sensor로 물류 AMMR 입고·출고를 인지하고, 작업 시작·종료는 외부 QR 보고가 있을 때만 기록된다 (외부 측 자율, 없으면 미기록). 물류 AMMR은 HW 상태 전이·Slot 점유 변화·Job 수행 결과를 외부 사건으로 보고한다.

## 10. Recipe 및 공정 관리

이 챕터는 Unit 단위로 운영하는 개별 Recipe와 변경 제약 조건을 정의한다.

### 10.1 Unit별 개별 Recipe 관리

같은 투입코드로 생성된 Unit들은 초기에 동일 Recipe를 가지나, 담당자가 Unit별로 공정을 추가/삭제/변경할 수 있다.

**※ 개별 Recipe 관리 기능**

- **Unit별 공정 조회**: 현재 Unit의 공정 순서 확인
- **공정 추가**: 특정 Unit에 공정 삽입 (위치 지정 가능)
- **공정 삭제**: 특정 Unit에서 공정 Skip 처리
- **Tray 되담기 공정 추가**: 담당자가 Unit별로 명시적으로 추가
- **일괄 변경**: 여러 Unit 선택 후 동일 변경 적용
- **적용 범위 선택**: ① 이 Unit만 적용 ② 같은 투입코드에서 해당 공정이 있는 Unit에만 적용

### 10.2 변경 제약 조건

**※ Recipe 변경 제약**

- **완료된 공정**: 이미 완료된 공정은 삭제/변경 불가
- **이동 중인 Unit**: 물류 AMMR 이동 완료 후에만 변경 가능
- **작업 중인 Unit**: CNC 공정 AMMR 작업 완료 후에만 변경 가능
- **다음 공정 배정됨**: 경고 표시 → 담당자 확인 후 변경
- **이미 적층 Tray인 Unit**: Tray 되담기 공정 추가 불가 (이미 완료 상태)

## 11. 외주 이력 관리

이 챕터는 외주 구간 Unit 이력 관리 정책을 정의한다.

### 11.1 추적 정책

외주 구간에서는 Unit이 재편되기 때문에 외주 배치 단위로 이력을 관리한다.

**※ 외주 구간 추적 보장**

- **외주 출발·복귀 Unit Mapping**: 외주 배치 단위로 폐기 Unit ID ↔ 신규 Unit ID Mapping 보존
- **외주 복귀 후 공정 이력**: 신규 Unit 단위 추적

## 12. Dashboard 사용자 권한 및 보안 Log

이 챕터는 Dashboard 구성과 사용자 권한, 보안 Log 정책을 정의한다.

### 12.1 로그인 및 권한 등급

Dashboard 접근은 IP Whitelist 등록된 단말기만 허용한다. 시스템 관리자는 모든 기능에 접근 가능하다. 그 외 운영 사용자는 기본 기능에 접근 가능하며, 아래 표는 역할별 추가 접근 가능 기능을 정리한다.

**※ 역할별 추가 접근 가능 기능**

## 13. 장애 대응 및 복구 정책

이 챕터는 Core 시스템·외부 구성요소의 장애 대응과 복구 정책을 정의한다.

### 13.1 Core 다운 시 대응

이 절은 Core 다운 시 시스템·외부 구성요소의 동작과 부활 시 자연 정합 회복을 정의한다.

**※ Core 다운 시 Dashboard 사용자 명령 차단**

Core 다운을 Dashboard가 감지한 동안에는 사용자 명령 입력을 차단한다. 사용자의 명령이 실패하지 않도록 입력 자체를 막아 운영 안전을 확보한다. Core 복구 시 자동으로 정상 운영 화면으로 복귀한다.

**※ Core 다운 중 물류 AMMR 자율 동작**

Core 다운 중 진행 중이던 Job은 완료까지 수행하며, 완료 후 AMMR은 그 위치에서 대기한다. 담당자가 수동으로 충전존 복귀 처리한다. 단, 저전력 진입 시 AMMR HW 자율 충전존 이동·충전·완충 동작은 AMMR HW 자체 동작이므로 Core 다운 여부와 무관하게 자연 작동한다.

**※ Core 부활 시 자연 정합 회복**

Core 부활 시 외부 시스템·AMMR과의 통신이 자동 재연결되며, 데이터를 자동 재수신하여 자연 정합 상태로 복구된다. 담당자의 별도 조치는 필요 없다.

### 13.2 구성요소 장애 대응

이 절은 외부 시스템·구성요소의 장애 감지·Core 응답·복구 방식을 정리한다.
