## 1. DI 등록

### 1.1 개념

.NET DI는 `Microsoft.Extensions.DependencyInjection` 패키지로 내장된 의존성 주입 Container이다. 핵심 역할은 **"어떤 인터페이스가 요청되면 어떤 구현체를 줄 것인가"의 레지스트리** 관리.

장점:

- 비즈니스 로직이 구체 구현을 몰라도 됨 → Port/Adapter 교체 자유
- 테스트 시 Mock 주입이 자연스러움
- 생성자 인자·생명주기 관리를 Container가 대행

### 1.2 등록 문법

앱 시작 지점(`Program.cs`)에서 `builder.Services`에 등록한다.

```csharp
var builder = WebApplication.CreateBuilder(args);

// 인터페이스 ← 구현체 Mapping
builder.Services.AddSingleton<IAmmrCommunication, MqttAmmrAdapter>();
builder.Services.AddSingleton<IStateService, StateService>();
builder.Services.AddSingleton<ITransferService, TransferService>();

var app = builder.Build();
app.Run();
```

`AddSingleton`은 "앱 전체 단일 인스턴스"로 등록한다는 뜻.

### 1.3 생명주기 3종

| 메서드 | 의미 | 용도 |
|---|---|---|
| `AddSingleton` | 앱 생명 전체에 하나 | 상태 보관·장기 연결 (State Service, AMMR Adapter) |
| `AddScoped` | HTTP 요청 1개에 하나 | 요청별 DB Context (Command API 내부) |
| `AddTransient` | 요청할 때마다 새로 | 가벼운 stateless 객체 |

아키의 서비스·Adapter는 거의 다 `AddSingleton`. AMMR MQTT Session·InMemory 상태는 앱 전체에 하나여야 하므로.

### 1.4 생성자 주입

등록된 서비스는 생성자 인자로 자동 주입된다.

```csharp
public class AmmrService : IAmmrService
{
    private readonly IAmmrCommunication _communication;
    private readonly IStateService _state;

    // 생성자 — DI Container가 두 인자를 알아서 채워줌
    public AmmrService(IAmmrCommunication communication, IStateService state)
    {
        _communication = communication;
        _state = state;
    }

    public void SendMoveCommand(string ammrId, string dest)
    {
        // 여기서 _communication이 MqttAmmrAdapter인지 SimulatorAmmrAdapter인지
        // AmmrService는 모름. Port 인터페이스에만 의존.
        // 비규범 예시 — DI 주입 설명용. 토픽만 정본화, 봉투·필드는 ICD 정본 참조. 방향: Core→AMMR.
        _communication.Publish($"core/ammr/{ammrId}/job/cmd", dest);
    }
}
```

### 1.5 환경별 Adapter 분기

아키 본체 1장 "AMMR Adapter 구현 4종" 표와 `appsettings.json` 분기의 실체 구현.

`appsettings.json`:

```json
{
  "AmmrAdapter": "Simulator"
}
```

`Program.cs`에서 값을 읽어 분기 등록:

```csharp
var adapterType = builder.Configuration["AmmrAdapter"];
switch (adapterType)
{
    case "RealMqtt":
        builder.Services.AddSingleton<IAmmrCommunication, MqttAmmrAdapter>();
        break;
    case "Simulator":
        builder.Services.AddSingleton<IAmmrCommunication, SimulatorAmmrAdapter>();
        break;
    case "Mock":
        builder.Services.AddSingleton<IAmmrCommunication, MockAmmrAdapter>();
        break;
    case "Replay":
        builder.Services.AddSingleton<IAmmrCommunication, ReplayAmmrAdapter>();
        break;
    default:
        throw new InvalidOperationException($"Unknown adapter: {adapterType}");
}
```

**핵심**: `AmmrService` 코드는 한 줄도 안 바뀌는데 설정값만 바꾸면 MQTT ↔ 시뮬레이터 ↔ Mock이 교체된다. Hexagonal의 Port/Adapter 교체가 DI로 실현되는 지점.

### 1.6 실전 팁

- **여러 구현 동시 등록**: `AddSingleton<IAmmrCommunication, A>()`, `AddSingleton<IAmmrCommunication, B>()` 둘 다 등록하면 `IEnumerable<IAmmrCommunication>`으로 전체 목록 주입받을 수 있음. "모든 Adapter에 broadcast" 같은 패턴에 활용.
- **Factory 패턴**: 런타임 조건으로 Adapter를 골라야 하면 `AddSingleton<IAmmrAdapterFactory, ...>()` 로 팩토리를 등록하고 팩토리가 내부에서 선택.
- **Options 패턴**: 설정값을 강타입 객체로 주입. `builder.Services.Configure<AmmrOptions>(builder.Configuration.GetSection("Ammr"));` 후 `IOptions<AmmrOptions>` 주입.
- **Channel Writer/Reader 분리 등록**: 계층 경계용 Channel은 `.Writer`와 `.Reader`를 각각 별도 DI 등록하여 단방향 흐름을 타입 시스템으로 강제할 수 있음 (상세는 2.12).

## 2. .NET Channels

### 2.1 개념

`System.Threading.Channels`는 **고성능 비동기 producer-consumer Queue**이다. `async/await`와 자연스럽게 결합되어 스레드를 차단하지 않고 대기할 수 있다.

Channel의 핵심 가치를 이해하려면 먼저 "동기/비동기"의 본질(2.2)과 기존 대안 도구들이 대기를 어떻게 처리하는지(2.3)를 짚어야 한다.

### 2.2 동기/비동기의 본질 — 스레드 관점

이 축의 핵심은 **스레드를 붙잡고 있느냐, 놓아주느냐**이다.

**동기 (Synchronous)**: 메서드를 호출하면 그 메서드가 작업을 완전히 끝낼 때까지 호출한 쪽이 멈춰 있음. 멈춰 있는 동안 해당 스레드는 다른 일을 하지 못함.

**비동기 (Asynchronous)**: 메서드를 호출하면 "아직 결과 없음" 표시(Task)를 즉시 돌려받음. 호출한 쪽은 그 표시를 `await`하면 결과가 준비될 때까지 기다리되, **그동안 스레드는 해방되어 다른 일을 처리할 수 있음**. 결과가 준비되면 스레드 풀에서 아무 스레드나 하나가 이어받아 실행을 계속함.

**"스레드가 붙잡힌다"는 게 왜 중요한가**:

.NET의 스레드는 OS가 관리하는 실행 단위로, **생성 비용이 크고 개수가 유한**하다. ASP.NET Core 기본 스레드 풀은 수백 개 수준. 이 스레드들이 블로킹되면:

- 블로킹된 스레드는 CPU는 쓰지 않지만 "자리"는 점유함
- 요청이 많이 들어오는데 스레드가 다 블로킹되어 있으면 신규 요청 처리 지연
- 극단적 경우 스레드 풀 고갈 → 서버 전체 응답 정지

반대로 비동기(`await`)는 대기 중 스레드를 풀에 반납하므로, **적은 수의 스레드로 수백~수천 개의 동시 작업**을 처리할 수 있다.

### 2.3 네 도구의 대기 메커니즘 비교

| 도구 | 동시성 지원 | 대기 구현 | 스레드 상태 |
|---|---|---|---|
| `Queue<T>` | 없음(단일 스레드) | — | — |
| `ConcurrentQueue<T>` | 동기 multi-thread | 폴링 필요 | 점유 + CPU 일부 소모 |
| `BlockingCollection<T>` | 동기 blocking | OS 레벨 sleep/wake | 점유, CPU 0 |
| **`Channel<T>`** | 비동기 | Task 상태 전이 | **해방, 풀에 반납** |

각 도구의 실제 대기 방식:

**`Queue<T>`** — 스레드 안전 아님. 여러 스레드가 동시에 건드리면 race condition. 동시성 지원 자체가 없으므로 기준점으로만 언급.

**`ConcurrentQueue<T>`** — 스레드 안전하지만 **대기 기능이 없음**. 비어있을 때 "들어올 때까지 기다려"라는 명령이 없어 직접 폴링 루프 작성 필요:

```csharp
while (!queue.TryDequeue(out var item))
{
    Thread.Sleep(10);  // 10ms마다 다시 확인
}
Process(item);
```

폴링 간격을 줄이면 지연은 감소하지만 CPU 낭비 증가. 트레이드오프 해결 불가.

**`BlockingCollection<T>`** — ConcurrentQueue의 대기 문제를 해결하되 방식이 **스레드 블로킹**:

```csharp
var bc = new BlockingCollection<int>();

new Thread(() => {
    while (true)
    {
        int item = bc.Take();  // 비어있으면 이 스레드가 OS 수준에서 멈춤
        Process(item);
    }
}).Start();
```

폴링이 아니라서 CPU 낭비 없고 즉각 반응. 그러나 **해당 스레드는 블로킹 상태로 자리만 차지**. Consumer가 10개면 스레드 10개가 대기 상태로 점유. `async/await`와 호환 안 됨.

**`Channel<T>`** — 대기 기능을 **Task 기반**으로 구현:

```csharp
var channel = Channel.CreateUnbounded<int>();

await foreach (var item in channel.Reader.ReadAllAsync())
{
    Process(item);
}
```

`ReadAsync()`는 Queue가 비면 Task를 돌려주고, `await`하면 호출 메서드는 "완료 대기" 상태로 진입하며 **스레드는 풀에 반납**된다. 항목이 들어오면 스레드 풀에서 아무 스레드나 하나가 이어받아 실행. Consumer가 100개여도 실제 스레드 블로킹은 0.

**정리**: `BlockingCollection`과 `Channel`은 둘 다 CPU 낭비가 없지만 **스레드 자원 소비 관점에서는 완전히 다르다**. BlockingCollection은 스레드 하나를 잠재운 채 점유하고, Channel은 스레드를 놓아준다.

Core 시스템의 State Service는 여러 Adapter가 Producer로 Event를 쏘는 구조라 Channel의 "비동기 대기 + 다수 Writer" 특성이 정확히 맞으며, ASP.NET Core·MQTTnet 등 async 기반 라이브러리와 성격이 일치한다.

### 2.4 기본 사용

```csharp
// 1. 생성
var channel = Channel.CreateUnbounded<string>();

// 2. Producer — 쓰기
await channel.Writer.WriteAsync("message 1");
await channel.Writer.WriteAsync("message 2");

// 3. Consumer — 읽기
var msg = await channel.Reader.ReadAsync();  // "message 1"

// 또는 Stream 방식
await foreach (var m in channel.Reader.ReadAllAsync())
{
    Console.WriteLine(m);
}
```

### 2.5 Bounded vs Unbounded

**Unbounded (크기 무제한)**:

```csharp
var ch = Channel.CreateUnbounded<T>();
// 빠르지만 Producer가 Consumer보다 빠르면 메모리 폭증 위험
```

**Bounded (크기 제한)** — 권장:

```csharp
var ch = Channel.CreateBounded<T>(new BoundedChannelOptions(capacity: 1000)
{
    FullMode = BoundedChannelFullMode.Wait,  // 가득 차면 Producer 대기 (backpressure)
    SingleReader = true,
    SingleWriter = false
});
```

`FullMode` 옵션:

- `Wait`: Producer가 await로 대기 (가장 안전)
- `DropOldest`: 가장 오래된 항목 버림
- `DropNewest`: 새 항목 버림
- `DropWrite`: 새 항목 버림 + 즉시 완료

### 2.6 SingleReader / SingleWriter 힌트

이 두 옵션이 성능 최적화의 핵심.

- `SingleReader = true`: Consumer가 1개임을 보장 → 내부 lock 제거 → 성능 향상
- `SingleWriter = true`: Producer가 1개임을 보장 → 마찬가지

아키의 "**State Service 입력 Channel — 단일 Consumer FIFO**"는 정확히 `SingleReader = true` 설정.

### 2.7 State Service 입력 Channel의 실제 패턴

아키 설계 그대로 옮기면:

```csharp
public class StateService : IHostedService
{
    private readonly Channel<IDomainEvent> _input;
    private Task? _consumerTask;
    private CancellationTokenSource? _cts;

    public StateService()
    {
        _input = Channel.CreateBounded<IDomainEvent>(
            new BoundedChannelOptions(1000)
            {
                FullMode = BoundedChannelFullMode.Wait,
                SingleReader = true,    // ← 단일 Consumer 보장
                SingleWriter = false    // ← 여러 Adapter가 발행
            });
    }

    // Producer API — Adapter들이 호출
    public ValueTask PublishAsync(IDomainEvent evt, CancellationToken ct = default)
    {
        return _input.Writer.WriteAsync(evt, ct);
    }

    // IHostedService — 앱 시작 시 consumer 루프 기동
    public Task StartAsync(CancellationToken cancellationToken)
    {
        _cts = CancellationTokenSource.CreateLinkedTokenSource(cancellationToken);
        _consumerTask = Task.Run(() => ConsumeLoop(_cts.Token));
        return Task.CompletedTask;
    }

    public async Task StopAsync(CancellationToken cancellationToken)
    {
        _input.Writer.Complete();            // 더 이상 입력 없음 신호
        _cts?.Cancel();
        if (_consumerTask != null)
            await _consumerTask;
    }

    // Consumer 루프 — 유일한 읽기 주체
    private async Task ConsumeLoop(CancellationToken ct)
    {
        await foreach (var evt in _input.Reader.ReadAllAsync(ct))
        {
            ProcessOne(evt);   // 직렬 처리 — 순서 보장, 경합 없음
        }
    }

    // 한 덩어리의 atomic 단위 (아키 4.5/6장 서술)
    private void ProcessOne(IDomainEvent evt)
    {
        // ① InMemory 상태 갱신
        UpdateInMemoryState(evt);

        // ② 정합성 검사
        VerifyConsistency(evt);

        // ③ 도메인 Event 발행
        PublishDomainEvent(evt);
    }
}
```

### 2.8 왜 이 구조가 atomic을 보장하는가

- Consumer가 단 하나의 루프. 한 번에 한 Event만 꺼내서 처리.
- `ProcessOne` 안의 3단계(갱신 → 검사 → 발행)는 다른 Event 처리와 절대 교차되지 않음.
- 즉 "갱신 중에 다른 Event의 검사가 끼어드는" 동시성 문제 원천 차단.
- Producer(Adapter들)가 여러 스레드에서 동시에 `WriteAsync` 호출해도 순서는 Channel이 직렬화.

이것이 아키 본체 `한 덩어리의 atomic 단위(InMemory 갱신 → 정합성 검사 → Event 발행)로` 서술의 실체 메커니즘이다.

### 2.9 Backpressure 실전 효과

Bounded Channel + `FullMode.Wait` 조합은:

- Producer(Adapter)가 너무 빠르면 `WriteAsync`가 await 상태로 멈춤
- 이는 Adapter가 자연스럽게 속도를 늦추는 효과
- AMMR Adapter가 메시지를 너무 빨리 받으면 → Queue가 차면 → MQTT 수신 스레드가 await → MQTT Broker가 KeepAlive로 속도 조절
- 메모리 폭증 없이 부하 조정이 end-to-end로 전파

### 2.10 도메인 Event 발행

State Service 안에서 외부 발행도 또 다른 Channel로 구현 가능:

```csharp
private readonly Channel<IDomainEvent> _output = Channel.CreateUnbounded<IDomainEvent>();

public ChannelReader<IDomainEvent> DomainEvents => _output.Reader;

private void PublishDomainEvent(IDomainEvent evt)
{
    _output.Writer.TryWrite(evt);  // non-blocking, 실패는 거의 없음(Unbounded)
}
```

다른 서비스(Transfer Service 등)는 이 reader를 주입받아 구독:

```csharp
await foreach (var evt in stateService.DomainEvents.ReadAllAsync(ct))
{
    HandleEvent(evt);
}
```

### 2.11 함정 3가지

1. **`Writer.Complete()` 미호출**: Consumer의 `ReadAllAsync`는 Complete가 없으면 영원히 대기. 앱 종료 시 반드시 호출.
2. **`TryWrite` vs `WriteAsync`**: Unbounded는 TryWrite로 동기 쓰기 가능. Bounded는 WriteAsync로 await 필요.
3. **Exception 처리**: Consumer 루프에서 예외가 새면 루프 전체가 죽음. 각 `ProcessOne` 호출을 try/catch로 감싸고, 오류는 Logging + Event Skip 또는 dead-letter queue로.

### 2.12 ChannelWriter / ChannelReader 분리 DI 패턴

아키 4.5 ※ 세 서비스 입력·관리 장치 구조 비교에서 State Service 입력 Channel의 배치 근거로 든 "독립된 계층 경계 장치로 두고 양쪽이 같은 추상(`ChannelWriter<T>` / `ChannelReader<T>`)에 의존하게 하는 것"이 실제로 어떻게 구현되는지 설명한다.

Channel은 `Writer`와 `Reader`를 **별도의 타입**으로 분리해 제공한다. 하나의 `Channel<T>` 객체가 `.Writer`(쓰기 전용 face)와 `.Reader`(읽기 전용 face) 두 face를 가지며, DI에서 각 face를 독립 등록하여 필요한 쪽에만 흘려 보낼 수 있다.

**DI 등록 패턴**:

```csharp
// Program.cs — Channel 원본은 여기서만 보유
var channel = Channel.CreateBounded<IDomainEvent>(
    new BoundedChannelOptions(1000)
    {
        FullMode = BoundedChannelFullMode.Wait,
        SingleReader = true,
        SingleWriter = false
    });

// Adapter들에게는 Writer만 주입
builder.Services.AddSingleton<ChannelWriter<IDomainEvent>>(channel.Writer);

// State Service에게는 Reader만 주입
builder.Services.AddSingleton<ChannelReader<IDomainEvent>>(channel.Reader);
```

**Adapter 쪽 — Writer만 보임**:

```csharp
public class MqttAmmrAdapter
{
    private readonly ChannelWriter<IDomainEvent> _writer;

    public MqttAmmrAdapter(ChannelWriter<IDomainEvent> writer)
    {
        _writer = writer;  // 쓰기 기능만 가진 객체
    }

    public async Task OnMqttMessage(...)
    {
        await _writer.WriteAsync(evt);   // OK
        // _writer.ReadAsync() — 존재하지 않음. 컴파일 단계에서 차단.
    }
}
```

**State Service 쪽 — Reader만 보임**:

```csharp
public class StateService
{
    private readonly ChannelReader<IDomainEvent> _reader;

    public StateService(ChannelReader<IDomainEvent> reader)
    {
        _reader = reader;  // 읽기 기능만 가진 객체
    }

    private async Task ConsumeLoop(CancellationToken ct)
    {
        await foreach (var evt in _reader.ReadAllAsync(ct))  // OK
            ProcessOne(evt);
        // _reader.WriteAsync() — 존재하지 않음. 자기 Channel에 self-write 불가.
    }
}
```

**이 분리가 주는 이점** (아키 4.5 본문이 "의존성 역전·인터페이스 분리·테스트 용이성"으로 요약한 세 축):

- **인터페이스 분리 원칙(ISP)** — 각 계층이 "필요한 기능만" 보게 됨. Adapter가 실수로 `ReadAsync`를 호출해 State Service가 Event를 놓치는 상황, State Service가 자기 Channel에 self-write하는 이상 패턴이 **컴파일 시점에 원천 차단**됨. 타입 시스템이 단방향 데이터 흐름을 강제.
- **의존성 역전 원칙(DIP)** — Adapter는 `ChannelWriter<IDomainEvent>`만 알고 State Service가 어떻게 생겼는지 모름. State Service는 `ChannelReader<IDomainEvent>`만 알고 어떤 Adapter들이 보내는지 모름. 두 계층이 **서로를 직접 결합하지 않고** Channel 추상 경계에서만 만남. 아키 4.5가 지적한 "계층 역전" 방지의 실제 메커니즘.
- **테스트 용이성** — Adapter 단위 테스트에선 `ChannelWriter<IDomainEvent>` Mock만 주입(State Service 없이 테스트). State Service 단위 테스트에선 `ChannelReader<IDomainEvent>` Mock만 주입(Adapter 없이 테스트). 서로의 영역을 침범하지 않는 독립 단위 테스트.

**이 패턴이 State Service 입력 Channel에 특히 중요한 이유**:

State Service 입력 Channel은 **계층 경계 장치**(Adapter ↔ 서비스)라 양쪽이 서로 몰라야 한다. 반면 서비스 내부 자료구조(공유 TransferList, AmmrList)는 해당 서비스가 단독 소유하므로 Writer/Reader 분리의 효용이 작다 — 한 객체가 양쪽 역할을 겸해도 무방. 이 차이가 아키 4.5 ※ 세 서비스 입력·관리 장치 구조 비교 표 "DI 노출" 열의 차이 — State Service 입력 Channel만 "Writer·Reader 각각 DI 등록"이고, 나머지는 "List 객체는 private, 서비스만 DI 등록" — 로 드러난다.

## 3. ConcurrentDictionary

아키 4.6에서 "인프라 계층"에 Channel과 함께 놓인 도구. Channel과는 용도가 완전히 달라 비교 대상이 아니라 협력 대상이다.

### 3.1 Queue와 Dictionary의 용도 구분

| 분류 | 목적 | 데이터 성격 | 대표 도구 |
|---|---|---|---|
| Queue 계열 | 한 방향 **Stream** | 시간 순서, 한 번 소비 | Channel, ConcurrentQueue |
| Dictionary 계열 | 키 기반 **랜덤 액세스** | 현재 상태, 반복 조회 | ConcurrentDictionary |

Channel은 "Event가 흘러가는 파이프"고, ConcurrentDictionary는 "현재 상태를 담아두는 보관소". 같은 데이터라도 보는 관점이 다르다.

- Channel: "방금 AMMR 2번이 Slot 1에 Unit을 놓았다" ← 시간 축의 한 사건
- ConcurrentDictionary: "지금 AMMR 2번 Slot 1에는 Unit-045가 있다" ← 현재 상태의 Snapshot

### 3.2 State Service가 보관하는 InMemory 상태

아키 4.5에서 State Service가 권위로 유지하는 것들:

- AMMR 운영 상태 (키: AMMR ID → 값: 정상/만재대기/장애 등)
- AMMR Slot 상태 (키: {AMMR ID, Slot Index} → 값: 점유 여부 + Unit 식별값)
- WIP/CNC Slot 상태 (키: Slot 위치 → 값: 센서·QR 두 필드)
- Block 마킹 상태

전부 "키로 찍어서 즉시 찾아봐야 하는" 구조. "AMMR 2번 Slot 3의 지금 상태" 같은 질문에 답하려면 랜덤 액세스가 필요하고, 이건 Queue로는 표현 불가. Queue는 소비 순서대로 꺼내지, 중간에서 "3번째 것만 골라줘"가 불가능하기 때문.

### 3.3 왜 "Concurrent" 버전인가

State Service 루프가 **유일한 Writer**지만 Reader는 여럿. 아키 6장 "조회 채널 원칙"이 이를 규정: 단위 지시 직전 검증 등 시점 검증은 Push 채널이 아니라 State Service InMemory 직접 조회(Pull)로 처리.

Reader 목록 예시:

- AMMR Service — 단위 지시 직전 Slot 상태 확인
- Transfer Service — 출고 Slot 점유 확인
- SignalR Hub — Dashboard 실시간 조회 응답
- Command API — 사용자 상태 조회 요청

Reader들은 각자 다른 스레드에서 동시에 접근. 그 사이 Writer(State Service 루프)가 값을 갱신 중일 수 있음.

**일반 Dictionary를 쓰면**:

```csharp
var dict = new Dictionary<string, SlotState>();

// Reader 스레드 (AMMR Service)
var state = dict[ammrSlotKey];   // 만약 이 순간 Writer가 갱신 중이면?
// → InvalidOperationException 또는 부분 갱신 값 노출
```

Reader가 enumerate 중에 Writer가 수정하면 예외 발생. 개별 키 읽기도 race condition으로 부분 갱신 상태가 노출될 수 있음.

**ConcurrentDictionary는**:

- Reader: lock 없이 원자적 Snapshot 획득 (CAS 기반)
- Writer: 해당 **Bucket만** 잠금, 다른 키 접근은 방해 안 함
- Reader 다수여도 경합 없음

### 3.4 일반 Dictionary + lock으로 대체하면

기능적으로는 가능하지만 비효율:

```csharp
lock (_dictLock)
{
    return _dict[key];  // 모든 읽기가 같은 락을 경유
}
```

- Reader 100개가 동시 조회 → 전원이 직렬화됨 (같은 락 대기)
- Writer가 한 키 갱신 중 → 무관한 다른 키 조회까지 대기
- 조회 성능 급락

ConcurrentDictionary는 이 문제를 Bucket 단위 세분 락킹으로 해결하고, 추가로 **원자적 복합 연산 API**를 제공:

```csharp
// 조회 후 갱신을 원자적으로
dict.AddOrUpdate(key,
    addValue: newState,
    updateValueFactory: (k, old) => UpdateState(old, newState));

// 없으면 만들고, 있으면 기존 값 반환
var state = dict.GetOrAdd(key, k => CreateDefaultState(k));
```

일반 Dictionary + lock으로 같은 걸 하려면 lock 안에서 TryGetValue → 분기 → Add/Update를 직접 짜야 한다. 실수 여지가 늘어나고 락 범위도 커짐.

### 3.5 Channel과 ConcurrentDictionary의 협력

둘이 State Service 안에서 같이 돌아간다.

```csharp
public class StateService
{
    // 들어오는 Event Stream (시간 축)
    private readonly Channel<IDomainEvent> _input;

    // 현재 상태 보관 (공간 축)
    private readonly ConcurrentDictionary<SlotKey, SlotState> _slots;
    private readonly ConcurrentDictionary<string, AmmrState> _ammrs;

    // Consumer 루프 — 아키의 atomic 단위
    private async Task ConsumeLoop(CancellationToken ct)
    {
        await foreach (var evt in _input.Reader.ReadAllAsync(ct))
        {
            // ① InMemory 상태 갱신 — Dictionary 쓰기
            _slots.AddOrUpdate(evt.SlotKey, ...);

            // ② 정합성 검사 — Dictionary 읽기
            var current = _slots[evt.SlotKey];
            VerifyConsistency(current);

            // ③ 도메인 Event 발행 — 다른 Channel에 쓰기
            _output.Writer.TryWrite(new DomainEvent(...));
        }
    }

    // 다른 서비스가 호출하는 조회 API — Dictionary 읽기
    public SlotState GetSlotState(SlotKey key) => _slots[key];
}
```

- **Channel**: "Event가 어떻게 흘러 들어오고 나가는가" — 시간 축
- **ConcurrentDictionary**: "현재 상태가 어떤가" — 공간 축

State Service 루프가 유일한 Writer이므로 **Dictionary에 대한 동시 쓰기는 발생하지 않는다**. 그런데도 Concurrent가 필요한 이유는 **Reader 다수가 다른 스레드에서 동시 조회**하기 때문. 쓰기·읽기가 동시에 일어나는 상황에서 Reader의 안전을 보장하는 것이 ConcurrentDictionary의 역할.

## 4. 아키 자료구조 4종의 실제 구현

아키 본체에 등장하는 동시성 자료구조가 4종 있다. 이름과 구현이 1:1 대응하도록 정비되어 있다 — 아키 본체가 "State Service 입력 Channel"·"Channels (Event 버스)"로 부르는 것은 .NET `System.Threading.Channels` 구현체이며, **TransferList·AmmrList는 이름·구현 모두 List**(`List<T>` + 락). 이 장은 4종의 실제 구현이 어떻게 결정되는지 설명한다.

### 4.1 4종 현황 — 어떤 자료구조가 Channel인가

| 자료구조                    | 아키 위치                                                            | 실제 구현                                                                |
|-----------------------------|---------------------------------------------------------------------|--------------------------------------------------------------------------|
| State Service 입력 Channel  | 인프라 계층 (4.6)                                                    | **Channel** (`System.Threading.Channels`)                                |
| Channels (Event 버스)       | 인프라 계층 (4.6)                                                    | **Channels** (도메인 Event 발행/구독)                                    |
| 공유 TransferList           | **State Service InMemory 권위** (Transfer Service는 읽기 전용 참조)   | **`List<T>` + 락 + LINQ OrderBy** (본체 SAD §4.5 결정)                    |
| AmmrList                    | **State Service InMemory 권위** (AMMR Service는 읽기 전용 참조)        | **`List<T>` + 락 + LINQ OrderBy** (본체 SAD §4.5 결정)                    |

앞 두 개(입력 Channel + Event 버스)는 .NET Channels로 구현. 뒤 두 개(TransferList·AmmrList)는 두 자료구조 모두 동일 패턴으로 List + 락 채택. 두 자료구조 모두 **State Service InMemory 권위 단일** (쓰기 권위 State Service 단일, 다른 서비스는 읽기 전용 참조로 접근).

### 4.2 TransferList·AmmrList의 실제 구현

이 절은 본체 SAD §4.5의 두 자료구조가 왜 List + 락 + LINQ OrderBy 패턴으로 확정되었는지를 단계적으로 보인다 — Channel 후보 탈락 → FIFO Queue 후보 탈락 → 정렬 컬렉션 후보 비교 → 이 시스템 선택.

#### 4.2.1 Channel이 아닌 이유

Channel의 특성:

- **FIFO 전용** — 들어온 순서대로 나감
- **우선순위 재정렬 없음**
- **중간 요소 검색·선택 불가** — 앞부터 꺼내는 것만 가능
- **원소 수정·삭제 API 없음**

본체 SAD를 보면 TransferList·AmmrList는 Channel로는 구현 불가한 요구를 가진다.

**TransferList (본체 §4.5)**: "우선순위에 따라 정렬한다"

- FIFO가 아니라 **우선순위 정렬**. 긴급 Transfer가 먼저 배정되어야 함.
- Channel은 우선순위 지원 안 함.

**TransferList (본체 §5.6·§5.8)**: "만재 → skip → TransferList의 다른 Transfer 재탐색"

- 맨 앞 Transfer를 꺼냈는데 조건 불일치면 건너뛰고 뒤의 것 탐색.
- Channel의 `ReadAsync`로는 불가능. 한 번 꺼내면 끝이고, 뒤를 뒤져볼 수 없음.

**AmmrList (본체 §5.6)**: "선택 메커니즘 상세(자격 판단, 점수 계산, 다중 배정, Slot Block 시 Transfer 반환)"

- 단순히 꺼내는 것이 아니라 **조건 평가 후 선택**. "만재 AMMR skip", "Battery 충분 AMMR 우선" 같은 필터·점수 기준 필요.
- Channel의 Pull로는 표현 불가.

#### 4.2.2 FIFO Queue도 아닌 이유

`Queue<T>`·`ConcurrentQueue<T>` 같은 FIFO 자료구조도 후보에서 탈락한다.

- **`Queue<T>`** — 스레드 안전 아님. 두 자료구조는 다중 스레드 접근(State Service 쓰기 + Transfer/AMMR Service 읽기 전용 참조)이라 단독 사용 불가. `lock` 추가하면 작동은 하지만 FIFO 전용이라 우선순위 정렬·조건부 선택 요구를 만족 못 함.
- **`ConcurrentQueue<T>`** — 스레드 안전은 해결되지만 두 한계가 결정적이다:
  - **비어있을 때 대기 API 없음** — 폴링(`while (true) { TryDequeue(...) }`) 강제. CPU 낭비 + 정확한 타이밍 제어 불가.
  - **중간 원소 조작 API 없음** — 정렬·skip·무효화·중간 제거 모두 불가.

이 시스템 요구(다중 스레드 접근 + 조건부 선택 + 무효화)는 FIFO로 처리할 수 없다.

#### 4.2.3 정렬 컬렉션 후보 비교

남는 후보는 정렬을 지원하거나 임의 위치 접근을 허용하는 컬렉션이다.

- **`PriorityQueue<TElement, TPriority>`** (.NET 6+) — 표준 우선순위 Queue. 그러나 **맨 앞 원소만 접근 가능**, 중간 순회 불가. "skip 후 다음 후보 탐색" 패턴이 부적합.
- **`SortedSet<T>`** — 정렬 상태 유지 + 임의 순회 가능. 규모 100+ 시 LINQ OrderBy 대비 유리하나 이 시스템 규모(수십 단위)엔 과잉 최적화.
- **`List<T>` + 락 + LINQ OrderBy 탐색** — 정렬 기준을 탐색 메서드 한 곳에 집중, 삽입은 단순 `Add`. 규모 가정 하 OrderBy 매번 재정렬 무시 가능.

#### 4.2.4 이 시스템 선택 — List + 락 + LINQ OrderBy

이 시스템은 두 자료구조 모두 동일하게 **`List<T>` + 락(Monitor/`lock`) + LINQ OrderBy 탐색** 패턴 채택. 명사도 자료구조에 맞춰 **TransferList·AmmrList**.

채택 근거 요약:

1. 정렬 기준이 탐색 메서드 한 곳에 집중 → 변경 유연성
2. 삽입 실수 여지 최소 (`Add` 한 줄)
3. TransferList·AmmrList 동일 패턴으로 학습 부담 축소
4. 규모 가정(수십 단위) 하 성능 차이 실질 무시
5. LINQ 친화성 — 특히 AmmrList의 Transfer별 동적 점수 정렬에 적합

이 결정은 코드 레벨이 아니라 아키 결정 영역으로 이관되어 본체 SAD §4.5에 명시된다.

### 4.3 계층 경계 vs 권위 위치 — 판정 기준

본체 SAD §4.5 ※ 세 서비스 입력·관리 장치 구조 비교에 실마리가 있다.

| 자료구조                    | 경계 성격                                                                                  | Writer·Reader 관계                                                                  |
|-----------------------------|--------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| State Service 입력 Channel  | **계층 간 경계** (Adapter → 서비스)                                                          | 다수 Writer + 단일 Reader                                                            |
| 공유 TransferList           | **State Service InMemory 권위**                                                             | State Service 쓰기 단일, Transfer Service 읽기 전용 참조                              |
| AmmrList                    | **State Service InMemory 권위**                                                             | State Service 쓰기 단일, AMMR Service 읽기 전용 참조                                  |

**입력 Channel은 "계층 경계"**를 넘는 Stream이라 양쪽이 서로 알면 안 된다. 추상화(`ChannelWriter<T>` / `ChannelReader<T>`)에 양쪽이 의존하는 것이 의존성 역전 원칙에 맞음. 그리고 Adapter 여럿이 동시 비동기로 쏘는 상황이라 Channel의 "**비동기 대기 + 다수 Writer**" 특성이 정확히 맞음. (실제 DI 등록·생성자 주입 패턴은 2.12 참조.)

**TransferList·AmmrList는 "State Service InMemory 권위"** 자료구조 (쓰기 권위 State Service 단일, 다른 서비스는 읽기 전용 참조). 외부는 State Service 공개 API로만 갱신 요청, 다른 서비스는 read-only 참조로 조회. 그리고 "우선순위 정렬"과 "조건부 선택"이 핵심 기능이라 Channel이 오히려 표현 부족. 이 경우 Channel을 쓸 이유가 없고, 락 기반 정렬 컬렉션이 자연스러움.

**판정 기준 요약**:

- **인프라 계층 자료구조** → Channel (계층 경계 Stream 장치라 Channel의 특성이 딱 맞음)
- **State Service InMemory 권위 자료구조** → Channel 아님 (정렬·선택·조건부 처리 요구가 큼)

이름은 "Channel"·"List"로 1:1 대응되며 실제로 "**Stream Channel**"과 "**정렬 List(List + 락)**"라는 서로 다른 자료구조다.

## 5. 세 기술이 아키에서 맞물리는 지점

1. **앱 시작**: DI Container가 `StateService`·각 `Adapter`를 Singleton으로 등록.
2. **Adapter 생성**: DI가 `MqttAmmrAdapter` 생성 시 생성자 인자로 `StateService` 주입.
3. **MQTT 수신**: `MqttAmmrAdapter`가 AMMR 메시지 수신 → `_stateService.PublishAsync(evt)` 호출.
4. **Channel Queue잉**: Event가 State Service 입력 Channel에 들어감.
5. **단일 Consumer 처리**: State Service의 백그라운드 루프가 하나씩 꺼내 atomic 처리.
6. **ConcurrentDictionary 갱신**: atomic 단위 ①에서 InMemory 상태 Dictionary가 갱신됨.
7. **정합성 검사**: atomic 단위 ②에서 Dictionary 값 조회로 정합성 확인.
8. **Channel 출력**: atomic 단위 ③에서 도메인 Event가 Event 버스 Channel로 나감.
9. **Pull 조회**: 다른 서비스(Transfer Service·AMMR Service 등)가 시점 검증 시 ConcurrentDictionary와 TransferList·AmmrList의 read-only 참조를 직접 Pull 조회.
10. **State Service InMemory 권위**: State Service InMemory 권위의 TransferList·AmmrList는 정렬·선택 기능이 필요하여 Channel이 아닌 List + 락으로 구현(본체 SAD §4.5 결정). Transfer Service·AMMR Service는 읽기 전용 참조 + State Service 쓰기 메서드 경유.

**세 기술의 역할 분담**:

- **DI** = 누가 누구를 아는가 (정적 구조, 의존성 주입 경로)
- **Channel** = Event가 어떻게 흐르는가 (동적 Stream, 계층 경계 장치)
- **ConcurrentDictionary** = 현재 상태는 무엇인가 (키 기반 랜덤 액세스, InMemory 권위 저장소)

셋이 맞물려서 아키의 "Adapter 계층 → State Service 입력 Channel → 서비스 계층 → Event 버스 → 구독 서비스 + Pull 조회" 구조가 실현된다. State Service InMemory 권위의 TransferList·AmmrList는 결정 로직을 지원하는 별도 자료구조로, Channel과는 다른 성격의 도구(List + 락)로 구현된다.

**본체 SAD §1.2 실패 시나리오와의 대응**

위 흐름의 각 도구가 본체 §1.2의 9개 실패 시나리오를 어떻게 막는지를 짧게 정리한다(본체 §1.3 3축 Mapping 경유).

- **State Service 입력 Channel + 단일 Consumer + atomic 처리** (위 4·5·8) — §1.3 순서·atomic 보존 축 → 시나리오 1(Interleave 상태 역전)·2(Event 발행 선행)·3(다수 Writer 경합) 방어
- **두 필드 분리 + Block 표시** — §1.3 인식 경계 축 → 시나리오 4(추정 확정)·5("모른다" 표현 불가)·6(센서 OFF 무시) 방어
- **ConcurrentDictionary InMemory 권위 + DB 이력 권위 분리** (위 6·9) — §1.3 권위 매체 축 → 시나리오 7(DB 실시간 권위)·8(다중 권위)·9(InMemory 없음) 방어

각 시나리오의 구체 묘사는 본체 §1.2 참조. 이 해설의 도구는 §1.3 축의 구현체이며, 결과적으로 §1.2 시나리오의 방어 장치다.
