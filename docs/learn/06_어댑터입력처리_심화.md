# Core 구현참고 Adapter 입력처리 구조 v1.0_d19

이 파일은 Core 아키 본체 §4.4(Adapter 계층)·§4.5(서비스 계층)·§4.6(State Service 입력 Channel)·§5장 Adapter 책임 원칙·다소비처 분기 구조를 "Adapter 입력 처리"라는 단일 횡단 관점으로 통합 해설한 학습 자료다. 외부 Payload가 도메인 객체로 변환되어 State Service 입력 Channel에 흘러 들어가고, 단일 Consumer가 타입별로 분기 처리하는 전 과정을 코드 수준으로 풀어낸다. 본체와 독립 수명이며, Core 프로젝트 zip 묶음에 동반 자료로 포함된다.

보조 자료 구조: Channels 해설(`Core_구현참고_Channels해설`)이 Channel 자체의 두 구조(FIFO 파이프 / Event 버스)을, DI와 동시성 해설(`Core_기술참고_DI와동시성해설`)이 DI 등록·동시성 자료구조 4종을 다룬다면, 이 문서는 **"Adapter에서 시작해 State Service 분기 처리에 도착하는 한 입력의 여정"**에 초점을 둔다. 중복 영역은 핵심만 요약하고 해당 자료를 참조 표시한다.

## 1. 검토 배경·이 문서 위상

### 1.1 검토 배경

이 문서는 Adapter → State Service 입력 경로를 코드 수준으로 통합 해설한다. (발생 경위·인수인계 근거는 _변경기록/근거모음으로) 광역 구조 [I](InMemory 권위 일원화 + State Service 단일 게이트웨이 + atomic 직접 판정 / AMMR Service = Transfer 매칭 + Job Sequence 진행 + 도메인 운영 결정 + 단위 지시 결정) 아래에서 Adapter → State Service 입력 경로는 다음 구조를 가진다 — 모든 Adapter(GM/SM/General Process WIP/Restack Process WIP/MQTT/Command API) 6종이 외부 프로토콜 차이와 무관하게 동일한 추상(`ChannelWriter<StateServiceInput>`)을 통해 State Service에 도달하고, State Service가 단일 Consumer로 타입별 분기 처리한다.

본체 §4.4·§4.5·§4.6은 이 구조를 텍스트와 표·다이어그램으로 명시하고 있으나, **"실제로 Adapter가 외부 프로토콜을 받아 어떤 도메인 객체로 변환하고, 단일 Channel을 통과한 다형 객체가 State Service 안에서 어떻게 타입별로 분기되어 처리되는가"**의 코드 수준 흐름은 한 곳에 집약된 형태로는 어디에도 없다. Channels 해설이 Channel 사용 패턴을, DI와 동시성 해설이 DI 등록·자료구조 선택을 다루지만, "한 입력의 여정"을 Adapter에서 분기 처리까지 끝까지 따라가는 구조는 이 문서가 처음이다.

특히 §5 **분기 구현 방식 후보 비교**는 코드 레벨 결정 영역이라 아키 본체에 담을 곳이 없다(아키 본체는 L3 — Component 수준). 그러나 분기 구현 방식 선택은 향후 Adapter·도메인 입력 추가 시 매번 부딪히는 결정이라, 후보별 구조·장단점을 미리 정리해두면 결정 비용이 낮아진다. 이 문서가 이 부분을 메운다.

### 1.2 이 문서 위상

- **위상**: 영구 보존 보조 자료. 아키 본체와 독립 수명. 본체 d 변경에 종속되지 않는다.
- **수명**: Channels 해설·DI와 동시성 해설과 동일 구조로 영구 보존(소멸 조건 없음). Core 프로젝트 zip 묶음에 동반.
- **갱신 Trigger**: (1) 본체 §4.4·§4.5·§4.6의 Adapter-서비스 경계 구조가 변경되어 이 문서 서술이 어긋날 때, (2) 분기 구현 방식 후보가 추가·기각될 때, (3) 신규 Adapter 추가로 6종 구조가 변경될 때.
- **사용자 가독성**: 보장 대상. 학습 자료 성격이라 강함축 비적용(Channels 해설·DI와 동시성 해설과 동일 구조).

### 1.3 다른 보조 자료와의 영역 분리

- 타입별 분기 구현 방식 후보 5종 비교 (§5)
- 한 입력의 여정 — 시작에서 분기 처리까지 (§7)

## 2. Adapter 측 처리 — 외부 Payload → 도메인 객체 변환

### 2.1 Adapter 책임 원칙의 코드 구조

본체 §5장 공통 원칙의 Adapter 책임 원칙은 다음을 규정한다.

> 모든 Adapter(GM/SM/WIP/MQTT/Command API)는 외부 프로토콜 수신 + 도메인 객체 변환만 담당한다. 비교·변화 감지·도메인 판단·Event 발행은 서비스 계층의 책임이며, Adapter에 섞이지 않는다.

코드 구조로 풀어보면:

```csharp
// === 잘못된 방식 — Adapter에 도메인 판단 섞임 ===
public class WipAdapter
{
    public async Task OnQr(WipQrDto dto, CancellationToken ct)
    {
        var current = _stateService.GetSlot(dto.WipId, dto.SlotId);  // ← Adapter가 상태 조회
        if (current.UnitId == dto.UnitId)
        {
            // 변화 없음 — skip
            return;  // ← Adapter가 "변화 감지" 판단
        }
        await _stateService.UpdateSlot(dto.WipId, dto.SlotId, dto.UnitId);  // ← Adapter가 상태 직접 갱신
        await _bus.Publish(new QrRecognizedEvent(...));  // ← Adapter가 Event 발행
    }
}
```

위 구조는 Adapter가 도메인 판단(변화 감지) + 상태 갱신 + Event 발행 3중 책임을 가진다. 정책이 다른 Adapter(GM/SM/MQTT)에 같은 패턴으로 흩어지면 정책 변경 시 5곳을 동시 수정해야 하고, 한 곳을 빠뜨리면 정합성이 깨진다.

```csharp
// === 올바른 방식 — Adapter는 변환·전달만 ===
public class WipAdapter
{
    private readonly ChannelWriter<StateServiceInput> _writer;

    public async Task OnQr(WipQrDto dto, CancellationToken ct)
    {
        // 외부 dto → 도메인 입력 객체 변환만 (Adapter 책임 원칙)
        var input = new WipQrInput(dto.WipId, dto.SlotId, dto.UnitId, dto.RestackDone, dto.RestackPlacements);
        await _writer.WriteAsync(input, ct);
        // 끝. 비교·변화 감지·상태 갱신·Event 발행은 State Service 책임.
    }
}
```

Adapter의 책임은 두 단계로만 구성된다:

1. **수신**: 외부 프로토콜(REST·MQTT·Polling·HTTP 수신)로 들어온 raw Payload 수령
2. **변환·전달**: raw Payload를 도메인 입력 객체(`StateServiceInput` 계층)로 변환 후 입력 Channel에 Write

이 구조의 효과:

- 정책 일관성 — 비교·판정 로직이 State Service 한 곳에 집중되어 정책 변경 시 한 곳만 수정
- 테스트 단순화 — Adapter 테스트는 "이 Payload → 이 도메인 객체로 변환되었는가"만, State Service 테스트는 "이 도메인 객체 → 이 InMemory 변화·Event 발행"만 분리 테스트
- 직렬화 구조 보존 — Adapter가 State Service InMemory에 직접 손대면 입력 Channel 직렬화 구조가 무너짐

### 2.2 6종 Adapter별 Payload 형태와 변환 패턴

각 Adapter는 외부 프로토콜이 다르므로 Payload 형태와 수신 방식이 모두 다르다. 그러나 변환 후 도착하는 도메인 객체는 동일한 봉투(`StateServiceInput` 계층) 안에 들어간다.

#### 2.2.1 GM Adapter — REST Polling, 마스터 데이터

**외부 프로토콜**: REST API GET, 1시간 주기 Polling.

**Payload 형태**: GM 시스템이 응답 본문에 제품 카탈로그 전체를 JSON으로 반환.

```json
{
  "products": [
    { "id": "P-001", "recipe": "...", "weight": 12.5 },
    { "id": "P-002", "recipe": "...", "weight": 8.3 }
  ],
  "polled_at": "2026-05-01T03:00:00Z"
}
```

**변환 패턴**: GM 시스템이 응답하는 카탈로그 전체를 한 번에 받아 `IReadOnlyList<ProductDto>` 형태로 도메인 입력 객체에 담는다. 비교(어떤 제품이 신규인가·갱신되었는가·삭제되었는가)는 State Service가 DB 기존 데이터와 대조하여 판단.

```csharp
public class GmAdapter : BackgroundService
{
    private readonly ChannelWriter<StateServiceInput> _writer;
    private readonly HttpClient _http;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        var timer = new PeriodicTimer(TimeSpan.FromHours(1));
        while (await timer.WaitForNextTickAsync(ct))
        {
            try
            {
                var dto = await _http.GetFromJsonAsync<GmCatalogResponse>("/products", ct);
                if (dto is null) continue;

                // 외부 dto → 도메인 입력 객체 (변환만)
                var input = new GmMasterDataInput(
                    Products: dto.Products.Select(p => new ProductDto(p.Id, p.Recipe, p.Weight)).ToList());
                await _writer.WriteAsync(input, ct);
            }
            catch (HttpRequestException)
            {
                // 두절 감지는 별도 (§5.11) — 이 Adapter는 입력만 흘림
            }
        }
    }
}
```

**특기 사항**: GM 입력은 도메인 Event 발행 없음·Transfer Trigger 없음·런타임 상태 갱신 없음(마스터 데이터 DB 영속화만, 본체 §5.1). State Service의 atomic 처리에 들어가지만 다소비처 분기가 0인 케이스.

#### 2.2.2 SM Adapter — REST Polling, CNC 작업대 센서

**외부 프로토콜**: REST API GET, 5초 주기 Polling.

**Payload 형태**: 모든 CNC 작업대 Slot의 입/출고 센서 ON/OFF 상태를 한 번에 반환.

```json
{
  "slots": [
    { "cnc_id": 1, "slot_id": 1, "in_occupied": true, "out_occupied": false },
    { "cnc_id": 1, "slot_id": 2, "in_occupied": false, "out_occupied": true },
    { "cnc_id": 2, "slot_id": 1, "in_occupied": false, "out_occupied": false }
  ]
}
```

**변환 패턴**: 응답에 담긴 모든 Slot 센서 상태를 1:1로 `SmSensorInput` 도메인 객체로 분해해 각각 Channel에 Write.

```csharp
public class SmAdapter : BackgroundService
{
    private readonly ChannelWriter<StateServiceInput> _writer;
    private readonly HttpClient _http;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
        while (await timer.WaitForNextTickAsync(ct))
        {
            try
            {
                var dto = await _http.GetFromJsonAsync<SmPollResponse>("/poll", ct);
                if (dto is null) continue;

                foreach (var slot in dto.Slots)
                {
                    // 한 Slot의 센서 상태 = 한 입력 객체
                    var input = new SmSensorInput(
                        slot.CncId, slot.SlotId, slot.InOccupied, slot.OutOccupied);
                    await _writer.WriteAsync(input, ct);
                }
            }
            catch (HttpRequestException) { }
        }
    }
}
```

**특기 사항**: 한 Polling 주기에 N Slot 입력이 동시 Write된다(Channel 직렬화로 순서 보존). 4상태 Mapping(빈 Slot/작업 대기/작업 시작/작업 종료)은 State Service 안에서 직전값과 비교해 수행(본체 §5.2). Adapter는 raw 센서값만 전달.

#### 2.2.3 WIP Adapter — REST 수신/응답, 양방향 의미 데이터

**외부 프로토콜**: REST API POST 수신, 응답 본문에 의미 데이터 회신.

**Payload 형태 — 수신**: WIP 중계 프로그램이 입/출고 시점에 QR 인식 결과 + Slot 센서 상태를 별도 Endpoint로 Push.

```json
// POST /wip/qr
{
  "wip_id": 1, "slot_id": 3, "unit_id": "7f3d9e2a-1b4c-4f8a-9d6e-5c2b3a7e1f8d", "restack_done": false,
  "request_id": "req-abc-123"
}

// POST /wip/sensor
{ "wip_id": 1, "slot_id": 3, "slot_state": "occupied" }
```

**Payload 형태 — 응답**: QR Endpoint의 응답 본문에 Unit 정보(되담기 Pair 판단용)를 회신.

```json
// 응답 본문
{
  "unit_id": "7f3d9e2a-1b4c-4f8a-9d6e-5c2b3a7e1f8d",
  "tray_shape": "Process",        // 공정 Tray | 적층 Tray
  "pair_unit_id": "2c6b1f84-9a3d-4e7c-b5f0-8d1e3a2c9b40",       // Pair Unit ID (있으면)
  "current_location": "WIP-1-3"
}
```

**변환 패턴**: 수신 측은 다른 Adapter와 동일하게 도메인 입력 객체 변환 + Channel Write. 응답 측은 별도 응답 채널(`IUnitInfoReplyService`)을 두어 State Service 처리 완료 후 회신 데이터를 받아 응답.

```csharp
[ApiController]
[Route("wip")]
public class WipAdapter : ControllerBase
{
    private readonly ChannelWriter<StateServiceInput> _writer;
    private readonly IUnitInfoReplyService _reply;

    [HttpPost("qr")]
    public async Task<UnitInfoResponse> OnQr(WipQrDto dto, CancellationToken ct)
    {
        var input = new WipQrInput(dto.WipId, dto.SlotId, dto.UnitId, dto.RestackDone, dto.RestackPlacements, dto.RequestId);
        await _writer.WriteAsync(input, ct);

        // 응답은 별도 채널 — State Service가 처리 완료 시 _reply에 결과 등록
        return await _reply.WaitForReply(dto.RequestId, ct);
    }

    [HttpPost("sensor")]
    public async Task OnSensor(WipSensorDto dto, CancellationToken ct)
    {
        var input = new WipSlotStateInput(dto.WipId, dto.SlotId, dto.SlotState);
        await _writer.WriteAsync(input, ct);
        // 센서 신호는 단순 확인 응답
    }
}

// State Service 측 — atomic 처리 끝에 응답 등록
public class StateService
{
    private readonly IUnitInfoReplyService _reply;

    private async Task ProcessWipQr(WipQrInput input)
    {
        // ... atomic 처리 ...
        var unitInfo = BuildUnitInfoResponse(input);
        _reply.PostReply(input.RequestId, unitInfo);  // 응답 채널에 결과 등록
    }
}
```

**특기 사항**: WIP Adapter만 응답 본문에 의미 있는 도메인 데이터를 실어 보낸다(다른 Adapter의 REST 응답은 단순 확인 응답). 응답 채널은 입력 Channel과 분리되어, 입력 Channel은 단방향 Write만 흘리고 응답은 별도 IUnitInfoReplyService가 RequestId 기준 매칭으로 처리한다.

#### 2.2.4 MQTT Adapter — MQTT 양방향, AMMR HW 통신

**외부 프로토콜**: MQTT pub/sub, 양방향(수신·발신).

**Payload 형태 — 수신**: AMMR HW가 Topic별로 다른 구조의 메시지를 publish.

```
Topic: ammr/{ammr_id}/state/hw       Payload: { "hw_state": "move" }
```

**Payload 형태 — 발신**: State Service 발신 게이트웨이로부터 받은 단위 지시 도메인 객체를 Payload로 변환해 publish.

```
Topic: core/ammr/{ammr_id}/job/cmd        Payload: { "job_id": 1234, "job_type": "move", "work_location": "42" }
```

**변환 패턴 — 수신**: Topic별로 다른 도메인 입력 객체 타입으로 분지하여 변환.

```csharp
public class MqttAdapter : BackgroundService
{
    private readonly ChannelWriter<StateServiceInput> _writer;
    private readonly IMqttClient _mqtt;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        _mqtt.OnMessageReceived += async (topic, payload) =>
        {
            // Topic별 분기 — Payload → 도메인 객체 변환
            StateServiceInput input = ParseTopic(topic, payload);
            await _writer.WriteAsync(input, ct);
        };

        await _mqtt.SubscribeAsync("ammr/+/#");
    }

    private static StateServiceInput ParseTopic(string topic, byte[] payload)
    {
        // Topic 파싱 — ammr/{ammr_id}/{kind}/...
        var parts = topic.Split('/');
        var ammrId = int.Parse(parts[1]);
        var kind = parts[2];

        return kind switch
        {
            "state"    => parts[3] switch
            {
                "hw"   => Deserialize<AmmrStateInput>(payload, ammrId),
                "slot" => Deserialize<AmmrSlotStateInput>(payload, ammrId),
                _      => throw new InvalidOperationException($"Unknown topic: {topic}")
            },
            "telemetry" => parts[3] switch
            {
                "pose" => Deserialize<AmmrPositionInput>(payload, ammrId),
                "bms"  => Deserialize<AmmrBmsInput>(payload, ammrId),
                _      => throw new InvalidOperationException($"Unknown topic: {topic}")
            },

            "job"      => Deserialize<AmmrJobResultInput>(payload, ammrId),
            _          => throw new InvalidOperationException($"Unknown topic: {topic}")
        };
    }
}
```

**변환 패턴 — 발신**: State Service의 발신 게이트웨이가 호출. 도메인 객체 → MQTT Payload 변환 후 publish.

```csharp
public interface IMqttPublishGateway
{
    Task PublishCommand(JobCommand cmd);
}

public class MqttAdapter : ..., IMqttPublishGateway
{
    public async Task PublishCommand(JobCommand cmd)
    {
        var topic = $"core/ammr/{cmd.AmmrId}/job/cmd";
        var payload = JsonSerializer.SerializeToUtf8Bytes(new
        {
            job_id = cmd.JobId, job_type = cmd.JobType.ToString().ToLowerInvariant(), work_location = cmd.WorkLocation
        });
        await _mqtt.PublishAsync(topic, payload);
    }
}
```

**특기 사항**: Topic 자체가 메시지 종류를 지시하므로 Topic 파싱이 변환 첫 단계. Payload JSON은 Topic별로 스키마가 다르므로 Topic별 deserialize 타입을 분기. AMMR별 last seen 추적도 MQTT Adapter의 책임(본체 §5.5·§5.11) — 단절 신호 발신은 별도 도메인 입력 타입(`AmmrDisconnectionInput` 등)으로 처리.

#### 2.2.5 Command API Controller — REST 사용자 명령

**외부 프로토콜**: REST API POST, Dashboard 사용자 명령.

**Payload 형태**: 명령 종류와 Payload.

```json

{
  "kind": "ManualUnitIdInput",
  "operator": "user01",
  "payload": {
    "slot_kind": "WIP",
```

**변환 패턴**: 명령 종류는 enum, Payload는 명령마다 다르므로 dictionary 또는 JsonElement 형태로 담아 도메인 입력 객체 변환.

```csharp
[ApiController]
```

### 2.3 도메인 객체 변환의 구조

6종 Adapter가 외부 프로토콜은 다르지만, 도메인 객체 변환 단계에서 공통 구조를 따른다.

**1. immutable record 사용**

도메인 입력 객체는 모두 `record` 타입(또는 `sealed record`)으로 정의. 채널 통과 중 변형이 차단되어 Adapter-State Service 사이의 의도 전달이 정확.

```csharp
public abstract record StateServiceInput
{
    public DateTimeOffset ReceivedAt { get; init; } = DateTimeOffset.UtcNow;
}

public record SmSensorInput(int CncId, int SlotId, bool InOccupied, bool OutOccupied)
    : StateServiceInput;
```

**2. ReceivedAt 공통 필드**

모든 입력은 Adapter에서 받은 시각을 보존(`DateTimeOffset.UtcNow` 기본값). 이력 DB 기록 시 입력 도착 시각으로 사용. 다중 입력의 도착 순서를 시각으로도 검증 가능.

**3. raw 데이터 손대지 않음**

Adapter는 외부 시스템이 보낸 raw 값을 그대로 옮긴다. 정규화·반올림·단위 변환·기본값 채움 등을 Adapter가 임의로 수행하면 State Service 측 결정 근거가 흐려진다. 단위 변환이 필요하면 별도 변환 단계를 명시적으로 표면화.

```csharp
// 부적절 — Adapter가 임의로 단위 변환
public class MqttAdapter
{
    private static AmmrPositionInput ParsePosition(byte[] payload, int ammrId)
    {
        var raw = Deserialize<RawPosition>(payload);
        // Adapter가 mm → m 임의 변환 (← State Service 결정 근거 흐려짐)
        return new AmmrPositionInput(ammrId, raw.X / 1000.0, raw.Y / 1000.0, raw.Heading);
    }
}

// 적절 — raw 그대로 전달, 변환은 State Service에서 명시적
public class MqttAdapter
{
    private static AmmrPositionInput ParsePosition(byte[] payload, int ammrId)
    {
        var raw = Deserialize<RawPosition>(payload);
        return new AmmrPositionInput(ammrId, raw.X, raw.Y, raw.Heading, raw.Unit);  // 방향각·단위 정보 유지
    }
}
```

**4. 외부 dto와 도메인 입력 객체 분리**

외부 시스템이 보낸 dto(`SmPollResponse`·`WipQrDto` 등)는 외부 시스템 스키마에 종속된 형태. 도메인 입력 객체는 Core 내부 구조로 정의. 둘이 같으면 외부 스키마 변경이 Core 내부에 직결되어 결합도가 올라간다.

```csharp
// 외부 dto — JSON 역직렬화 대상
public record SmPollResponse(IReadOnlyList<SmSlotDto> Slots);
public record SmSlotDto(int CncId, int SlotId, bool InOccupied, bool OutOccupied);

// 도메인 입력 객체 — Core 내부용
public record SmSensorInput(int CncId, int SlotId, bool InOccupied, bool OutOccupied)
    : StateServiceInput;
```

이 예에서는 두 타입의 필드가 일치하나, 외부 시스템 스키마가 변하면 dto만 변경하고 변환 단계에서 받아 처리 가능.

## 3. State Service 입력 Channel — 다형 객체 직렬화

### 3.1 단일 Channel에 다형 객체를 흘리는 구조

본체 §4.6은 State Service 입력 Channel을 다음과 같이 규정한다.

> System.Threading.Channels 기반, State Service가 단일 Consumer. Adapter 6종(GM/SM/General Process WIP/Restack Process WIP/MQTT/Command API) 입력을 모두 본 Channel에 전달하면 State Service가 FIFO 순 atomic 처리한다.

6종 Adapter가 보내는 도메인 입력 객체는 형태가 모두 다르다(SmSensorInput·WipQrInput·AmmrJobResultInput·CommandInput·GmMasterDataInput·...). 그런데 Channel은 type parameter `T` 한 개에 묶인 Queue다. 어떻게 한 Channel로 다형 객체를 흘릴까?

답은 **공통 base abstract record를 type parameter로 두기**.

```csharp
// Channel 정의
var channel = Channel.CreateUnbounded<StateServiceInput>(
    new UnboundedChannelOptions { SingleReader = true, SingleWriter = false });

// Adapter 측 — 자기 도메인 객체를 base 타입에 담아 Write
await _writer.WriteAsync(new SmSensorInput(1, 2, true, false));     // OK — 자식 타입은 base에 자동 캐스팅
await _writer.WriteAsync(new WipQrInput(1, 3, "U-001", false, null));     // OK
await _writer.WriteAsync(new AmmrJobResultInput(...));              // OK
await _writer.WriteAsync(new CommandInput(...));                    // OK
await _writer.WriteAsync(new GmMasterDataInput(...));               // OK

// State Service 측 — base 타입으로 받음, 실제 타입은 분기 단계에서 식별
await foreach (StateServiceInput input in _reader.ReadAllAsync(ct))
{
    // input의 실제 타입은 SmSensorInput | WipQrInput | ... 중 하나
    HandleByType(input);
}
```

이 구조의 핵심은 **공통 base + 다형 자식 = 단일 Channel로 직렬화 가능**.

### 3.2 sealed record 계층의 의미

base 타입을 그냥 `abstract record`가 아니라 `sealed record` 계층으로 두면 추가 구조가 따라온다.

```csharp
// 닫힌 계층 — 외부에서 새 자식 못 만듦
public abstract record StateServiceInput
{
    public DateTimeOffset ReceivedAt { get; init; } = DateTimeOffset.UtcNow;
}

public sealed record SmSensorInput(int CncId, int SlotId, bool InOccupied, bool OutOccupied)
    : StateServiceInput;

public sealed record WipQrInput(
    int WipId, int SlotId, string UnitId, bool RestackDone,
    IReadOnlyList<RestackPlacement>? RestackPlacements, string RequestId)   // RestackPlacements: 되담기 완료 시 적층 구성(어느 Tray에 무엇을 담았는지)
    : StateServiceInput;

// 되담기(Restack) 적층 구성 한 항목 — 한 Tray와 그 Tray에 담긴 Unit 목록
public sealed record RestackPlacement(int TrayId, string TrayShape, IReadOnlyList<string> UnitIds);

public sealed record WipSlotStateInput(int WipId, int SlotId, SlotState SlotState)   // 클라(WIP 중계) 판정 slot_state
    : StateServiceInput;

public sealed record AmmrStateInput(int AmmrId, HwState HwState)
    : StateServiceInput;

public sealed record AmmrPositionInput(int AmmrId, double X, double Y, double A, string Unit)   // A: 방향각(도, 0~360)
    : StateServiceInput;

public sealed record AmmrBmsInput(
    int AmmrId, string BatteryId, double Soc,
    double Current, double Temperature, double Voltage, bool BmuError) : StateServiceInput;

public sealed record AmmrSlotStateInput(int AmmrId, int SlotId, SlotState SlotState)   // 클라(AMMR 태블릿) 판정 slot_state (occupied/empty/blocked/job_failed) — job_failed→Block JobFailed
    : StateServiceInput;

public sealed record AmmrJobResultInput(
    int AmmrId, int JobId, JobType JobType, JobResult JobResult,
    FailureReason? Reason, AmmrSlotReport? Slot) : StateServiceInput;

// A-8 job/report 의 slot 객체 (Pickup·Dropoff 시 AMMR 자기 Slot 하나)
public sealed record AmmrSlotReport(int SlotId, SlotState SlotState, string? UnitId);

public sealed record AmmrDisconnectionInput(int AmmrId) : StateServiceInput;

public sealed record GmMasterDataInput(IReadOnlyList<ProductDto> Products) : StateServiceInput;

public sealed record CommandInput(CommandKind Kind, string Operator, IDictionary<string, object> Payload)
    : StateServiceInput;
```

**sealed의 효과**:

1. **자식 타입 전체 목록이 이 파일 안에 닫힘** — 외부 어셈블리가 새 자식을 만들 수 없으므로 Core 안에서 입력 종류가 폭발하지 않는다. 신규 입력 추가는 이 파일 편집을 강제 → 검토 통제.
2. **C# pattern matching의 exhaustiveness 검사 활용** — switch 표현식이 모든 자식을 처리했는지 컴파일러가 추론 가능(C# 9+의 sealed/closed type pattern). 자식 추가 누락 시 compile-time warning.
3. **JSON 직렬화 시 polymorphism 명시 가능** — `[JsonPolymorphic]`·`[JsonDerivedType]` 어노테이션으로 직렬화/역직렬화 구조 명시. 이 시스템에서는 InMemory Channel만 통과하므로 직렬화 무관이나, 디버깅 Log·dead-letter queue 등에 유용.

**계층 깊이**: 이 시스템에서는 한 단계만 둔다(`StateServiceInput` → 자식 N개). 두 단계 이상은 분기 패턴이 복잡해져 유지비가 늘어나므로 도입 신호 발생 시 별도 검토.

### 3.3 ChannelWriter/ChannelReader 분리 DI 패턴

상세는 DI와 동시성 해설 §2.12 참조. 핵심만 요약:

```csharp
// Program.cs — Channel 인스턴스는 한 개, Writer/Reader 분리 DI 등록
var channel = Channel.CreateUnbounded<StateServiceInput>(
    new UnboundedChannelOptions { SingleReader = true, SingleWriter = false });

builder.Services.AddSingleton<ChannelWriter<StateServiceInput>>(channel.Writer);
builder.Services.AddSingleton<ChannelReader<StateServiceInput>>(channel.Reader);

// Adapter들에는 Writer만 주입
builder.Services.AddSingleton<SmAdapter>();
builder.Services.AddSingleton<WipAdapter>();
builder.Services.AddSingleton<MqttAdapter>();
builder.Services.AddSingleton<GmAdapter>();
// CommandApiController는 ASP.NET Core가 자동 등록

// State Service에는 Reader만 주입
builder.Services.AddSingleton<StateService>();
builder.Services.AddHostedService<StateServiceWorker>();
```

분리의 효과 3종:

- **인터페이스 분리(ISP)**: Adapter는 Read 권한 없음(컴파일 타임 차단), State Service는 Write 권한 없음(self-write 차단)
- **의존성 역전(DIP)**: Adapter-State Service가 서로를 직접 결합하지 않고 Channel 추상에서만 만남
- **테스트 용이성**: Adapter 단위 테스트는 mock Writer만, State Service 단위 테스트는 mock Reader만

**이 시스템에서 이 패턴이 특히 중요한 이유**: 6종 Adapter가 동시에 Write하는 상황에서 어느 Adapter가 실수로 Read하면 다른 Adapter의 입력이 사라지는 사고가 발생할 수 있다. 컴파일 타임에 원천 차단되어야 한다.

## 4. State Service 측 처리 — 단일 Consumer + 타입별 분기 + atomic

### 4.1 단일 Consumer Pull 루프

State Service Worker는 BackgroundService로 기동되어 Channel Reader를 단일 Consumer로 무한 루프 처리한다.

> 표준 Consumer 루프 코드(`StateServiceWorker.ExecuteAsync` — `await foreach` + try/catch + `_state.Process(input)`)는 Channels 해설 §2.4에 전문을 둔다. 같은 코드를 두 곳에 두지 않으려 여기서는 전문을 생략하고 아래 핵심 구조만 짚는다.

**핵심 구조**:

- `await foreach` 한 바퀴 = atomic 단위 (Channels 해설 §2.5 참조)
- 처리 도중 다른 입력이 끼어들지 못함 → 시나리오 1(Interleave 상태 역전) 방어
- 한 입력 처리 예외가 발생해도 try/catch로 잡아 다음 입력으로 진행 → 단일 입력 실패가 전체 루프를 죽이지 않음
- `SingleReader = true` 옵션으로 내부 lock 제거되어 처리 핫패스 최적화

### 4.2 타입별 분기 — 이 문서의 핵심 부분

`State.Process(input)`이 호출되는 시점에 input은 `StateServiceInput` base 타입으로 들어온다. 실제 타입(`SmSensorInput`·`WipQrInput`·...)에 따라 처리 흐름이 갈라져야 한다.

```csharp
public class StateService
{
    public async Task Process(StateServiceInput input)
    {
        // 여기서 input의 실제 타입을 식별하고 해당 처리 로직으로 분기해야 함
        // ↓ 분기 구현 방식 후보 5종 (§5에서 비교)
    }
}
```

여기가 **분기 구현 방식 선택** 지점이다. 단순해 보이지만 후보가 많고, 선택에 따라 코드 구조·유지비·확장성·디버깅 비용이 달라진다. §5에서 5종 후보를 비교한다.

분기 후 각 처리 흐름은 본체 §5.1~§5.5의 다소비처 분기 표가 명시하는 구조를 따른다 — InMemory 갱신, 정합성 검사, 도메인 Event 발행, 이력 영속화 요청, SignalR push.

### 4.3 atomic 단위 경계와의 관계

분기 처리는 **atomic 단위 안에서 수행**된다. 즉:

```
[입력 N 도착]
    ↓
[Process(input) 호출 — atomic 단위 시작]
    ├─ ① 타입 분기 → 해당 처리 로직 진입
    ├─ ② InMemory 갱신
    ├─ ③ 정합성 검사
    ├─ ④ 도메인 Event 발행
    ├─ ⑤ 이력 영속화 Queue 적재
    └─ ⑥ SignalR push
[Process 반환 — atomic 단위 끝]
    ↓
[입력 N+1 도착] ← 비로소 다음 입력 꺼냄
```

분기 자체가 atomic 안의 한 단계이므로, **분기 구현 방식이 빠를수록 atomic 시간이 짧아진다**. atomic 시간이 짧을수록 다음 입력 처리 지연이 줄어든다(throughput 향상). 따라서 분기 구현 방식 선택 시 성능 차이도 평가 축에 포함.

다만 이 시스템 입력 빈도 추정(SM 5초·WIP 입출고 시 Event·MQTT 1초 위치 + 10초 BMS·N AMMR + N Slot)을 고려하면 분기 자체의 성능보다는 **유지보수 비용**이 결정 무게가 더 크다.

## 5. 분기 구현 방식 — 후보 5종 비교

### 5.1 평가 축

분기 구현 방식 선택의 평가 축 5종.

| 축 | 의미 |
|---|---|
| **확장성** | 신규 입력 타입 추가 시 수정 범위. 한 곳이면 OK, 여러 곳에 흩어지면 누락 위험. |
| **컴파일 안전성** | 자식 타입 처리 누락 시 컴파일러가 잡아주는가. exhaustiveness 검사 가능 여부. |
| **유지보수성** | 코드 가독성·디버깅 용이성·중복 줄임. |
| **성능** | 분기 자체의 핫패스 비용. 이 시스템에서는 우선순위 낮음. |
| **테스트 용이성** | 처리 로직 단위 테스트 분리 가능 여부. |

### 5.2 후보 (a) — switch + pattern matching

C# 9+의 switch 표현식 + sealed type pattern.

```csharp
public class StateService
{
    public async Task Process(StateServiceInput input)
    {
        var task = input switch
        {
            SmSensorInput sm           => ProcessSmSensor(sm),
            WipQrInput qr              => ProcessWipQr(qr),
            WipSlotStateInput ws       => ProcessWipSlotState(ws),
            AmmrStateInput ast         => ProcessAmmrState(ast),
            AmmrPositionInput ap       => ProcessAmmrPosition(ap),
            AmmrBmsInput ab            => ProcessAmmrBms(ab),
            AmmrSlotStateInput ass     => ProcessAmmrSlotState(ass),
            AmmrJobResultInput jr      => ProcessAmmrJobResult(jr),
            AmmrDisconnectionInput ad  => ProcessAmmrDisconnection(ad),
            GmMasterDataInput gm       => ProcessGmMasterData(gm),
            CommandInput cmd           => ProcessCommand(cmd),
            _                          => throw new InvalidOperationException(
                $"Unknown input type: {input.GetType().Name}")
        };
        await task;
    }

    private async Task ProcessSmSensor(SmSensorInput sm) { /* ... */ }
    private async Task ProcessWipQr(WipQrInput qr) { /* ... */ }
    // ... 자식 타입별 메서드
}
```

**평가**:

| 축 | 평가 |
|---|---|
| 확장성 | 새 타입 추가 시 switch 한 줄 + 메서드 한 개 추가. 한 곳 수정. |
| 컴파일 안전성 | sealed 계층 + C# 9+ switch 표현식은 누락 자식 타입에 대해 warning(CS8524 등). exhaustiveness 부분 지원. |
| 유지보수성 | 분기와 처리가 한 Class 안에 모여 가독성 좋음. 단점: State Service Class가 비대해질 수 있음. |
| 성능 | jump table 컴파일로 O(1) 분기. 가장 빠름. |
| 테스트 용이성 | 각 Process 메서드를 internal로 노출하면 단위 테스트 가능. private이면 통합 테스트 의존. |

**이 시스템에의 적합도**: 중상. 단순하고 빠르며 가독성이 좋다. 가장 큰 단점은 State Service Class 비대화이나, 처리 메서드를 partial class로 분리(`StateService.SmSensor.cs`·`StateService.WipQr.cs` 등)하면 완화 가능.

### 5.3 후보 (b) — Visitor 패턴

이중 디스패치(double dispatch) 패턴. 입력 객체에 `Accept(IVisitor)` 메서드를 두고, Visitor가 자식 타입별 처리를 가진다.

```csharp
public abstract record StateServiceInput
{
    public abstract Task Accept(IStateServiceInputVisitor visitor);
}

public sealed record SmSensorInput(int CncId, int SlotId, bool InOccupied, bool OutOccupied)
    : StateServiceInput
{
    public override Task Accept(IStateServiceInputVisitor visitor)
        => visitor.VisitSmSensor(this);
}

public sealed record WipQrInput(...) : StateServiceInput
{
    public override Task Accept(IStateServiceInputVisitor visitor)
        => visitor.VisitWipQr(this);
}
// ... 다른 자식

public interface IStateServiceInputVisitor
{
    Task VisitSmSensor(SmSensorInput input);
    Task VisitWipQr(WipQrInput input);
    Task VisitWipSlotState(WipSlotStateInput input);
    Task VisitAmmrState(AmmrStateInput input);
    // ... 자식별 메서드 11개
}

public class StateService : IStateServiceInputVisitor
{
    public async Task Process(StateServiceInput input)
    {
        await input.Accept(this);  // 이중 디스패치 — input 타입에 따라 자동 분기
    }

    public Task VisitSmSensor(SmSensorInput input) { /* ... */ }
    public Task VisitWipQr(WipQrInput input) { /* ... */ }
    // ... 자식별 처리 메서드
}
```

**평가**:

| 축 | 평가 |
|---|---|
| 확장성 | 새 타입 추가 시 IVisitor 인터페이스 + 모든 구현체에 메서드 추가. 여러 곳 수정. |
| 컴파일 안전성 | IVisitor에 메서드 추가 시 모든 구현체가 강제 구현. **누락 100% 컴파일 오류**. exhaustiveness 보장. |
| 유지보수성 | record에 Accept 메서드 추가가 강제되어 입력 객체 정의가 무거워짐. Visitor 인터페이스가 거대해짐(메서드 11개+). |
| 성능 | 가상 호출 한 번. switch와 거의 동등. |
| 테스트 용이성 | IVisitor 구현체를 mock으로 대체해 단위 테스트 가능. 좋음. |

**이 시스템에의 적합도**: 중. 컴파일 안전성은 가장 강하나 record에 Accept 메서드 추가 강제가 boilerplate 비용. C# 11+의 abstract method on record 지원이 있어 문법은 깔끔하지만, 입력 객체가 처리 책임을 직접 가져 Adapter 책임 원칙(데이터만)과 미세하게 어긋난다(record는 순수 데이터여야 함).

### 5.4 후보 (c) — MediatR

[MediatR](https://github.com/jbogard/MediatR) 라이브러리의 IRequest/IRequestHandler 패턴. 입력 객체가 IRequest로 정의되고, 각 Handler가 해당 입력 타입을 처리.

```csharp
public abstract record StateServiceInput : IRequest;  // MediatR IRequest 상속

public sealed record SmSensorInput(int CncId, int SlotId, bool InOccupied, bool OutOccupied)
    : StateServiceInput;

public sealed record WipQrInput(...) : StateServiceInput;
// ...

// Handler — 입력 타입별 한 Class
public class SmSensorHandler : IRequestHandler<SmSensorInput>
{
    private readonly StateService _state;
    public SmSensorHandler(StateService state) { _state = state; }

    public async Task Handle(SmSensorInput input, CancellationToken ct)
    {
        // ... 처리 로직
    }
}

public class WipQrHandler : IRequestHandler<WipQrInput>
{
    // ...
}
// ... 자식별 Handler 11개

// Worker
public class StateServiceWorker : BackgroundService
{
    private readonly ChannelReader<StateServiceInput> _reader;
    private readonly IMediator _mediator;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        await foreach (var input in _reader.ReadAllAsync(ct))
        {
            await _mediator.Send(input, ct);  // MediatR이 타입별 Handler 자동 분기
```

**평가**:

| 축 | 평가 |
|---|---|
| 확장성 | 새 타입 추가 시 record + Handler 한 Class 추가. 한 곳 수정(자동 등록). |
| 컴파일 안전성 | IRequestHandler<T>가 등록 안 되면 **런타임 오류** — MediatR이 reflection 기반이라 컴파일러는 모름. exhaustiveness 미보장. |
| 유지보수성 | Handler가 입력 타입당 한 Class로 분리되어 코드 분리도 좋음. 외부 의존성 추가(MediatR 라이브러리). |
| 성능 | reflection 기반 dispatch. switch보다 느림(미세 차이). |
| 테스트 용이성 | Handler를 단독 단위 테스트 가능. 가장 좋음. |

**이 시스템에의 적합도**: 하중. 코드 분리도는 좋으나 **컴파일 안전성 없음**이 이 시스템 구조와 어긋난다. 이 시스템은 입력 종류가 닫혀 있고(sealed) 신규 추가가 검토 통제 대상인데, MediatR은 Handler 등록 누락이 런타임에야 드러난다. 또한 외부 라이브러리 의존성을 입력 분기에 들이는 것은 구조의 단순성에서 멀어진다 — Channels 해설 §4.3에서도 동기 in-process 구조로 atomic 단위 안에서 구독자 처리까지 대기하는 구조의 부적합성이 지적되었다(이 경우는 다른 구조이지만, MediatR의 reflection·런타임 의존 구조는 동일).

다만 MediatR을 도메인 Event 발행 측에 쓰는 것은 별개 결정 — 이 경우는 Adapter → State Service 분기 지점.

### 5.5 후보 (d) — Dictionary 기반 lookup

`Dictionary<Type, Func<StateServiceInput, Task>>`로 타입별 Handler를 등록.

```csharp
public class StateService
{
    private readonly Dictionary<Type, Func<StateServiceInput, Task>> _handlers;

    public StateService()
    {
        _handlers = new()
        {
            [typeof(SmSensorInput)]          = i => ProcessSmSensor((SmSensorInput)i),
            [typeof(WipQrInput)]             = i => ProcessWipQr((WipQrInput)i),
            [typeof(WipSlotStateInput)]      = i => ProcessWipSlotState((WipSlotStateInput)i),
            [typeof(AmmrStateInput)]         = i => ProcessAmmrState((AmmrStateInput)i),
            [typeof(AmmrPositionInput)]      = i => ProcessAmmrPosition((AmmrPositionInput)i),
            [typeof(AmmrBmsInput)]           = i => ProcessAmmrBms((AmmrBmsInput)i),
            [typeof(AmmrSlotStateInput)]     = i => ProcessAmmrSlotState((AmmrSlotStateInput)i),
            [typeof(AmmrJobResultInput)]     = i => ProcessAmmrJobResult((AmmrJobResultInput)i),
            [typeof(AmmrDisconnectionInput)] = i => ProcessAmmrDisconnection((AmmrDisconnectionInput)i),
            [typeof(GmMasterDataInput)]      = i => ProcessGmMasterData((GmMasterDataInput)i),
            [typeof(CommandInput)]           = i => ProcessCommand((CommandInput)i),
        };
    }

    public Task Process(StateServiceInput input)
    {
        if (_handlers.TryGetValue(input.GetType(), out var handler))
            return handler(input);
        throw new InvalidOperationException($"Unknown input type: {input.GetType().Name}");
    }

    private Task ProcessSmSensor(SmSensorInput sm) { /* ... */ }
    // ... 자식별 메서드
}
```

**평가**:

| 축 | 평가 |
|---|---|
| 확장성 | 새 타입 추가 시 dictionary 항목 + 메서드 추가. 두 곳 수정. |
| 컴파일 안전성 | dictionary 등록 누락이 **런타임 오류**. exhaustiveness 미보장. |
| 유지보수성 | 등록부와 처리부가 분리되어 가독성 떨어짐. switch 대비 장점 적음. |
| 성능 | dictionary lookup O(1). switch jump table보다 미세 느림. |
| 테스트 용이성 | switch와 동일. |

**이 시스템에의 적합도**: 하. switch 대비 장점이 거의 없다. 동적 등록(런타임에 Handler 교체)이 필요하면 의미가 있으나 이 시스템에는 그런 요구 없음. 컴파일 안전성도 switch보다 약하다.

### 5.6 후보 (e) — Strategy + DI 다중 등록 (IInputHandler\<T\>)

각 입력 타입별 Handler를 IInputHandler\<T\>로 정의하고 DI 다중 등록.

```csharp
public interface IInputHandler<in T> where T : StateServiceInput
{
    Task Handle(T input);
}

public class SmSensorHandler : IInputHandler<SmSensorInput>
{
    private readonly StateService _state;
    public SmSensorHandler(StateService state) { _state = state; }

    public Task Handle(SmSensorInput input) { /* ... */ }
}

public class WipQrHandler : IInputHandler<WipQrInput>
{
    // ...
}
// ... 자식별 Handler 11개

// 분기 Adapter — Handler 호출
public class StateServiceDispatcher
{
    private readonly IServiceProvider _sp;
    public StateServiceDispatcher(IServiceProvider sp) { _sp = sp; }

    public Task Process(StateServiceInput input)
    {
        // 런타임 reflection으로 IInputHandler<{실제 타입}> 조회
        var handlerType = typeof(IInputHandler<>).MakeGenericType(input.GetType());
        var handler = _sp.GetService(handlerType);
        if (handler is null)
            throw new InvalidOperationException($"No handler for {input.GetType().Name}");

        // 동적 호출
        var method = handlerType.GetMethod(nameof(IInputHandler<StateServiceInput>.Handle))!;
        return (Task)method.Invoke(handler, new object[] { input })!;
    }
}

// DI 등록 — 각 Handler 등록
builder.Services.AddSingleton<IInputHandler<SmSensorInput>, SmSensorHandler>();
builder.Services.AddSingleton<IInputHandler<WipQrInput>, WipQrHandler>();
// ... 11개 등록
```

**평가**:

| 축 | 평가 |
|---|---|
| 확장성 | 새 타입 추가 시 record + Handler + DI 등록. 세 곳 수정. |
| 컴파일 안전성 | DI 등록 누락이 **런타임 오류**. exhaustiveness 미보장. |
| 유지보수성 | Handler가 분리되어 단일 책임 좋으나 reflection 기반 dispatch가 디버깅 어려움. |
| 성능 | reflection 호출 — 가장 느림. 캐싱하면 개선 가능. |
| 테스트 용이성 | Handler를 단독 단위 테스트 가능. 좋음. |

**이 시스템에의 적합도**: 하. MediatR의 수제 버전에 가까우며 외부 라이브러리 의존이 없다는 장점은 있으나 reflection 비용·컴파일 안전성 없음이 이 시스템에 부적합.

다만 DI 다중 등록 + 분기 지점에 keyed DI(.NET 8+의 `[FromKeyedServices]`)를 활용하면 reflection 없이 컴파일 안전성을 일부 회복할 수 있다. 그러나 그 구조로 가면 switch와 차이가 좁혀져 switch 대비 장점이 사라진다.

### 5.7 비교 정리표·이 시스템 권장 구조

**※ 후보 5종 비교**

| 후보 | 확장성 | 컴파일 안전성 | 유지보수성 | 성능 | 테스트 용이성 | 외부 의존 |
|---|---|---|---|---|---|---|
| (a) switch + pattern matching | 한 곳 수정 | warning 수준 | 비대화 위험 (partial class로 완화) | O(1) jump table | 중 | 없음 |
| (b) Visitor | 여러 곳 수정 | **컴파일 강제** | record boilerplate | 가상 호출 | 좋음 | 없음 |
| (c) MediatR | 한 곳 수정 (자동) | **런타임** | Handler 분리 좋음 | reflection | 좋음 | MediatR 라이브러리 |
| (d) Dictionary lookup | 두 곳 수정 | **런타임** | switch보다 떨어짐 | dict lookup | 중 | 없음 |
| (e) Strategy + DI 다중 | 세 곳 수정 | **런타임** | Handler 분리 좋으나 reflection | reflection 비용 | 좋음 | 없음 |

**※ 이 시스템 권장 구조 — (a) switch + pattern matching**

이 시스템에서는 (a)를 메인으로 권장한다. 근거:

1. **입력 종류가 닫혀 있다(sealed)** — 신규 입력은 검토 통제 대상이라 외부 어셈블리에서 자유롭게 추가될 일 없음. exhaustiveness가 사실상 sealed로 통제되며, switch warning이 보조 안전장치.
2. **분기 비용이 핫패스에 직접 위치** — atomic 단위 안의 한 단계라 switch jump table의 O(1)이 자연스럽게 유리.
3. **외부 라이브러리 의존 회피** — 이 시스템은 ASP.NET Core 표준 위에 두는 구조를 유지(MQTTnet·EF Core 등 필수 외에는 추가 의존 최소화).
4. **State Service Class 비대화는 partial class로 완화** — `StateService.SmSensor.cs`·`StateService.WipQr.cs`·`StateService.Ammr.cs` 등으로 파일 분리하면 가독성 회복.
5. **유지보수자 학습 부담 최소** — Visitor·MediatR·DI 다중 등록은 패턴 학습이 필요하나 switch는 C# 표준 문법.

**도입 신호 — (b) Visitor 또는 (c) MediatR로 전환 검토**

다음 신호가 발생하면 후보 (b) 또는 (c)로 전환을 검토한다.

- 입력 타입이 25종을 넘어 switch가 한 화면에 들어오지 않게 됨 → (a) partial class로도 한계, (b)·(c) 분리 검토
- Handler 단위 테스트 분리 요구가 강해짐(Handler별 mock·통합 테스트 분리 등) → (c) MediatR
- Plugin 구조로 외부 어셈블리에서 입력 추가 요구(현재 없음) → (c) MediatR(자동 등록) 또는 (e) Strategy

**도입 신호 — 변경 비용 평가**

전환 비용은 (a) → (c)가 가장 작고(switch 본문을 Handler 11개로 분해하면 됨), (a) → (b)가 가장 큼(record에 Accept 메서드 추가 강제). 따라서 신호 발생 시 (c)부터 검토.

이 시스템 현재 입력 종류는 11종 수준이라 (a)로 충분하며, 신호 발생 전에는 구조 단순성을 우선한다.

## 6. 6종 Adapter 결합 — 입력 Channel 단일 진입점

### 6.1 결합 구조 — Writer 단일 추상 + sealed record 분지

6종 Adapter(GM/SM/General Process WIP/Restack Process WIP/MQTT/Command API)가 모두 동일 추상(`ChannelWriter<StateServiceInput>`)에 의존한다. 외부 프로토콜 차이는 Adapter 안에서 처리되고, Channel 진입 후에는 sealed record 자식 타입으로만 식별된다.

```
[GM Adapter]      [SM Adapter]      [WIP Adapter]      [MQTT Adapter]      [Command API]
     │                 │                  │                   │                    │
GmMasterDataInput  SmSensorInput      WipQrInput          AmmrStateInput      CommandInput
                                      WipSlotStateInput   AmmrPositionInput
                                                          AmmrBmsInput
                                                          AmmrSlotStateInput
                                                          AmmrJobResultInput
                                                          AmmrDisconnectionInput
     │                 │                  │                   │                    │
     ↓                 ↓                  ↓                   ↓                    ↓
                ChannelWriter<StateServiceInput> (단일 추상)
                                       │
                                       ↓
                  [State Service 입력 Channel — 단일 인스턴스]
                                       │
                                       ↓
                ChannelReader<StateServiceInput> (단일 추상)
                                       │
                                       ↓
              [State Service Worker — 단일 Consumer + 타입별 분기]
```

**핵심 구조**: Adapter는 Writer 한 인터페이스에만 의존, State Service는 Reader 한 인터페이스에만 의존, 둘 사이는 sealed record 계층으로 표현된 도메인 입력 객체만 흐른다. 이로써:

- Adapter 6종이 서로를 모름 — 한 Adapter의 구조 변경이 다른 Adapter에 전이 안 됨
- State Service가 Adapter 6종을 모름 — Adapter 추가/제거가 State Service 코드 변경 강제 안 함(분기 지점에 자식 추가만 강제)
- §1.1 단일 Port 구현 교체 구조의 자연스러운 적용 — 운영/시뮬/테스트 환경별 Adapter 교체가 Writer 인터페이스 차원에서 투명하게 일어남(DI와 동시성 해설 §1.5 참조)

### 6.2 발신 결과의 비대칭 — MQTT 양방향과 SignalR Hub

수신 측 6종 Adapter는 위와 같이 단일 추상에 결합되나, **발신 측은 구조가 다르다**.

본체 §4.4·§5.6 [발신]·§5.10에서 명시되듯, State Service는 외부 인터페이스 게이트웨이(DB 영속화·SignalR Push·MQTT 발신)의 단일 경로다. 발신 측 Adapter는 다음 두 종.

| Adapter | 발신 구조 | State Service 측 호출 인터페이스 |
|---|---|---|
| MQTT Adapter | AMMR HW 단위 지시 송신 | `IMqttPublishGateway.PublishCommand(JobCommand)` |
| SignalR Hub | Dashboard 실시간 push | `ISignalRPushGateway.Push(DashboardSnapshot)` |

발신 측은 **State Service가 Adapter를 능동 호출**하는 구조라 입력 Channel을 거치지 않는다. atomic 단위 안에서 직접 호출.

```csharp
public class StateService
{
    private readonly IMqttPublishGateway _mqtt;
    private readonly ISignalRPushGateway _signalR;

    public async Task Process(StateServiceInput input)
    {
        // ① InMemory 갱신
        var changes = UpdateInMemory(input);

        // ② 정합성 검사
        var integrity = CheckIntegrity(changes);

        // ③ 도메인 Event 발행 (Transfer/AMMR Service에 전달)
        var events = DeriveDomainEvents(changes, integrity);
        foreach (var evt in events) await _bus.Publish(evt);

        // ④ 이력 영속화 Queue 적재
        _history.Enqueue(BuildHistoryRecord(input, changes, integrity));

        // ⑤ SignalR push (자기 권위 변화)
        await _signalR.Push(BuildSnapshot(changes));

        // ⑥ MQTT 발신 (AMMR Service의 단위 지시 결정 결과 — 본 input 처리에서 발생한 경우)
        // 단, 단위 지시 결정은 별도 atomic 단위에서 발생 (AMMR Service → State Service 발신 게이트웨이 호출 경로)
    }
}
```

**비대칭의 의미**: 입력 Channel은 5종 Adapter의 **다대일 직렬화 구조**, 발신은 State Service의 **일대다 발산 구조**. 둘은 다른 방식의 자료구조를 가진다(전자는 Channel, 후자는 직접 호출 + 인터페이스 분리).

발신 게이트웨이도 Writer/Reader처럼 인터페이스 분리 가능하나, 발신은 단방향 호출이라 분리 효용이 적다. 인터페이스(`IMqttPublishGateway`·`ISignalRPushGateway`)만 두면 충분.

### 6.3 Adapter 두절 신호의 합류

각 Adapter는 자기 외부 시스템 가용성을 독립 감시한다(본체 §4.4 마지막 단락·§5.11). 두절 감지 시 두절 사실을 도메인 입력 객체로 변환해 동일 입력 Channel에 흘려 보낸다.

```csharp
// MQTT Adapter — AMMR별 last seen 추적
public class MqttAdapter : BackgroundService
{
    private readonly Dictionary<int, DateTimeOffset> _lastSeen = new();
    private readonly ChannelWriter<StateServiceInput> _writer;

    private async Task CheckLastSeen(CancellationToken ct)
    {
        var threshold = DateTimeOffset.UtcNow - TimeSpan.FromSeconds(15);
        foreach (var (ammrId, lastAt) in _lastSeen.ToArray())
        {
            if (lastAt < threshold)
            {
                // 단절 신호 — 도메인 입력 객체로 흘려 보냄
                await _writer.WriteAsync(new AmmrDisconnectionInput(ammrId), ct);
                _lastSeen.Remove(ammrId);
            }
        }
    }
}
```

**핵심 구조**: 두절 신호도 일반 입력과 동일 Channel·동일 추상으로 흐른다. State Service는 `AmmrDisconnectionInput` 자식 타입으로 분기하여 atomic 안에서 InMemory 통째 NULL 초기화 + Block 마킹 + Job/Transfer 종료 일괄 처리(본체 §5.5 코드블록·§5.6·§5.11). 두절 처리도 일반 입력과 같은 직렬화 구조를 따르므로, 두절 직전 도착한 정상 입력은 두절 신호 이전에 atomic 처리되어 정합성이 깨지지 않는다.

각 Adapter의 두절 감지 수단은 다르다(REST Timeout·Last Will·HealthCheck 미수신 등, 본체 §5.11 표 참조). 그러나 감지 후 처리는 모두 도메인 입력 객체 발신으로 정규화된다.

## 7. 코드 스케치 — 한 입력의 여정

§7은 한 시나리오를 정해 시작에서 분기 처리 끝까지 따라가는 통합 예시다. 앞 절에서 다룬 결들이 실제 흐름에서 어떻게 맞물리는지 보인다.

### 7.1 시나리오 — AMMR Pickup 성공 보고

**상황**: AMMR-1이 WIP-2 Slot-3에 있던 Unit 7f3d9e2a-1b4c-4f8a-9d6e-5c2b3a7e1f8d를 자기 Slot-2에 Pickup 성공. AMMR HW가 결과를 MQTT로 publish.

**기대 결과**:

- State Service InMemory에서 WIP-2 Slot-3의 식별값이 NULL로 갱신, AMMR-1 Slot-2의 식별값이 7f3d9e2a-1b4c-4f8a-9d6e-5c2b3a7e1f8d로 갱신(추정 등급)
- Job 이력 + Unit 위치 변경 이력 DB 기록 Queue 적재
- `JobResultEvent` 도메인 Event 발행 (Slot 점유 변화는 별도 Event 아닌 클라 판정 slot_state 보고로 반영)
- AMMR Service가 다음 Job(Move) 결정
- Dashboard에 변경분 push

### 7.2 시작 — MQTT Payload 도착

```
Payload: {
```

### 7.3 Adapter — 파싱·변환·Write

```csharp
public class MqttAdapter : BackgroundService

    private async Task OnMqttMessage(string topic, byte[] payload, CancellationToken ct)
    {

        var parts = topic.Split('/');
        var ammrId = int.Parse(parts[1]);
        var kind = parts[2];

        if (kind == "job" && parts[3] == "report")
        {

                JobResult: Enum.Parse<JobResult>(dto.JobResult, ignoreCase: true),

                Slot: dto.Slot);
            await _writer.WriteAsync(input, ct);

            // last seen 갱신

        }

    }
```

### 7.4 Channel 통과

`AmmrJobResultInput` 인스턴스가 Channel Queue에 enqueue된다. `SingleReader = true` 옵션으로 State Service Worker만 dequeue 가능. 다른 Adapter의 동시 입력이 있으면 Channel이 도착 순서를 보존.

### 7.5 State Service — 분기·atomic·Event 발행

State Service Worker가 dequeue → State.Process() 호출 → switch 분기 → ProcessAmmrJobResult() 진입.

```csharp
// === StateService.Ammr.cs (partial class) ===
public partial class StateService
{
    private async Task ProcessAmmrJobResult(AmmrJobResultInput input)
    {
        // === atomic 단위 시작 ===

        // ① InMemory 갱신
        var changes = new List<StateChange>();

        if (input.JobResult == JobResult.Success)
        {
            switch (input.JobType)
            {
                case JobType.Pickup:
                    // A-8 job/report는 AMMR 자기 Slot 하나만 싣는다 (출발 외부 Slot은 없음)
                    var slot = input.Slot!;
                    var unitId = slot.UnitId!;

                    // 출발 Slot(외부 WIP) 식별값 NULL 갱신 — 출발 위치는 이 Unit의 직전 InMemory 위치로 되짚는다(Core 권위)
                    var src = (WipSlotLocation)_units[unitId].Location;
                    var sourceSlot = _slots[(SlotKind.Wip, src.WipId, src.SlotId)];
                    _slots[(SlotKind.Wip, src.WipId, src.SlotId)] =
                        sourceSlot with { UnitId = null };
                    changes.Add(new SlotIdChange(SlotKind.Wip, src.WipId, src.SlotId, null));

                    // AMMR Slot 식별값 갱신 (추정 등급 — AMMR HW 동작 보고)
                    var ammrSlot = _ammrSlots[(input.AmmrId, slot.SlotId)];
                    _ammrSlots[(input.AmmrId, slot.SlotId)] =
                        ammrSlot with { UnitId = unitId, Grade = IdGrade.Estimated };
                    changes.Add(new AmmrSlotIdChange(input.AmmrId, slot.SlotId, unitId));

                    // Unit 위치 갱신 (AMMR Slot N)
                    _units[unitId] = _units[unitId] with
                    {
                        Location = new AmmrSlotLocation(input.AmmrId, slot.SlotId)
                    };
                    changes.Add(new UnitLocationChange(unitId, ...));
                    break;

                case JobType.Dropoff:  // ... 별도 분기
                case JobType.Move:     // ... 별도 분기
                case JobType.Charge:   // ... 별도 분기
            }
        }

        // ② 정합성 검사 (§5.7) — 이 변경분에 대해
        var integrity = CheckIntegrity(changes);

        // ③ 도메인 Event 도출
        var events = new List<DomainEvent>
        {
            new JobResultEvent(
                input.AmmrId, input.JobId, input.JobType, input.JobResult,
                input.Slot?.UnitId, input.Reason),
        };
        // Slot 점유 변화(AMMR Slot 점유·출발 Slot 비움)는 클라이언트 판정 slot_state 입력으로 반영되며, InMemory 갱신(①)과 SignalR push(⑥)로 Dashboard에 전달된다.
        // Block 마킹은 자기소비 DB 이력으로 남긴다(구독자 없음).
        foreach (var blocked in integrity.NewlyBlocked)
            _history.Enqueue(new SlotBlockedHistoryRecord(blocked.WipId, blocked.SlotId, blocked.Kind, blocked.Reason));

        // ④ 도메인 Event 발행 (도메인 Event 버스 — Channels 해설 §4.2 갈래 b)
        foreach (var evt in events)
            await _bus.Publish(evt);

        // ⑤ 이력 영속화 Queue 적재 (비동기 Worker 위임 — DB 왕복 비대기)
        _history.Enqueue(new JobResultHistoryRecord(
            input.AmmrId, input.JobId, input.JobType, input.JobResult, input.Slot?.UnitId, input.Reason));
        if (input.JobResult == JobResult.Success && (input.JobType == JobType.Pickup || input.JobType == JobType.Dropoff))
        {
            _history.Enqueue(new UnitLocationChangedHistoryRecord(input.Slot!.UnitId!, ...));
        }

        // ⑥ SignalR Hub push (변경분 + 이력)
        await _signalR.Push(BuildSnapshot(changes, events));

        // === atomic 단위 끝 ===
    }
}
```

### 7.6 후속 — AMMR Service 구독자 처리

`JobResultEvent`가 도메인 Event 버스로 발행되면, AMMR Service가 자기 Channel Reader로 비동기 수신.

```csharp
public class AmmrServiceWorker : BackgroundService
{
    private readonly ChannelReader<JobResultEvent> _jobResult;
    private readonly AmmrService _service;

    protected override async Task ExecuteAsync(CancellationToken ct)
    {
        await foreach (var evt in _jobResult.ReadAllAsync(ct))
            await _service.OnJobResult(evt);
    }
}

public class AmmrService
{
    private readonly IStateServicePullApi _state;
    private readonly IMqttPublishGateway _mqtt;

    public async Task OnJobResult(JobResultEvent evt)
    {
        if (evt.Success)
        {
            // 다음 Job 결정 (§5.6 결과 수신 처리 표)
            var ammr = _state.GetAmmrById(evt.AmmrId);   // Pull (시점 검증 — §5.6 조회 채널 원칙)
            var nextJob = _jobSequencer.Next(ammr, evt);

            if (nextJob != null)
            {
                // Pickup 직전 검증 등 (§5.6) — Pull로 시점 검증
                if (await VerifyBeforeJob(nextJob))
                {
                    // State Service 발신 게이트웨이 호출 (단일 게이트웨이 — §4.5)
                    await _state.IssueAmmrCommand(nextJob);
                    // State Service가 atomic 안에서 Job 발행 이력 기록 + SignalR push + MQTT publish 일괄
                }
                else
                {
                    await HandleVerificationFail(nextJob);
                }
            }
            else
            {
                // Transfer 완료 — State Service에 사실 전달
                await _state.RecordTransferCompleted(ammr.CurrentTransfer!);
                _state.ReturnToAmmrList(ammr.AmmrId);  // AmmrList 복귀
            }
        }
        else
        {
            // 실패 처리 — §5.6 결과 수신 처리 표 분기
            await HandleJobFailure(evt);
        }
    }
}
```

**전체 여정 요약**:

```
[AMMR HW] MQTT publish
    ↓
[MQTT Adapter] Topic 파싱 → AmmrJobResultInput 변환 → ChannelWriter.WriteAsync
    ↓
[State Service 입력 Channel] FIFO 직렬화 (다형 객체)
    ↓
[State Service Worker] ChannelReader.ReadAllAsync
    ↓
[State Service.Process] switch 분기 → ProcessAmmrJobResult
    ├─ ① InMemory 갱신 (Slot 식별값·Unit 위치)
    ├─ ② 정합성 검사
    ├─ ③ 도메인 Event 도출 (JobResultEvent)
    ├─ ④ 도메인 Event 버스 Publish (fan-out)
    ├─ ⑤ 이력 영속화 Queue 적재
    └─ ⑥ SignalR push
    ↓ (atomic 단위 끝, 다음 입력 처리 가능)
[도메인 Event 버스] Dispatcher → 구독자별 Channel
    ↓
[AMMR Service Worker] JobResultEvent 구독 → OnJobResult
    ├─ Pull 조회 (시점 검증)
    ├─ 다음 Job 결정
    ├─ Pickup 직전 검증
    └─ State Service 발신 게이트웨이 호출 → MQTT publish (다음 단위 지시)
```

이 한 흐름이 이 시스템의 Adapter 입력 처리의 전형이다. 다른 입력(SmSensorInput·WipQrInput·CommandInput 등)도 같은 구조를 따른다 — Adapter에서 도메인 객체로 변환 → 입력 Channel → State Service에서 타입별 분기 → atomic 처리 → 도메인 Event 발행 → 구독자 후속 처리.
