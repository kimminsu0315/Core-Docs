## 1. Event Queue vs 락 (Lock)

### 배경
Core는 여러 WIP Event를 동시에 받을 수 있다. 동시에 여러 Event가 들어왔을 때 Pipeline이 중복 실행되면 같은 AMMR이나 같은 Slot에 Job이 중복 배정되는 문제가 생긴다. 이를 방지하기 위한 두 가지 방법이 있다.

### Event Queue 방식 (우리 시스템 채택)

Event가 들어오는 순서대로 줄을 세워서 하나씩 처리하는 방식.

**예시: 동시에 두 개의 Event 발생**
- 세척 WIP 출고 (U-001)
- CNC 작업대 출고 (U-002)
- AMMR 1호기, 2호기 대기 중

```
1. 세척 WIP 출고 Event → Queue 입력
2. CNC 작업대 출고 Event → Queue 입력
3. Core가 Queue에서 1번 꺼냄 → Pipeline 실행 → AMMR 1호기에 U-001 Job 배정
4. Core가 Queue에서 2번 꺼냄 → Pipeline 실행 → AMMR 2호기에 U-002 Job 배정
```

→ 순서대로 처리하니까 중복 배정 없음.

**장점:** 구현이 단순, 중복 배정 문제 없음  
**단점:** Event가 많이 몰리면 처리 지연 발생 (이 시스템 규모에서는 체감 없음)

### 락 방식

Pipeline 실행 중에 다른 Event가 못 들어오게 잠그는 방식.

**예시:**
```
1. 세척 WIP 출고 Event → Pipeline 실행 시작 → [문 잠금]
2. CNC 작업대 출고 Event 도착 → 문이 잠겨있어서 [대기]
3. AMMR 1호기에 U-001 Job 배정 완료 → [문 열림]
4. CNC 작업대 출고 Event → Pipeline 실행 → AMMR 2호기에 U-002 Job 배정
```

**장점:** 실시간성 좋음  
**단점:** 구현 복잡, Deadlock 위험

> **Deadlock이란?** Pipeline A가 잠근 상태에서 Pipeline B를 기다리고, B도 A를 기다리면 둘 다 영원히 못 끝나는 상황.

### 락이 필요한 경우

Queue와 락은 경쟁 개념이 아니다. Queue로 Event 순서를 잡으면서, DB 데이터를 읽고 쓸 때는 락으로 데이터 정합성을 보호하는 방식으로 함께 사용한다.

**예시: AMMR Battery 임계값 문제**
```
- Pipeline A: AMMR 1호기 Battery 확인 → 21% → 배정 가능 → Job 배정 시작
- 동시에: AMMR 1호기 Battery 19%로 갱신 (임계값 20% 이하)
- Pipeline A: 이미 배정 가능으로 판단 → Job 배정 완료
→ Battery 임계값 이하인데 Job이 배정되는 문제 발생
```

이걸 막으려면 "AMMR 1호기 데이터를 읽는 동안 다른 누구도 수정 못하게 잠가야" 한다. 이게 DB 레벨 락이 필요한 이유.

### 멱등성 (Idempotency) - 중복 Job 생성 방지

> **멱등성이란?** 같은 요청을 여러 번 해도 결과가 한 번 한 것과 동일한 성질을 말한다.
>
> **예시:**
> - 전등 스위치를 켜는 행위 → 이미 켜져 있는데 또 켜도 결과는 동일 (멱등)
> - 은행 이체 → 같은 요청을 2번 보내면 2번 이체됨 (멱등 아님)
>
> 우리 시스템에서는 "동일한 Job 생성 요청이 2번 들어와도 Job은 1개만 생성된다"가 멱등성을 보장하는 것이다.

**문제 상황:**

Event Queue로 Event 순서는 보장되지만, 네트워크 불안정 등으로 동일한 QR Scan Event가 2번 전송되는 경우가 있다. 이때 Queue에 동일한 Event가 2개 쌓이면, 순서대로 처리하더라도 같은 Unit에 Job이 중복 생성될 수 있다.

```
1. U-045 세척 WIP 출고 Event → Queue 입력 (1번째)
2. U-045 세척 WIP 출고 Event → Queue 입력 (2번째, 네트워크 재전송)
3. Core가 1번 꺼냄 → Job 생성 → U-045에 Pickup Job 배정
4. Core가 2번 꺼냄 → 동일한 Job 또 생성 시도 → ?
```

**처리 방식:**

Job 생성 시점에 Job 리스트에 동일한 Job이 이미 존재하는지 확인한다. 타임스탬프, GUID 등 고유 식별값을 제외한 나머지 속성이 모두 일치하는 잡이 이미 존재하는 경우 새 Job을 생성하지 않는다.

```
4. Core가 2번 꺼냄 → Job 생성 시도
   → Job 리스트 확인: U-045의 동일 Job 이미 존재
   → 새 Job 생성하지 않음
```

→ Event가 중복 전송되어도 Job은 1개만 생성됨.

## 2. 동기 vs 비동기

### 개념

- **동기(Sync):** 요청을 보내고 응답이 올 때까지 기다림. 요청-응답이 한 쌍.
- **비동기(Async):** 요청을 보내고 기다리지 않고 다른 일을 함. 나중에 결과를 받음.

### 우리 시스템에서

**중계 프로그램 ↔ Core 통신: 동기**
```
중계 프로그램 → Core: GET (Unit ID 조회)
Core: 판단 후 결과 리턴 (중계 프로그램은 응답 기다림)
중계 프로그램: 응답 받고 작업 수행
중계 프로그램 → Core: PATCH (완료 보고)
```

**Core 서버 자체 (ASP.NET Core 웹 서버): 비동기 (Event 발생 시 반응하는 구조 = Event 드리븐)**

Core는 동시에 여러 Event를 받아야 한다:
- WIP QR Scan Event
- AMMR MQTT 상태 보고
- 중계 프로그램 GET/PATCH
- 웹 Dashboard REST 요청
- 장비 AMMR Event

이것들이 동시다발로 들어오므로 Core 서버 자체 (ASP.NET Core 웹 서버)는 비동기/Event 드리븐으로 구현해야 한다. 하나의 요청을 처리하는 동안 다른 요청도 받을 수 있어야 하기 때문.

> **요약:**
> - 중계 ↔ Core 통신 방식: 동기 (요청-응답)
> - Core 서버 내부 구현: 비동기 (여러 요청 동시 처리) = Event 드리븐
> - 두 가지는 다른 레이어의 이야기

## 3. 폴링 vs Webhook

### 배경
SM 시스템의 장비 상태(Idle 여부)를 Core가 알아야 한다. SM은 외부 시스템이라 통신 방식 선택에 제약이 있다.

### 폴링 방식

Core가 SM에 주기적으로 물어보는 방식.

```
Core → SM: "장비 상태 어때?" (1분마다)
SM → Core: "CNC 1호기 Idle"
Core → SM: "장비 상태 어때?" (1분 후)
SM → Core: "CNC 1호기 작업중"
```

→ Core가 계속 먼저 물어봐야 함.

**비유:** 택배 왔는지 현관문을 1분마다 나가서 확인

### Webhook 방식

SM이 상태 변경 시 Core에 직접 알려주는 방식.

```
CNC 1호기 상태 변경 발생
SM → Core: "CNC 1호기 Idle로 바뀌었어" (자동으로 즉시)
```

→ Core가 기다리고 있다가 받기만 하면 됨.

**비유:** 택배기사가 도착하면 초인종 눌러주는 것

> **핵심:** Webhook = 특별한 기술이 아니라, Event 발생 시 상대방 API로 REST POST를 자동으로 쏴주는 것.
> SM 개발사에 협의할 때: "장비 상태 변경 시 우리 Core API로 REST POST 쏴주실 수 있나요?"

### SM 장비 상태 수신 방식 4가지 옵션 (협의 이력 보존)

| 방법                  | 설명                                          | 협의 난이도                 | 비고                              |
|-----------------------|-----------------------------------------------|-----------------------------|-----------------------------------|
| ① Webhook                | SM이 상태 변경 시 Core로 REST POST            | 낮음 (코드 몇 줄 추가)       | 가장 권장 (옛 검토)               |
| ② SM DB 직접 접근     | Core가 SM DB에 읽기 권한으로 폴링              | 중간 (DB 구조 노출 부담)     | 차선책                            |
| ③ REST 폴링           | Core가 SM API 주기적 호출                      | 낮음                        | **설계 확정** (SM = REST 폴링)    |
| ④ 장비 I/F 직접 구현  | 별도 독립 모듈로 분리, Core와 REST 통신        | 높음 (해당 장비 I/F 방법 필요) | 최후의 수단                       |

**설계 확정 기준 (SAD v1.0_d118 §5.2 / SRS v2.0_d68 §8.3)**: ③ REST 폴링 확정. SM은 REST API만 제공, Core가 주기적으로 폴링하여 상태 변화를 직접 감지하는 방식으로 확정. 옛 검토 내용 (① Webhook 권장)은 확정 내용과 다름 — 이 자료는 의사결정 이력 보존 목적으로 4옵션 표 유지.

### Webhook 구현 샘플 (SM 개발사 참고용)

SM 개발사가 추가해야 하는 코드 (Python 예시):

```python

            "status": new_status,           # 예: "IDLE"
            "timestamp": "2024-01-01T10:00:00Z"
        }
```

## 4. Redis란 무엇인가

### 개념

Redis는 **메모리에 데이터를 저장하는 DB**다. 일반 DB(PostgreSQL)는 디스크에 저장하는데, Redis는 메모리에 저장하므로 속도가 매우 빠르다.

**특징:**
- 속도가 매우 빠름 (메모리 기반)
- 재시작하면 데이터가 사라짐 (영속성 설정 가능)
- Key-Value 구조로 단순

### 우리 시스템에서 Redis를 쓴다면 - 두 가지 용도

#### 용도 1: Cache

자주 조회되는 데이터를 Redis에 저장해두고, 매번 DB를 조회하는 대신 Redis에서 빠르게 가져오는 방식.

```
웹 Dashboard → Core → DB 조회 (느림)         ← Cache 없을 때
웹 Dashboard → Core → Redis 조회 (빠름)       ← Cache 있을 때
                    (없으면 DB → Redis 저장 후 리턴)
```

**우리 시스템에서 Cache 대상 후보:**
- AMMR 현재 상태 (위치/Battery)
- WIP Slot 점유 현황

#### 용도 2: Pub/Sub (발행/구독)

Event를 Redis를 통해 주고받는 방식. Core가 Event를 Redis에 발행하면, 구독하고 있는 다른 시스템이 자동으로 수신한다.

```
Core → Redis에 Event 발행 (Publish)
웹 Dashboard → Redis 구독 (Subscribe) → 실시간 수신
```

### 우리 시스템에 Redis 도입 시 가능한 구성

#### 구성 1: 한 서버에 모두 (권장 - 이 시스템 규모)

```
[서버 1대]
- PostgreSQL (메인 DB)
- Redis (Cache/Pub/Sub)
- Core
- 웹 Dashboard
```

**장점:** 인프라 단순, 비용 저렴, 관리 쉬움  
**단점:** 서버 하나 장애 시 전체 다운

#### 구성 2: 서버 분리 (대규모 시스템용)

```
[서버 1] Core
[서버 2] PostgreSQL + Redis
[서버 3] 웹 Dashboard
```

**장점:** 안정성 높음, 스케일 아웃 가능  
**단점:** 비용, 관리 복잡도 증가

> **결론:** 공장 사내 내부망 시스템이고 접속자가 많지 않으므로 구성 1이 충분함.
> Redis 도입 여부는 실제 운영 후 성능 이슈 발생 시 검토 권장.

## 5. 푸시 서버란 무엇인가

### 개념

WebSocket 연결을 전담해서 관리하는 별도 프로그램. Core와 웹 Dashboard 사이에서 실시간 데이터 전달을 담당한다.

### 현재 우리 시스템 구조 (푸시 서버 없음)

```
Core → WebSocket → 웹 Dashboard
```

Core가 직접 WebSocket 연결을 관리하면서 Dashboard에 Push한다. 이 시스템 규모에서는 이 구조로 충분하다.

### 푸시 서버를 쓰는 구조 (대규모 시스템용)

```
Core → Redis Pub/Sub → 푸시 서버 → WebSocket → 웹 Dashboard
```

Core는 Redis에 Event만 발행하고, 푸시 서버가 Redis를 구독해서 WebSocket으로 Dashboard에 전달한다.

**장점:** Dashboard 접속자가 수백 명 이상이어도 Core 부하 없음  
**단점:** Redis + 푸시 서버 추가 관리 필요

> **결론:** 우리 시스템은 공장 내부 사용자만 접속하므로 푸시 서버 불필요. Core가 직접 WebSocket으로 Push하는 현재 구조 유지.

## 6. HTTP 메서드 정리 (REST API)

### 메서드를 구분하는 이유

기술적으로는 GET 하나로 모든 요청을 처리할 수 있다. 하지만 메서드를 구분하는 이유는 두 가지다.

**① 협업과 유지보수**

메서드만 봐도 이 API가 뭘 하는지 바로 알 수 있다. 나중에 다른 개발자가 코드를 볼 때 `GET /units/U-045`이면 "조회구나", `PATCH /units/U-045`이면 "수정이구나"를 즉시 파악할 수 있다. 반면 모든 요청이 GET이라면 URL과 내부 코드를 열어봐야만 의도를 알 수 있다.

**② 브라우저/네트워크 장비의 기본 동작**

GET 요청은 브라우저나 중간 네트워크 장비(Cache 서버 등)가 "안전하게 반복해도 되는 요청"으로 간주해서 캐싱하거나 자동 재시도한다. 만약 GET으로 데이터를 삭제하는 API를 만들면, Cache 서버가 임의로 재시도해서 의도치 않은 삭제가 발생할 수 있다.

### 우리 시스템에서 사용하는 HTTP 메서드

| 메서드 | 용도 | 우리 시스템 예시 |
|--------|------|-----------------|
| GET | 데이터 조회 | 중계 프로그램 → Core: QR 인식 후 작업 여부 조회 |
| POST | 새 데이터 생성 | WIP QR 리더기 → Core: 입출고 Event 전송 |
| PATCH | 일부 데이터 수정 | 중계 프로그램 → Core: 작업 완료 후 Unit 상태 업데이트 |
| PUT | 전체 데이터 교체 | (현재 미사용) |
| DELETE | 데이터 삭제 | (현재 미사용) |

> **PATCH vs PUT 차이:**
> - PUT: 리소스 전체를 교체 (기존 데이터 모두 덮어씀)
> - PATCH: 리소스의 일부만 변경 (변경할 필드만 전달)
> - 작업 완료 보고는 Unit 상태 일부만 바꾸는 것이므로 PATCH가 적합

## 7. DI (의존성 주입)

### 배경

Core 시스템은 여러 파트가 하나의 ASP.NET Core 프로세스 안에서 동작한다. 파트 2(Pipeline)가 WIP Slot 현재 상태를 확인하려면 파트 4가 업데이트하는 데이터에 접근해야 하고, 파트 3(AMMR 제어)이 Job을 받으려면 파트 2가 생성한 데이터에 접근해야 한다. 이때 각 파트가 서로의 서비스를 어떻게 가져다 쓸 것인가가 DI의 핵심이다.

### DI란?

DI(Dependency Injection, 의존성 주입)는 "내가 필요한 도구를 직접 만들지 않고, 누군가가 만들어서 건네주는 방식"이다.

**비유: 공장 작업자와 공구**

DI 없는 방식:
```
작업자 A가 드릴이 필요함
→ 작업자 A가 직접 공구함에 가서 드릴을 찾아옴
→ 작업자 B도 드릴이 필요함
→ 작업자 B도 직접 공구함에 가서 드릴을 찾아옴
→ 같은 드릴이 필요한데 각자 따로 찾아다님
```

DI 방식:
```
공구 관리자가 시작 전에 "누가 뭘 쓰는지" 등록해둠
→ 작업자 A: "드릴 주세요" → 공구 관리자가 건네줌
→ 작업자 B: "드릴 주세요" → 공구 관리자가 같은 드릴을 건네줌
→ 작업자들은 드릴이 어디 있는지 몰라도 됨
```

ASP.NET Core에서 "공구 관리자" 역할을 하는 게 DI Container다.

### 우리 시스템에서 DI가 쓰이는 예시

파트 1이 WIP Slot 상태를 관리하는 공유 서비스를 만들었다고 하면:

```csharp
// 파트 1이 만드는 공유 서비스 (공구에 해당)
public class SlotStateService
{
    // WIP Slot 현재 상태를 메모리에 보관
    public SlotInfo GetSlotInfo(string wipId, int slotNumber) { ... }
    public void UpdateSlotState(string wipId, int slotNumber, SlotState state) { ... }
}
```

```csharp
// 파트 1이 프로그램 시작 시 DI에 등록 (공구 관리자에게 맡기는 것)
builder.Services.AddSingleton<SlotStateService>();
```

```csharp
// 파트 2가 이 서비스를 가져다 쓸 때 (공구 관리자한테 달라고 하는 것)
public class JobPipelineService
{
    private readonly SlotStateService _slotState;

    // 생성자에 적어두면 DI가 자동으로 넣어줌
    public JobPipelineService(SlotStateService slotState)
    {
        _slotState = slotState;
    }

    public void RecalculateJobs()
    {
        var info = _slotState.GetSlotInfo("세척WIP", 1);
        // Slot 상태 확인 후 Job 수행 가능 여부 판단
    }
}
```

```csharp
// 파트 4도 같은 서비스를 가져다 쓸 때
public class ExternalSystemService
{
    private readonly SlotStateService _slotState;

    public ExternalSystemService(SlotStateService slotState)
    {
        _slotState = slotState;
    }

    public void OnSlotEvent(string wipId, int slotNumber, SlotState newState)
    {
        _slotState.UpdateSlotState(wipId, slotNumber, newState);
        // 파트 2에 Event 전달
    }
}
```

핵심: 파트 2와 파트 4가 **같은 SlotStateService 인스턴스**를 공유한다. 파트 4가 상태를 업데이트하면 파트 2가 읽을 때 최신 상태가 반영되어 있다. 이게 가능한 이유는 DI에 `AddSingleton`으로 등록했기 때문이다 (프로그램 실행 동안 인스턴스가 하나만 존재).

### DI 등록 방식 3가지

| 방식 | 인스턴스 수명 | 우리 시스템 사용 예시 |
|------|-------------|---------------------|
| AddSingleton | 프로그램 실행 동안 1개만 존재 | SlotStateService, AmmrStateService 등 공유 상태 서비스. 모든 파트가 같은 인스턴스를 사용 |
| AddScoped | HTTP 요청 하나당 1개 생성 | Controller에서 사용하는 서비스. 요청이 끝나면 사라짐 |
| AddTransient | 요청할 때마다 새로 생성 | 매번 새 인스턴스가 필요한 경우 (우리 시스템에서는 거의 없음) |

> **결론:** WIP Slot 상태, AMMR 상태처럼 여러 파트가 공유해야 하는 InMemory 데이터는 `AddSingleton`으로 등록한다. 파트 1이 이 서비스들을 만들어서 DI에 등록해두면, 나머지 파트는 생성자에 적기만 하면 자동으로 주입된다.

## 8. 스레드 안전 자료구조 (ConcurrentDictionary)

### 배경

Core는 비동기/Event 드리븐으로 동작한다 (섹션 2 참고). 여러 요청이 동시에 들어오면 같은 데이터를 동시에 읽고 쓰는 상황이 발생한다. 일반 자료구조(Dictionary)를 쓰면 동시 접근 시 데이터가 깨질 수 있다.

### 문제 상황

일반 Dictionary를 사용할 때:

```
파트 4: 세척 WIP Slot 1 상태를 "점유"로 업데이트 중
파트 2: 동시에 세척 WIP Slot 1 상태를 읽으려고 접근
→ 파트 4가 쓰는 도중에 파트 2가 읽으면 불완전한 데이터를 볼 수 있음
→ 또는 두 곳에서 동시에 쓰면 데이터가 유실될 수 있음
```

**비유: 공유 화이트보드**

```
일반 화이트보드:
작업자 A가 "세척 WIP Slot 1 = 점유"를 쓰는 도중에
작업자 B가 동시에 같은 줄을 지우고 "세척 WIP Slot 1 = 비어있음"을 씀
→ 글씨가 겹치거나 어느 쪽이 맞는지 알 수 없게 됨
```

### ConcurrentDictionary란?

ConcurrentDictionary는 "동시에 여러 사람이 읽고 써도 안전한 Dictionary"다. 내부적으로 읽기/쓰기 시 자동으로 보호 처리를 해주므로 데이터가 깨지지 않는다.

**비유:**
```
안전한 화이트보드 (ConcurrentDictionary):
작업자 A가 "세척 WIP Slot 1 = 점유"를 쓰려고 함
→ 화이트보드가 자동으로 해당 줄을 잠금
→ 작업자 B가 같은 줄을 쓰려 하면 A가 끝날 때까지 잠깐 기다림
→ A 완료 후 B가 쓰기 시작
→ 데이터가 항상 정확함
```

### 우리 시스템에서 사용 예시

```csharp
// 파트 1이 만드는 WIP Slot 상태 공유 서비스
public class SlotStateService
{
    // ConcurrentDictionary: 여러 파트가 동시에 읽고 써도 안전
    private readonly ConcurrentDictionary<string, SlotInfo> _slots = new();

    // 파트 2가 호출: Slot 상태 읽기 (안전하게 읽기 가능)
    public SlotInfo GetSlotInfo(string key)
    {
        _slots.TryGetValue(key, out var info);
        return info;
    }

    // 파트 4가 호출: Slot 상태 업데이트 (안전하게 쓰기 가능)
    public void UpdateSlotState(string key, SlotInfo newInfo)
    {
        _slots[key] = newInfo;
    }
}
```

파트 4가 중계에서 Event를 받아 `UpdateSlotState`를 호출하고, 동시에 파트 2가 `GetSlotInfo`를 호출해도 데이터가 깨지지 않는다.

### 일반 Dictionary vs ConcurrentDictionary

| 항목 | Dictionary | ConcurrentDictionary |
|------|-----------|---------------------|
| 동시 읽기 | 안전하지 않음 | 안전 |
| 동시 쓰기 | 안전하지 않음 (데이터 유실 가능) | 안전 |
| 읽기 + 쓰기 동시 | 안전하지 않음 | 안전 |
| 속도 | 약간 빠름 | 약간 느림 (보호 처리 비용) |
| 사용 상황 | 한 곳에서만 읽고 쓸 때 | 여러 곳에서 동시에 읽고 쓸 때 |

> **결론:** 우리 시스템에서 WIP Slot 상태, AMMR 상태처럼 여러 파트가 동시에 접근하는 InMemory 데이터는 반드시 ConcurrentDictionary를 사용해야 한다. 파트 1이 이 자료구조를 사용한 공유 서비스를 만들어서 DI에 등록하면, 나머지 파트는 안전하게 읽고 쓸 수 있다.
