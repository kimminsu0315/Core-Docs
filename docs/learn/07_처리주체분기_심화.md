## 1. 구조의 의도

본체 §5.6 ※ 결과 수신 처리 표 + 처리 주체 분기 단락에 담긴 구조의 핵심:

> 이 표의 처리 주체는 사건 성격에 따라 자연 분기한다. 도메인 분기 판정이 필요한 케이스는 AMMR Service가 도메인 판정 후 State Service에 갱신 요청 → State Service atomic 처리. 분기 판정 불필요·강제 종료 케이스는 State Service가 단일 권위 사건으로 직접 판정·atomic 일괄 처리. 분기 기준은 "도메인 분기 판정 유무"이며, 결정 주체가 사건 성격에 따라 자연 분기되는 방식으로 일관성을 갖는다.

이 구조를 두 그룹으로 정리하면:

**그룹 A — 도메인 분기 판정 필요 (AMMR Service 결정 주체)**

- **Pickup 실패** — 출발 Slot 센서 ON/OFF에 따라 처리가 다르다 (Block 마킹 + Transfer 종료 vs Transfer 종료만). 출발 Slot 센서 상태를 InMemory에서 조회·판정하는 도메인 결정이 AMMR Service의 책임 영역.
- **Dropoff 실패** — §5.8 동적 Fallback 경로 분기 (같은 WIP 재탐색 → 대체 경로 탐색 → 만재 라벨). Job 성격(일반 이동 vs CNC 관련 이동)에 따른 대체 경로 결정도 AMMR Service 영역.

**그룹 B — 분기 판정 불필요 강제 종료 (State Service 단일 권위 직접 판정)**

- **Move·Charge 실패** — AMMR HW가 '장애' 상태로 전이된 경우. 도메인 정보가 더 없고 AMMR HW가 더 이상 Job을 수행할 수 없는 상태이므로 강제 종료.
- **AMMR HW 자체 '장애' 전이** — Job 외 시점에 AMMR HW가 자체 전이 Event로 '장애'를 보고. 같은 강제 종료 케이스.
- **MQTT 두절** — Adapter가 AMMR과의 통신이 끊긴 경우. AMMR InMemory 통째 NULL 초기화·Job/Transfer 강제 종료.

분기 기준이 "**도메인 분기 판정 유무**"라는 단일 기준으로 통일되어 있어, 결정 주체가 사건 성격에 따라 자연스럽게 갈린다 — 도메인 정보가 살아있고 분기 판정이 필요하면 AMMR Service, 도메인 정보가 없거나 분기 판정이 불필요하면 State Service.

세 가지 핵심:

1. **Job 주체와 결정 주체는 분리될 수 있다.** AMMR Service는 모든 Job Sequence의 주체이지만, 그 Job 결과를 어떻게 처리할지는 사건 성격에 따라 다른 주체가 결정한다.
2. **State Service는 단일 권위 갱신 게이트웨이.** 어느 주체가 결정하든 InMemory 갱신·DB 영속화 요청·SignalR Push·MQTT 발신은 모두 State Service의 단일 메서드 경로를 거친다.
3. **모든 처리는 atomic 안에서 수행.** State Service 입력 Channel의 단일 Consumer가 FIFO 순으로 한 입력씩 처리하며, 한 atomic 처리 동안 다른 입력이 끼어들 수 없다.

## 3. 4 케이스 C# 샘플 코드

### 3.1 케이스 1 — Pickup 실패 (AMMR Service 도메인 판정)

Pickup 실패는 AMMR Service가 출발 Slot 센서 상태를 도메인 판정하여 처리 분기를 결정한다. State Service는 결정 결과를 atomic 안에서 갱신·기록·Push.

**AMMR Service — 도메인 판정 주체**

```csharp
public class AmmrService
{
    private readonly IStateReader _stateReader;            // InMemory 직접 조회 (Pull)
    private readonly IStateUpdateRequester _stateService;  // 갱신 요청 송신
    private readonly ILogger<AmmrService> _logger;

    public AmmrService(
        IStateReader stateReader,
        IStateUpdateRequester stateService,
        ILogger<AmmrService> logger)
    {
        _stateReader = stateReader;
        _stateService = stateService;
        _logger = logger;
    }

    // State Service가 발행한 Job 수행 결과 Event 구독자
    // (AMMR HW '장애' 전이 케이스는 State Service 단일 권위 직접 판정으로
    //  이 메서드에 도달하지 않는다 — 케이스 3 참조)
    public async Task OnJobResultEventAsync(JobResultEvent ev)
    {
        switch ((ev.JobType, ev.Success))
        {
            case (JobType.Move, true):
                await ProgressNextJobAsync(ev);
                break;

            case (JobType.Pickup, true):
                await ProgressNextJobAsync(ev);  // 다음 Job: Move (목적지로)
                break;

            case (JobType.Pickup, false):
                await HandlePickupFailureAsync(ev);  // 도메인 분기 판정
                break;

            case (JobType.Dropoff, true):
                await CompleteTransferAsync(ev);
                break;

            case (JobType.Dropoff, false):
                await HandleDropoffFailureFallbackAsync(ev);  // 케이스 2
                break;

            case (JobType.Charge, true):
                await CompleteChargeJobAsync(ev);
                break;

            // (JobType.Move, false) / (JobType.Charge, false)는
            // AMMR HW '장애' 전이를 동반하므로 State Service가 직접 판정 — 여기 도달 안 함
        }
    }

    // Pickup 실패 — 출발 Slot 센서 ON/OFF 도메인 분기 판정
    private async Task HandlePickupFailureAsync(JobResultEvent ev)
    {
        if (ev.SourceSlotId is null)
        {
            _logger.LogError("Pickup 실패 Event에 SourceSlotId 없음: {JobId}", ev.JobId);
            return;
        }

        // InMemory 직접 조회 (Pull) — 권위 매체
        // Event Cache가 아닌 권위 저장소에서 현 시점 값을 즉각 조회
        var sourceSensor = _stateReader.GetSlotSensor(ev.SourceSlotId.Value);

        var action = sourceSensor switch
        {
            // 출발 Slot 점유 상태 — Slot Block 마킹 + Transfer 실패 종료
            SensorState.On => PickupFailureAction.BlockSourceAndTerminateTransfer,
            // 출발 Slot 비어 있음 (사람 회수 등으로 Unit 소실) — Transfer 실패 종료만
            SensorState.Off => PickupFailureAction.TerminateTransferOnly,
            _ => throw new InvalidOperationException()
        };

        // State Service에 갱신 요청 — 결정 결과 전달
        // 입력 Channel 경유로 atomic 직렬화 보장
        await _stateService.RequestPickupFailureAsync(new PickupFailureRequest(
            JobId: ev.JobId,
            TransferId: ev.TransferId
                ?? throw new InvalidOperationException("Pickup Job에 TransferId 없음"),
            SourceSlotId: ev.SourceSlotId.Value,
            Action: action));
    }
}
```

**State Service — atomic 갱신 처리**

```csharp
public partial class StateService : BackgroundService
{
    private readonly ChannelWriter<IStateInput> _inputWriter;
    private readonly ChannelReader<IStateInput> _inputReader;
    private readonly IAmmrStateStore _ammrState;
    private readonly IBlockStateStore _blockState;
    private readonly IJobStateStore _jobState;
    private readonly ITransferStateStore _transferState;
    private readonly IPersistenceQueue _persistenceQueue;
    private readonly ISignalRPushGateway _signalRHub;
    private readonly IDomainEventBus _eventBus;

    // AMMR Service의 갱신 요청 수신 — Channel Writer로 송신만, atomic은 Reader 측
    public Task RequestPickupFailureAsync(PickupFailureRequest req) =>
        _inputWriter.WriteAsync(req).AsTask();

    // 단일 Consumer atomic 처리 분기 (전체는 §4 참조)
    private async Task HandleAtomicAsync(IStateInput input)
    {
        switch (input)
        {
            case PickupFailureRequest req:
                await HandlePickupFailureAtomicAsync(req);
                break;
            // ... 다른 입력들
        }
    }

    private async Task HandlePickupFailureAtomicAsync(PickupFailureRequest req)
    {
        // ① InMemory 갱신
        if (req.Action == PickupFailureAction.BlockSourceAndTerminateTransfer)
        {
            _blockState.MarkBlock(req.SourceSlotId, BlockReason.IntegrityMismatch);
            // ※ 출발 Slot 센서 ON + Pickup 실패 = 정합성 어긋남 (§5.7)
        }
        _jobState.Terminate(req.JobId, JobTerminationKind.Failure);
        _transferState.MarkFailed(req.TransferId);

        // ② DB 영속화 요청 (비동기 Worker에 위임 — 이 흐름 격리)
        _persistenceQueue.Enqueue(new JobHistoryRecord(req.JobId, success: false));
        _persistenceQueue.Enqueue(new TransferHistoryRecord(req.TransferId, kind: TransferKind.Failed));
        if (req.Action == PickupFailureAction.BlockSourceAndTerminateTransfer)
        {
            _persistenceQueue.Enqueue(new BlockSummaryRecord(
                slotId: req.SourceSlotId,
                marking: true,
                reason: BlockReason.IntegrityMismatch));
        }

        // ③ SignalR Push (atomic 변경분 일체)
        await _signalRHub.PushPickupFailureAsync(req);
    }
}
```

이 케이스의 핵심은 **AMMR Service가 도메인 판정**(출발 Slot 센서 ON/OFF) 후 **State Service가 결정 결과만 atomic 안에서 일괄 적용**하는 구조이다. State Service 자체가 출발 Slot 센서를 보고 분기하지 않는다.

### 3.2 케이스 2 — Dropoff 실패 (AMMR Service Fallback 진입)

Dropoff 실패는 §5.8 동적 Fallback 경로 분기를 AMMR Service가 결정한다. 분기 방식: 같은 WIP 재탐색 → 대체 경로(일반 이동/CNC 관련 이동에 따라 다름) → 만재 라벨 부착.

```csharp
public partial class AmmrService
{
    private async Task HandleDropoffFailureFallbackAsync(JobResultEvent ev)
    {
        if (ev.TransferId is null || ev.AmmrSlotIndex is null)
        {
            _logger.LogError("Dropoff 실패 Event에 필수 필드 없음: {JobId}", ev.JobId);
            return;
        }

        // 1순위 — 같은 WIP 내 다른 가용 Slot 재탐색
        var sameWipSlot = FindAvailableSlotInSameWip(ev.TargetSlotId);
        if (sameWipSlot is not null)
        {
            // 새 목적지로 Dropoff Job 재발행 요청
            await _stateService.RequestDropoffJobAsync(new DropoffJobRequest(
                AmmrId: ev.AmmrId,
                AmmrSlotIndex: ev.AmmrSlotIndex.Value,
                TargetSlotId: sameWipSlot.Value,
                TransferId: ev.TransferId.Value));
            return;
        }

        // 같은 WIP 가용 Slot 소진 — 대체 경로 탐색
        // §5.8 동적 Fallback 대체 경로:
        //   - 일반 이동: 출발 Slot 복귀
        //   - CNC 관련 이동: 1순위 되담기 WIP 출고 Slot, 2순위 출발 Slot 복귀
        var alternateSlot = ev.JobOrigin switch
        {
            JobOrigin.Cnc =>
                FindReturnWipOutputSlot()    // 1순위 되담기 WIP 출고 Slot
                ?? ev.SourceSlotId,          // 2순위 출발 Slot 복귀
            JobOrigin.General =>
                ev.SourceSlotId,             // 출발 Slot 복귀
            _ => null
        };

        if (alternateSlot is not null)
        {
            await _stateService.RequestDropoffJobAsync(new DropoffJobRequest(
                AmmrId: ev.AmmrId,
                AmmrSlotIndex: ev.AmmrSlotIndex.Value,
                TargetSlotId: alternateSlot.Value,
                TransferId: ev.TransferId.Value));
            return;
        }

        // 대체 경로 실패 — 만재 라벨 부착 요청
        await _stateService.RequestSlotFullLabelAsync(new SlotFullLabelRequest(
            AmmrId: ev.AmmrId,
            AmmrSlotIndex: ev.AmmrSlotIndex.Value,
            TransferId: ev.TransferId.Value));
    }

    // 같은 WIP 내 가용 Slot 재탐색 — Block 마킹 Slot 제외
    private SlotId? FindAvailableSlotInSameWip(SlotId? targetSlotId)
    {
        if (targetSlotId is null) return null;

        var wipId = _stateReader.GetWipIdOfSlot(targetSlotId.Value);
        var candidates = _stateReader.GetSlotsInWip(wipId);

        return candidates
            .Where(s => _stateReader.GetSlotSensor(s) == SensorState.Off)  // 비어 있음
            .Where(s => !_stateReader.IsBlocked(s))                         // Block 아님
            .Cast<SlotId?>()
            .FirstOrDefault();
    }

    // 되담기 WIP 출고 Slot 탐색 (CNC 관련 이동의 1순위 대체 경로)
    private SlotId? FindReturnWipOutputSlot()
    {
        var returnWipOutputSlots = _stateReader.GetReturnWipOutputSlots();
        return returnWipOutputSlots
            .Where(s => _stateReader.GetSlotSensor(s) == SensorState.Off)
            .Where(s => !_stateReader.IsBlocked(s))
            .Cast<SlotId?>()
            .FirstOrDefault();
    }
}
```

**State Service — 결정 결과 atomic 적용**

```csharp
public partial class StateService
{
    public Task RequestDropoffJobAsync(DropoffJobRequest req) =>
        _inputWriter.WriteAsync(req).AsTask();

    public Task RequestSlotFullLabelAsync(SlotFullLabelRequest req) =>
        _inputWriter.WriteAsync(req).AsTask();

    private async Task HandleDropoffJobRequestAsync(DropoffJobRequest req)
    {
        // 새 Dropoff Job 도메인 객체 생성
        var newJob = new DropoffJob(
            JobId: JobId.New(),
            AmmrId: req.AmmrId,
            AmmrSlotIndex: req.AmmrSlotIndex,
            TargetSlotId: req.TargetSlotId,
            TransferId: req.TransferId);

        // InMemory 갱신
        _jobState.RegisterInProgress(newJob);
        _ammrState.SetCoreLogicState(req.AmmrId, CoreLogicState.JobInProgress);

        // DB 이력 기록
        _persistenceQueue.Enqueue(new JobHistoryRecord(newJob.JobId, started: true));

        // MQTT 발신 (단일 게이트웨이 — §5.6)
        await _mqttGateway.SendJobAsync(newJob);

        // SignalR Push
        await _signalRHub.PushJobIssuedAsync(newJob);
    }

    private async Task HandleSlotFullLabelRequestAsync(SlotFullLabelRequest req)
    {
        // 만재 라벨 부착 — AMMR Slot 단위 잠정 상태
        _ammrState.AttachFullLabel(req.AmmrId, req.AmmrSlotIndex);

        // DB 이력
        _persistenceQueue.Enqueue(new SlotFullLabelRecord(req.AmmrId, req.AmmrSlotIndex, req.TransferId));

        // SignalR Push (Dashboard 만재 표시)
        await _signalRHub.PushFullLabelAsync(req);

        // 이후 §5.8 만재 라벨 해소 메커니즘으로 진행
    }
}
```

Fallback 경로 분기는 AMMR Service의 도메인 결정. State Service는 결정 결과(Dropoff Job 재발행·만재 라벨 부착)를 atomic 안에서 갱신·기록·Push.

### 3.3 케이스 3 — Move·Charge 실패 (State Service 직접 판정)

Move·Charge 실패는 AMMR HW '장애' 전이를 동반하는 강제 종료 케이스. AMMR HW가 더 이상 Job을 수행할 수 없으므로 도메인 분기 판정이 없음 — State Service가 단일 권위 사건으로 직접 판정·atomic 일괄 처리.

같은 방식으로 Job 외 AMMR HW 자체 '장애' 전이도 처리 — 보고 경로(통합 보고 Payload vs AMMR HW 자체 전이 Event)는 다르지만 사건 성격은 동일.

```csharp
public partial class StateService
{
    // 입력 Channel 단일 Consumer에서 모든 입력 분기 (전체는 §4 참조)
    private async Task HandleAtomicAsync(IStateInput input)
    {
        switch (input)
        {
            case JobResultReport report:
                await HandleJobResultReportAsync(report);
                break;

            case AmmrSelfTransitionReport stateChange:
                await HandleAmmrSelfTransitionAsync(stateChange);
                break;

            // ... 다른 입력들
        }
    }

    // Job 종료 통합 보고 처리 — Move/Pickup/Dropoff/Charge 공통
    private async Task HandleJobResultReportAsync(JobResultReport report)
    {
        // ★ AMMR HW '장애' 전이 검출 — 단일 권위 직접 판정
        // Move 실패·Charge 실패는 Payload의 AmmrHwStatus가 Failure로 실려서 옴
        if (report.AmmrHwStatus == AmmrHwStatus.Failure)
        {
            await HandleAmmrHwFailureAtomicAsync(
                ammrId: report.AmmrId,
                jobId: report.JobId,
                relatedTransferId: _transferState.GetTransferIdByJobId(report.JobId));

            // ※ AMMR Service에 Event 위임 안 함 — 분기 판정 불필요
            //   AMMR Service의 OnJobResultEventAsync는 호출되지 않는다
            return;
        }

        // 정상 흐름 (AMMR HW 정상) — 일반 처리 후 AMMR Service에 Event 위임
        await HandleNormalJobResultAsync(report);
    }

    private async Task HandleNormalJobResultAsync(JobResultReport report)
    {
        // ① InMemory 갱신 — AMMR HW 상태·Slot 센서·Slot 식별값
        _ammrState.UpdateHwStatus(report.AmmrId, report.AmmrHwStatus);

        if (report.SlotSensors is { Length: > 0 })
        {
            _ammrState.UpdateSlotSensors(report.AmmrId, report.SlotSensors);
        }

        // Pickup·Dropoff 시 Slot Unit 식별값 추정 등급 갱신 (§5.5)
        if (report.JobType == JobType.Pickup && report.Success)
        {
            UpdateSlotIdentifierForPickup(report);
        }
        else if (report.JobType == JobType.Dropoff && report.Success)
        {
            UpdateSlotIdentifierForDropoff(report);
        }

        // ② 정합성 검사 (§5.7)
        // ③ DB 영속화 요청
        _persistenceQueue.Enqueue(new JobHistoryRecord(report));

        // ④ Job 수행 결과 Event 발행 — AMMR Service가 구독
        await _eventBus.PublishAsync(new JobResultEvent(
            AmmrId: report.AmmrId,
            JobId: report.JobId,
            JobType: report.JobType,
            Success: report.Success,
            SourceSlotId: _jobState.GetSourceSlotId(report.JobId),
            TargetSlotId: _jobState.GetTargetSlotId(report.JobId),
            AmmrSlotIndex: _jobState.GetAmmrSlotIndex(report.JobId),
            TransferId: _transferState.GetTransferIdByJobId(report.JobId),
            JobOrigin: _jobState.GetJobOrigin(report.JobId),
            AmmrHwStatus: report.AmmrHwStatus));

        // ⑤ SignalR Push
        await _signalRHub.PushJobResultAsync(report);
    }

    // AMMR HW 자체 전이 Event 처리 — Job 외 단독 전이
    private async Task HandleAmmrSelfTransitionAsync(AmmrSelfTransitionReport stateChange)
    {
        // ★ '장애' 전이 검출 — 단일 권위 직접 판정 (Move·Charge 실패와 같은 방식)
        if (stateChange.NewHwStatus == AmmrHwStatus.Failure)
        {
            // 진행 중 Job·Transfer 강제 종료 (있을 수도, 없을 수도)
            var inProgressJobId = _jobState.GetInProgressJobId(stateChange.AmmrId);
            var inProgressTransferId = inProgressJobId.HasValue
                ? _transferState.GetTransferIdByJobId(inProgressJobId.Value)
                : null;

            await HandleAmmrHwFailureAtomicAsync(
                ammrId: stateChange.AmmrId,
                jobId: inProgressJobId,
                relatedTransferId: inProgressTransferId);
            return;
        }

        // 정상 전이 (대기·충전 중 등) — AMMR HW 상태 갱신만
        _ammrState.UpdateHwStatus(stateChange.AmmrId, stateChange.NewHwStatus);
        _persistenceQueue.Enqueue(new AmmrStateTransitionRecord(stateChange));

        await _eventBus.PublishAsync(new AmmrStateChangeEvent(
            AmmrId: stateChange.AmmrId,
            NewHwStatus: stateChange.NewHwStatus,
            NewCoreLogicState: null));

        await _signalRHub.PushAmmrStateChangeAsync(stateChange);
    }

    // ★ AMMR HW 자체 장애 atomic 처리 — Move·Charge 실패와 Job 외 자체 전이 공통
    //    State Service 단일 권위로 다른 서비스 호출 없이 일괄 처리
    private async Task HandleAmmrHwFailureAtomicAsync(
        AmmrId ammrId,
        JobId? jobId,
        TransferId? relatedTransferId)
    {
        // ① InMemory Null 초기화 — HW '장애' 시 AMMR 실시간 상태를 신뢰 못 함 (본체 §5.5)
        //    위치·Battery·6 Slot 점유 센서·6 Slot 식별값 모두 Null
        _ammrState.NullifyAll(ammrId);

        // ② Core 논리 AMMR 상태 = Null 전이 (광역 Block 시 §5.5)
        _ammrState.SetCoreLogicState(ammrId, CoreLogicState.None);

        // ③ AMMR HW 상태만 '장애' 부착 (케이스 4 두절은 HW 상태도 Null — 그 차이)
        _ammrState.UpdateHwStatus(ammrId, AmmrHwStatus.Failure);

        // ④ 해당 AMMR의 6 Slot 전체 Block 마킹
        var ammrSlotIds = _ammrState.GetSlotIds(ammrId);
        foreach (var slotId in ammrSlotIds)
        {
            _blockState.MarkBlock(slotId, BlockReason.AmmrHwError);
            // ※ 본체 §5.7 Block 사유 enum AmmrHwError 사용 (AMMR HW 장애)
            //    광역 마킹 일관 — §5.11 표면화 단락
        }

        // ⑤ 진행 중 Job Block 마킹 + 종료 (결과 미확정 강제 종료)
        if (jobId.HasValue)
        {
            _jobState.MarkBlockAndTerminate(jobId.Value);
            _persistenceQueue.Enqueue(new JobHistoryRecord(
                jobId.Value,
                kind: JobTerminationKind.IndeterminateForced));
        }

        // ⑥ 진행 중 Transfer 실패 종료
        if (relatedTransferId.HasValue)
        {
            _transferState.MarkFailed(relatedTransferId.Value);
            _persistenceQueue.Enqueue(new TransferHistoryRecord(
                relatedTransferId.Value,
                kind: TransferKind.Failed));
        }

        // ⑦ Block 요약 Event + AMMR 상태 전이 이력 DB 기록 (HW '장애' + Core 논리 Null)
        foreach (var slotId in ammrSlotIds)
        {
            _persistenceQueue.Enqueue(new BlockSummaryRecord(
                slotId: slotId,
                marking: true,
                reason: BlockReason.AmmrHwError));
        }
        _persistenceQueue.Enqueue(new AmmrStateTransitionRecord(
            ammrId,
            newHwStatus: AmmrHwStatus.Failure,
            newCoreLogicState: CoreLogicState.None));

        // ⑧ AMMR 상태 변화 Event 발행 (HW '장애' 전이 + Core 논리 Null 전이)
        await _eventBus.PublishAsync(new AmmrStateChangeEvent(
            AmmrId: ammrId,
            NewHwStatus: AmmrHwStatus.Failure,
            NewCoreLogicState: null));

        // ⑨ SignalR Push (atomic 변경분 일체)
        await _signalRHub.PushAmmrHwFailureAsync(ammrId);
    }
}
```

이 케이스의 핵심은 **State Service가 단일 권위로 직접 판정·일괄 처리**하는 구조이다. AMMR Service에 Event를 발행하지만 그것은 사실 전파일 뿐 도메인 처리 위임이 아니다 — 실제 처리는 atomic 안에서 모두 끝나 있다.

### 3.4 케이스 4 — MQTT 두절 (State Service 직접 판정)

MQTT Adapter 두절은 Adapter 측에서 last seen 임계 초과를 감지하여 단절 신호를 State Service에 전달. State Service는 단일 권위로 직접 판정 — InMemory 통째 NULL 초기화·Job/Transfer 강제 종료.

케이스 3과의 차이점: 케이스 3·4 모두 **InMemory Null 초기화**(위치·Battery·Slot·식별값·Core 논리)는 동일하다. 다른 점은 AMMR HW 상태 — 케이스 4는 두절이라 HW 상태 자체를 모르니 Null, 케이스 3은 HW '장애'를 payload로 받았으니 '장애' 부착이다.

```csharp
public class MqttAdapter
{
    private readonly ChannelWriter<IStateInput> _inputWriter;
    private readonly Dictionary<AmmrId, DateTime> _lastSeen = new();
    private readonly TimeSpan _disconnectThreshold = TimeSpan.FromSeconds(30);

    // 주기적 last seen 검사 (BackgroundService 내부에서 호출)
    private async Task CheckDisconnectionsAsync(CancellationToken ct)
    {
        var now = DateTime.UtcNow;
        foreach (var (ammrId, lastSeen) in _lastSeen.ToArray())
        {
            if (now - lastSeen > _disconnectThreshold)
            {
                // 단절 신호를 State Service 입력 Channel에 전달
                await _inputWriter.WriteAsync(new AmmrDisconnectSignal(
                    AmmrId: ammrId,
                    DetectedAt: now), ct);

                _lastSeen.Remove(ammrId);  // 재연결 시 재추가
            }
        }
    }

    // MQTT 메시지 수신 시마다 last seen 갱신
    public void OnAmmrMessageReceived(AmmrId ammrId)
    {
        _lastSeen[ammrId] = DateTime.UtcNow;
    }
}
```

## 4. atomic 구조의 구현 — 입력 Channel 단일 Consumer

State Service는 Adapter 6종(GM/SM/일반 공정 WIP/되담기 WIP/MQTT/Command API)이 다수 Writer로 쓰고 State Service가 단일 Reader로 소비하는 입력 Channel을 통해 모든 입력을 직렬화한다. 이 직렬화가 atomic 구조의 인프라 측 핵심.

```csharp
// 인프라 계층 — DI 등록
public static class StateServiceInfrastructure
{
    public static IServiceCollection AddStateInputChannel(this IServiceCollection services)
    {
        var channel = Channel.CreateUnbounded<IStateInput>(new UnboundedChannelOptions
        {
            SingleReader = true,    // State Service 단일 Consumer (성능·정합성)
            SingleWriter = false,   // Adapter 6종 + AMMR Service·Transfer Service 갱신 요청
            AllowSynchronousContinuations = false
        });

        services.AddSingleton(channel.Writer);
        services.AddSingleton(channel.Reader);

        return services;
    }
}

// State Service — BackgroundService로 단일 Consumer 실행
public partial class StateService : BackgroundService
{
    private readonly ChannelReader<IStateInput> _inputReader;

    public StateService(
        ChannelReader<IStateInput> inputReader,

        _logger = logger;

    // 한 바퀴 = 한 atomic 단위, 잡은 입력은 아래 HandleAtomicAsync로 분기
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await foreach (var input in _inputReader.ReadAllAsync(stoppingToken))
            await HandleAtomicAsync(input);   // try/catch 상세 = Channels 해설 §2.4
```

**Adapter 측 — Writer로 입력 Channel에 전달**

```csharp
public class MqttAdapter : BackgroundService
{
    private readonly ChannelWriter<IStateInput> _inputWriter;
    private readonly IMqttClient _mqttClient;

    public MqttAdapter(
        ChannelWriter<IStateInput> inputWriter,
        IMqttClient mqttClient)
    {
        _inputWriter = inputWriter;
        _mqttClient = mqttClient;
    }

    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        await _mqttClient.ConnectAsync(stoppingToken);

        // MQTT 메시지 구독 — Payload → 도메인 객체 변환 후 입력 Channel로
        _mqttClient.OnMessageReceived += async (msg) =>
        {
            // last seen 갱신 (단절 감지용)
            UpdateLastSeen(msg.AmmrId);

            // Payload → 도메인 객체 변환 (Adapter 책임 — §6장 공통 Adapter 책임 원칙)
            IStateInput input = ParseToDomainInput(msg);

            // State Service 입력 Channel에 전달 — Push 정규화
            // 비교·변화 감지·도메인 판단·Event 발행은 State Service의 책임이라
            // Adapter는 Push만 한다
            await _inputWriter.WriteAsync(input, stoppingToken);
        };

        // 단절 감지 루프
        while (!stoppingToken.IsCancellationRequested)
        {
            await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
            await CheckDisconnectionsAsync(stoppingToken);
        }
    }
}
```

**atomic 구조의 핵심**:

1. **단일 Consumer가 FIFO 순으로 한 입력씩 처리** — 처리 단계(InMemory 갱신·정합성 검사·Event 발행·DB 영속화 요청·SignalR Push) 동안 다른 입력이 끼어들 수 없다.
2. **ConcurrentDictionary의 다중 필드 복합 연산이 atomic 보장** — read-modify-write·정합성 검사 → 갱신 연쇄 같은 복합 연산이 입력 직렬화로 인터리브 위험 없이 안전.
3. **읽기는 별도 경로** — Transfer Service·AMMR Service는 읽기 전용 참조(`IReadOnlyDictionary`)로 InMemory 직접 조회. 쓰기 권위 일원화와 읽기 성능 양립.

```csharp
// InMemory 권위 자료구조 — 쓰기 권위는 State Service 내부 한정,
// 읽기는 IReadOnlyDictionary로 다른 서비스에 노출
public class AmmrStateStore : IAmmrStateStore, IAmmrStateReader
{
    // 쓰기 가능 — State Service 내부에서만 접근
    private readonly ConcurrentDictionary<AmmrId, AmmrSnapshot> _store = new();

    // 읽기 전용 — 다른 서비스에 DI로 노출
    public IReadOnlyDictionary<AmmrId, AmmrSnapshot> Store => _store;

    public AmmrSnapshot Get(AmmrId id) => _store[id];

    // 쓰기 메서드 — internal 접근자로 외부 직접 호출 차단
    internal void UpdateHwStatus(AmmrId id, AmmrHwStatus newStatus) =>
        _store.AddOrUpdate(
            id,
            _ => new AmmrSnapshot { HwStatus = newStatus },
            (_, prev) => prev with { HwStatus = newStatus });

    // ... 다른 쓰기 메서드들
}

// AmmrSnapshot은 immutable record — 외부 서비스가 내부 필드 직접 변경 불가
public record AmmrSnapshot
```

## 5. 구조의 일관성 검증

이 구조가 자연스러운지를 코드 상에서 다시 확인.

### 5.1 Job 주체와 결정 주체의 분리

| 사건 | Job 주체 | 결과 처리 결정 주체 | 분기 기준 |
|---|---|---|---|
| Pickup 실패 | AMMR Service | AMMR Service (출발 Slot 센서 ON/OFF 도메인 판정) | 도메인 분기 판정 필요 |
| Dropoff 실패 | AMMR Service | AMMR Service (Fallback 경로 결정) | 도메인 분기 판정 필요 |
| Move 실패 (AMMR HW '장애' 전이 동반) | AMMR Service | State Service (단일 권위 직접 판정) | 도메인 분기 판정 불필요 |
| Charge 실패 (AMMR HW '장애' 전이 동반) | AMMR Service | State Service (단일 권위 직접 판정) | 도메인 분기 판정 불필요 |
| AMMR HW 자체 '장애' 전이 (Job 외) | (Job 무관) | State Service (단일 권위 직접 판정) | 도메인 분기 판정 불필요 |
| MQTT Adapter 두절 | (Job 무관) | State Service (단일 권위 직접 판정) | 도메인 분기 판정 불필요 |

Job 주체와 결정 주체가 분리되는 것이 어색해 보일 수 있으나, 분기 기준 "**도메인 분기 판정 유무**"로 보면 자연스럽다:

- AMMR Service는 Job Sequence의 주체이며 **동시에** 도메인 판정 영역(어느 AMMR Slot에 어느 Transfer 배정·Fallback 경로 결정·Pickup 실패 후속 처리 등)
- State Service는 단일 권위 갱신 게이트웨이 (모든 InMemory 쓰기·DB 영속화 요청·SignalR Push·MQTT 발신의 단일 경로)
- 도메인 판정이 필요하면 AMMR Service가 결정 → State Service는 갱신만 처리
- 도메인 판정이 불필요한 강제 종료 케이스(AMMR HW 장애·Adapter 두절)에서는 AMMR Service에 위임할 도메인 정보가 없음 → State Service가 직접 처리하는 것이 자연

코드 상으로 보면, AMMR Service의 `OnJobResultEventAsync`는 Pickup/Dropoff 실패만 분기 처리하고, Move/Charge 실패는 받지 않는다 (State Service가 Event 발행 전에 이미 atomic 안에서 처리 종결). 즉 Event 자체가 "정상 흐름의 후속 처리만 위임" 방식으로 흐른다.

### 5.2 atomic 구조의 단순성

State Service 단일 권위 직접 판정 케이스(케이스 3·4)에서는 한 atomic 안에 다른 서비스 호출이 끼어들지 않는다.

```csharp
// 케이스 3 — AMMR HW 자체 장애 atomic
private async Task HandleAmmrHwFailureAtomicAsync(AmmrId ammrId, JobId? jobId, TransferId? relatedTransferId)
{
    // 모든 처리가 State Service 자체 자료구조에서만 일어남
    // AMMR Service · Transfer Service의 RPC/메서드 호출 없음
    _ammrState.UpdateHwStatus(...);
    foreach (var slotId in ammrSlotIds) _blockState.MarkBlock(...);
    if (jobId.HasValue) _jobState.MarkBlockAndTerminate(...);
    if (relatedTransferId.HasValue) _transferState.MarkFailed(...);
    foreach (var slotId in ammrSlotIds) _persistenceQueue.Enqueue(...);
    _persistenceQueue.Enqueue(...);
    
    // Event 발행은 사실 전파만 — 구독자 처리는 비동기, atomic 밖
    await _eventBus.PublishAsync(...);
    
    // SignalR Push도 atomic 안에서 한 번만
    await _signalRHub.PushAmmrHwFailureAsync(ammrId);
}
```

이 단순성이 두 가지 이점을 준다:

1. **추적·디버깅 비용 절감** — atomic 처리 안에서 일어나는 모든 변화가 한 메서드 안에 보인다. Log 한 줄로 사건 전후 상태 확인 가능.
2. **인터리브 위험 차단** — atomic 안에서 다른 서비스 메서드를 await하면 그 await 동안 다른 입력이 끼어들 수도 있다는 우려가 발생할 수 있으나, 이 케이스는 await가 없거나 마지막 SignalR Push 한 번뿐이라 우려 없음. State Service 입력 Channel의 단일 Consumer 방식이라 한 입력이 처리 완료될 때까지 다음 입력을 꺼내지 않는다.

대조적으로 그룹 A 케이스는 `AMMR Service → State Service에 갱신 요청 → State Service atomic`의 두 단계로 진행되지만, 두 단계가 별도 atomic 단위라 인터리브 위험 없음. AMMR Service의 도메인 판정 동안 State Service는 다른 입력을 처리할 수 있으며, 갱신 요청이 입력 Channel에 들어와야 비로소 그 처리가 atomic 안에서 일어난다.

### 5.3 State Service 단일 게이트웨이 결과의 호환

State Service는 모든 InMemory 권위 갱신·DB 영속화 요청·SignalR Push·MQTT 발신의 단일 게이트웨이(§4.5·§4.6). 처리 주체 분기 구조는 이 게이트웨이 구조를 깨지 않는다:

- **그룹 A — 도메인 판정 필요 케이스**: AMMR Service가 결정 → State Service에 갱신 요청 → State Service가 갱신·기록·Push (게이트웨이 단일 경유)
- **그룹 B — 도메인 판정 불필요 케이스**: State Service가 직접 판정 → State Service가 갱신·기록·Push (같은 게이트웨이)

```csharp
// 두 그룹 모두 InMemory 권위 갱신은 State Service의 단일 메서드 경로를 통과
public partial class StateService
{
    // 모든 atomic 처리가 ChannelReader.ReadAllAsync로 진입
    // 그룹 A 갱신 요청도, 그룹 B 직접 판정도 같은 Consumer 루프 안에서 처리
    private async Task HandleAtomicAsync(IStateInput input)
    {
        switch (input)
        {
            case PickupFailureRequest req:           // 그룹 A — AMMR Service의 결정 결과
                await HandlePickupFailureAtomicAsync(req);
                break;
            case JobResultReport { AmmrHwStatus: AmmrHwStatus.Failure } report:  // 그룹 B — State Service 직접 판정
                await HandleAmmrHwFailureAtomicAsync(report.AmmrId, report.JobId, ...);
                break;
            // ...
        }
    }
}
```

두 그룹 모두 입력 Channel을 거쳐 같은 Consumer 루프에서 처리되며, InMemory 갱신·정합성 검사·DB 영속화 요청·SignalR Push의 5단계 atomic 처리를 공통으로 따른다. 처리 주체 분기는 어디까지나 **결정의 영역 분리**일 뿐, **실행 인프라**는 단일.

이로써:

- **권위 매체 구조 보존** — InMemory 권위가 State Service 단일에 일원화 (§4.5 / §6장 공통 ※ 권위 매체 원칙)
- **순서·atomic 구조 보존** — 모든 갱신이 입력 Channel 단일 Consumer의 직렬 atomic 처리 (§1.3 순서·atomic 보존 축)
- **결정 주체의 자연 분기** — 도메인 판정 유무를 기준으로 AMMR Service vs State Service 자연 갈림

세 구조가 한 패턴 안에 일관 정합한다는 점이 이 처리 주체 분기 구조의 통합성 지표다.
