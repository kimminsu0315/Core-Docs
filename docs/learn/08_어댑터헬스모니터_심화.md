## 1. 이 문서 위상

이 문서는 Core 아키 §4.4 구조의 한 적용 사례 — **Adapter가 자기 외부 시스템 가용성을 독립 감시하는 구조**의 구현 방식. 다음 다섯 핵심을 한자리에서 학습:

1. ASP.NET Core 호스팅 환경에서 백그라운드 작업이 어떻게 살아 도는가
2. `BackgroundService` ↔ `while` ↔ `PeriodicTimer` 셋의 책임 분담
3. Adapter 책임 영역 안에 헬스 모니터를 결합하는 방식 (단일 Class vs 결합 Class vs 분리 Class)
4. DI Singleton 구조로 Adapter 인스턴스를 Controller·HealthMonitor·State Service가 공유하는 구조
5. 두절 감지 시점에 State Service까지 도달하는 흐름

**전제 지식**: C# 비동기 (`async/await`), 인터페이스, DI 개념. 모르는 구조는 `Core_기술참고_DI와동시성해설` 1·2장 먼저 정독.

## 2. ASP.NET Core 호스팅 방식

### 2.1 Host가 무엇을 하는가

ASP.NET Core 앱은 **단일 프로세스(.exe 또는 dotnet run)** 안에서 모든 구조가 같이 산다. 그 프로세스 안에서 "누가 무엇을 시작·종료·감시하는가"를 책임지는 구조가 **Host(Host)**.

Host가 책임지는 일:

- **DI Container 구성** — 어떤 Class가 어떤 인터페이스 구현체인지 등록
- **시작 시 모든 등록 서비스 인스턴스화** (Singleton·HostedService 등)
- **HTTP 서버 (Kestrel) 구동** — REST endpoint 수신 시작
- **Hosted Service 시작** — 백그라운드 작업 시작 신호
- **Graceful shutdown** — Ctrl+C·SIGTERM 수신 시 모든 서비스에 종료 신호 전파

비유 — **Host = 회사의 운영팀**:
- 직원(서비스)들 출근 시간 관리
- 사무실(DI Container) 자리 배정
- 외부 손님(HTTP 요청) 응대 가이드
- 퇴근 시간에 모든 직원 정리 신호

### 2.2 Program.cs로부터의 시작 흐름

ASP.NET Core 앱의 진입점은 `Program.cs`. 이 파일이 Host를 빌드하고 실행한다.

```csharp
var builder = WebApplication.CreateBuilder(args);
// 1. WebApplicationBuilder 생성
//    내부에 DI Container·Configuration·Logging이 미리 들어감

builder.Services.AddSingleton<IGeneralProcessWipAdapter, GeneralProcessWipAdapter>();
builder.Services.AddHostedService<GeneralProcessWipHealthMonitor>();
// 2. DI Container에 서비스 등록 단계
//    이 시점에는 인스턴스 생성 없음 — 등록 정보만 넣음

builder.Services.AddControllers();
// 3. Controller 자동 발견·등록

var app = builder.Build();
// 4. 빌드 시점에 DI Container 확정
//    Singleton은 첫 사용 시점에 생성, HostedService는 Host 시작 시점에 생성

app.MapControllers();
// 5. HTTP 라우팅 등록

app.Run();
// 6. Host 실행 — 이 호출이 block된 상태로 Host가 살아 있음
//    Kestrel HTTP 서버 시작
//    HostedService.StartAsync() 일괄 호출
//    이후 Ctrl+C·SIGTERM 수신 시까지 블록
```

### 2.3 Host 수명 구조

`app.Run()` 호출 시점부터 Host가 살아 있고, Ctrl+C 등 종료 신호 시점까지 다음 구조가 모두 한 프로세스 안에 담김:

```
[Core 프로세스]
├── Kestrel HTTP 서버 (Controller 응대)
├── DI Container (Singleton 인스턴스 보유)
├── Hosted Service들 (백그라운드 Worker들)
│   ├── GeneralProcessWipHealthMonitor
│   ├── RestackProcessWipHealthMonitor
│   └── (필요 시 추가)
└── ThreadPool (모든 비동기 작업 실행 공간)
```

**핵심**: HTTP 요청 처리·Hosted Service 백그라운드 작업·DI Container 모두 **같은 프로세스·같은 ThreadPool**. 별도 프로세스·별도 머신 없음.

종료 신호 수신 시:
1. Host가 모든 HostedService에 `StopAsync(token)` 호출
2. 진행 중 HTTP 요청 graceful 종료
3. `app.Run()` block 해제 + 프로세스 종료

## 3. IHostedService와 BackgroundService

### 3.1 IHostedService 인터페이스

ASP.NET Core에서 백그라운드 작업은 `IHostedService`로 정의된다.

```csharp
public interface IHostedService
{
    Task StartAsync(CancellationToken cancellationToken);
    Task StopAsync(CancellationToken cancellationToken);
}
```

이 인터페이스의 핵심:
- `StartAsync`: Host 시작 시점에 호출. 백그라운드 작업 시작.
- `StopAsync`: Host 종료 시점에 호출. 백그라운드 작업 종료.
- **호출 주체**: 개발자 아님, **Host가 호출**.

DI 등록은 `AddHostedService<T>()` 방식으로 등록 — Host가 이 등록 방식을 시작·종료 흐름에 자동 결합.

### 3.2 BackgroundService — IHostedService의 추상 Class

`IHostedService` 인터페이스를 직접 구현하면 `StartAsync`·`StopAsync` 둘 다 구현해야 하고 시작·종료 방식을 매번 짜는 부담. Microsoft가 제공하는 추상 Class `BackgroundService`가 그 부담을 대신한다.

```csharp
public abstract class BackgroundService : IHostedService, IDisposable
{
    protected abstract Task ExecuteAsync(CancellationToken stoppingToken);

    public virtual Task StartAsync(CancellationToken cancellationToken)
    {
        // ExecuteAsync 호출하고 즉시 반환 (백그라운드로 돎)
        _executingTask = ExecuteAsync(_stoppingCts.Token);
        return _executingTask.IsCompleted ? _executingTask : Task.CompletedTask;
    }

    public virtual async Task StopAsync(CancellationToken cancellationToken)
    {
        _stoppingCts.Cancel();
        // ExecuteAsync 종료 대기
        await Task.WhenAny(_executingTask, Task.Delay(Timeout.Infinite, cancellationToken));
    }

    public virtual void Dispose() { _stoppingCts.Cancel(); _stoppingCts.Dispose(); }
}
```

→ 개발자는 `ExecuteAsync` 한 메서드만 구현하면 된다. `stoppingToken`이 종료 신호로 자연히 들어옴.

```csharp
public class MyService : BackgroundService
{
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        // 이 메서드 안에 백그라운드 작업을 넣음
        // stoppingToken이 cancel되면 이 메서드 자연 종료
    }
}
```

**중요**: `ExecuteAsync`는 Host가 **한 번만** 호출. 이 메서드가 종료되면 BackgroundService도 종료된 상태.

### 3.3 왜 BackgroundService + while + PeriodicTimer 셋 다 쓰는가

세 구조의 책임이 다르다.

| 책임 | 담당 |
|---|---|
| Host 통합·자동 시작·종료 신호 | `BackgroundService` |
| "주기적으로 도는 구조" 그 자체 | `while` |
| 정확한 5초 간격 대기 | `PeriodicTimer` |

`BackgroundService`는 **틀**만 제공. 주기 기능 없음. 다음 코드는 한 번 출력하고 끝:

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    Console.WriteLine("Hello"); // 한 번만
    // 메서드 종료 = BackgroundService 종료
}
```

주기 구조 만들려면 `while` 루프:

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    while (!stoppingToken.IsCancellationRequested)
    {
        // 5초마다 점검함
        await Task.Delay(TimeSpan.FromSeconds(5), stoppingToken);
        DoCheck();
    }
}
```

`Task.Delay` 구조도 동작은 함. 다만 정확한 주기 보장 없음 — 점검이 1초 걸리면 다음 점검은 6초 후. 실제 5초 간격이 누적으로 어긋남.

`PeriodicTimer`는 정확한 주기 보장:

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        // 점검이 1초 걸려도 다음 점검은 정확히 5초 간격에 시작
        DoCheck();
    }
}
```

→ **`BackgroundService` = 틀, `while` = 반복 구조, `PeriodicTimer` = 정확한 주기**. 셋 다 책임 다름 + 모두 필요.

### 3.4 Timer Callback 구조와 비교

`while` 없이 짜는 방식도 가능 — `System.Threading.Timer` Callback 구조.

```csharp
public class MyService : IHostedService, IDisposable
{
    private Timer? _timer;

    public Task StartAsync(CancellationToken cancellationToken)
    {
        _timer = new Timer(DoCheck, null,
            TimeSpan.FromSeconds(5),  // 첫 호출 지연
            TimeSpan.FromSeconds(5)); // 주기
        return Task.CompletedTask;
    }

    private void DoCheck(object? state) { /* 점검 */ }

    public Task StopAsync(CancellationToken cancellationToken)
    {
        _timer?.Change(Timeout.Infinite, 0);
        return Task.CompletedTask;
    }

    public void Dispose() => _timer?.Dispose();
}
```

| 구조 | while + PeriodicTimer | Timer Callback |
|---|---|---|
| 표현 | 코드에 명시 (반복 구조 직접 씀) | framework 위임 (Callback 구조) |
| 동시성 | 한 Callback 종료 후 다음 (자연 직렬) | 시간 되면 즉시 (이전 Callback 미종료여도 새 Callback 발화 가능 — 동시 실행 구조) |
| 비동기 | `await` 자연 구조 | Callback 동기 구조 (`async void` 구조로 비동기 가능하나 예외 처리 어려움) |
| 종료 | `stoppingToken` 자연 구조 | `Change(Timeout.Infinite, 0)` 수동 구조 |

**Core 프로젝트 구조 추천**: `BackgroundService` + `while` + `PeriodicTimer`.

근거: 비동기 자연스러움·동시 실행 없음(직렬 방식)·`stoppingToken` 자연 graceful shutdown.

**비유**: 
- while + PeriodicTimer = "알람 맞춰놓고 깨면 점검, 다시 자기" (반복 구조 직접 씀)
- Timer Callback = "비서한테 5초마다 점검 부탁" (외부 위임)

Event 드리븐 시스템에서 "시간"도 Event다. while + `WaitForNextTickAsync`도 Event 대기다 (CPU 점유 없음 — block 상태). while 회피는 표현상 차이지 메커니즘 차이는 아님.

## 4. Adapter 안 헬스 모니터 결합 구조

### 4.1 책임 분담 구조 — SAD §4.4 정합

SAD §4.4 구조:

> 각 Adapter는 자기 외부 시스템·AMMR 가용성을 독립 감시한다. 두절 감지 시 Adapter는 사실을 도메인 객체로 변환해 State Service에 전달하며, State Service가 atomic 안에서 권위 측 InMemory 초기화 + Slot Block 마킹 + 진행 중 Job/Transfer 종료를 일괄 수행한다.

→ Adapter 책임 영역 명시. 가용성 감시는 **Adapter 책임**.

다만 SAD는 **책임 분담 구조**만 담음 — "코드 Class 1대1 Mapping 구조" 없음. 이 책임 구조를 코드로 어떻게 담을지는 SDD/코드 영역.

### 4.2 결합 구조 후보 α·β·γ

세 후보 구조:

| 후보 | 핵심 | Class 분할 |
|---|---|---|
| α | **단일 Class 구조** — Adapter 자체가 BackgroundService 구조, REST 수신·헬스 점검 모두 한 Class | 1 |
| β | **내부 결합 구조** — Adapter Class 안에 HealthMonitor 인스턴스 보유. Adapter가 외부 인터페이스, Monitor는 private | 2 (외부 1) |
| γ | **분리 구조** — Adapter ↔ HealthMonitor 별도 Class. 둘 다 DI 등록, Singleton 공유 구조로 상태 공유 | 2 (외부 2) |

**α 코드 예**:
```csharp
public class GeneralProcessWipAdapter : BackgroundService
{
    public async Task IngestEventAsync(...) { /* REST 수신 */ }
    protected override async Task ExecuteAsync(...) { /* 헬스 점검 */ }
}
```

**β 코드 예**:
```csharp
public class GeneralProcessWipAdapter : IGeneralProcessWipAdapter
{
    private readonly HealthMonitor _monitor;
    public GeneralProcessWipAdapter(...) { _monitor = new HealthMonitor(this); _monitor.Start(); }
    public async Task IngestEventAsync(...) { /* REST 수신 */ }
}

internal class HealthMonitor
{
    public void Start() { /* 백그라운드 시작 */ }
}
```

**γ 코드 예**:
```csharp
public class GeneralProcessWipAdapter : IGeneralProcessWipAdapter
{
    public async Task IngestEventAsync(...) { /* REST 수신 */ }
    public DateTime LastHeartbeatUtc { get; }
}

public class GeneralProcessWipHealthMonitor : BackgroundService
{
    private readonly IGeneralProcessWipAdapter _adapter;
    public GeneralProcessWipHealthMonitor(IGeneralProcessWipAdapter adapter) { _adapter = adapter; }
    protected override async Task ExecuteAsync(...) { /* 헬스 점검 */ }
}
```

### 4.3 Core 프로젝트 추천 구조

**γ 구조 추천**.

근거:

1. **단일 책임 원칙(SRP)** — Adapter = "외부 Payload 도메인 변환 책임", HealthMonitor = "주기 점검 책임". 다른 책임은 다른 Class.
2. **테스트 자연 구조** — Adapter 단위 테스트 시 HealthMonitor 구조 격리 자연 (Mock·격리 가능).
3. **DI 자연 구조** — 둘 다 Singleton 등록 시 DI Container가 같은 Adapter 인스턴스를 HealthMonitor에 자연 주입 (5장 구조).
4. **호스팅 통합 자연 구조** — `HealthMonitor : BackgroundService` 구조로 `AddHostedService<>` 등록 자연. α 구조도 가능하나 Adapter가 BackgroundService 상속 구조로 Adapter 책임 구조와 호스팅 구조 혼재.
5. **AMMR Adapter 구조와 일관 구조** — MQTT는 라이브러리(`MQTTnet` 등)가 disconnect Event Callback 구조로 제공 → AMMR Adapter 안 단일 구조로 담김. WIP는 라이브러리 없어 별도 모니터 구조 자연. **결합 구조 일관성**은 추구하지 않음 — 외부 구조 특성에 따라 자연 분기.

**β 구조 부적합**: HealthMonitor가 internal·생성자 안 시작 구조라 Host 통합 안 됨. ASP.NET Core 구조 어긋남.

**α 구조가 자연스러운 곳**: 작은 프로젝트·외부 시스템 1개라 책임 분리 부담 없을 때. Core 프로젝트 구조로는 SRP 어긋남.

## 5. DI 수명 구조

### 5.1 Singleton 상태 공유 구조

DI 구조의 핵심 = "여러 곳에서 같은 의존성을 받을 때, 누가 어떤 인스턴스를 받는가"를 Container가 결정.

수명 구조 3종 (`Core_기술참고_DI와동시성해설` §1.3 참조):

| 수명 | 핵심 |
|---|---|
| Singleton | 앱 시작 시점에 1번 생성, 모든 곳이 같은 인스턴스 공유 |
| Scoped | HTTP 요청 단위에 1번 생성, 같은 요청 안에서는 공유 |
| Transient | 매 주입마다 새 인스턴스 |

**WIP Adapter 구조**: 마지막 HealthCheck 수신 시각 상태 보유. Controller·HealthMonitor 둘 다 같은 상태 보아야 → **Singleton 자연**.

```csharp
builder.Services.AddSingleton<IGeneralProcessWipAdapter, GeneralProcessWipAdapter>();
```

이 등록 구조로:
- DI Container가 앱 시작 시점에 `GeneralProcessWipAdapter` 인스턴스 1개 생성
- HealthMonitor 생성자에서 `IGeneralProcessWipAdapter` 의존성 요청 시 → 같은 인스턴스 주입
- Controller 생성자에서 `IGeneralProcessWipAdapter` 의존성 요청 시 → 같은 인스턴스 주입
- 셋이 같은 `_lastHeartbeatTicks` 필드를 자연히 공유

### 5.2 이 Component별 수명

| Component | 수명 | 근거 |
|---|---|---|
| `IGeneralProcessWipAdapter` 구현체 | Singleton | 마지막 수신 시각 상태 공유 |
| `GeneralProcessWipHealthMonitor` | Singleton (HostedService 자연) | 백그라운드 단일 작업, Host 수명 동안 유지 |
| `IStateService` 구현체 | Singleton | Core 도메인 권위 일원화 |
| `GeneralProcessWipController` | Scoped (기본) | HTTP 요청 단위. Singleton Adapter 주입 자연 |

**Scoped Controller가 Singleton Adapter 주입 구조**: ASP.NET Core DI는 "더 긴 수명 구조를 짧은 수명 구조에서 주입받는 것"은 자연 구조 — Singleton은 항상 살아 있으니 Scoped가 받아도 자연. 반대(Singleton이 Scoped 받기)는 captive dependency 안티패턴.

### 5.3 DI 등록 구조

`Program.cs` 안 등록 구조:

```csharp
// 1. Adapter — Singleton (상태 공유)
builder.Services.AddSingleton<IGeneralProcessWipAdapter, GeneralProcessWipAdapter>();

// 2. HealthMonitor — Hosted Service
//    AddHostedService<T>()는 내부적으로 Singleton 등록 + Host가 시작·종료 결합
builder.Services.AddHostedService<GeneralProcessWipHealthMonitor>();

// 3. State Service — Singleton (Core 권위 일원화)
builder.Services.AddSingleton<IStateService, StateService>();

// 4. Controller — Scoped (자동 등록)
builder.Services.AddControllers();
```

이 등록 구조만으로 다음 흐름이 자동:

```
앱 시작
  ↓
DI Container 빌드
  ↓
GeneralProcessWipAdapter 인스턴스 1개 생성 (Singleton)
StateService 인스턴스 1개 생성 (Singleton)
  ↓
GeneralProcessWipHealthMonitor 인스턴스 생성
  - 생성자: IGeneralProcessWipAdapter (위 Adapter 인스턴스) + IStateService 주입
  ↓
HostedService.StartAsync 호출 → ExecuteAsync 시작 (백그라운드 흐름)
  ↓
Kestrel HTTP 서버 시작
  ↓
HTTP 요청 도달 시:
  Controller 인스턴스 생성 (Scoped)
  - 생성자: IGeneralProcessWipAdapter (위 Adapter 인스턴스 — Singleton이라 같은 것) 주입
  ↓
Controller가 Adapter.IngestEventAsync 호출
HealthMonitor가 Adapter.LastHeartbeatUtc 읽음
→ 같은 인스턴스 자연 공유
```

## 6. Controller → Adapter → State Service 흐름

### 6.1 REST endpoint 구조

WIP 중계 프로그램은 **외부 시스템**. Core를 REST API로 호출. 이 호출 수신 곳 = Controller.

```csharp
[ApiController]
[Route("api/wip/general")]
public class GeneralProcessWipController : ControllerBase
{
    private readonly IGeneralProcessWipAdapter _adapter;

    public GeneralProcessWipController(IGeneralProcessWipAdapter adapter)
    {
        // Singleton Adapter 주입 — DI Container 방식
        _adapter = adapter;
    }

    [HttpPost("event")]
    public async Task<IActionResult> ReceiveEvent([FromBody] WipEventDto dto)
    {
        // 외부 Payload → Adapter 전달
        var result = await _adapter.IngestEventAsync(dto);
        return Ok(result); // 도메인 Mapping 회신 (Unit ID 등)
    }

    [HttpPost("heartbeat")]
    public async Task<IActionResult> ReceiveHeartbeat()
    {
        // HealthCheck only — 도메인 변환 없음, 마지막 수신 시각만 갱신
        await _adapter.IngestEventAsync(WipEventDto.HeartbeatOnly);
        return Ok();
    }
}
```

**동작 순서**:
1. WIP 중계가 `POST /api/wip/general/heartbeat` 호출
2. Kestrel이 HTTP 요청 수신
3. ASP.NET Core 라우팅 구조로 `GeneralProcessWipController.ReceiveHeartbeat` 결정
4. DI Container가 Controller 인스턴스 생성 (Scoped) + Adapter 주입
5. `_adapter.IngestEventAsync` 호출
6. 200 OK 응답

### 6.2 도메인 변환 구조

Adapter 책임 = "외부 Payload → 도메인 객체 변환".

```csharp
public async Task<IngestResult> IngestEventAsync(WipEventDto dto)
{
    // 1. 마지막 수신 시각 갱신 (모든 호출이 헬스 신호)
    Interlocked.Exchange(ref _lastHeartbeatTicks, DateTime.UtcNow.Ticks);

    // 2. HealthCheck only면 도메인 변환 skip
    if (dto.IsHeartbeatOnly) return IngestResult.HeartbeatAck;

    // 3. 외부 DTO → 도메인 객체 변환 (Mapper 처리)
    var domainEvent = WipEventMapper.ToDomain(dto);

    // 4. State Service 입력 Channel에 전달
    await _stateService.IngestAsync(domainEvent);

    // 5. State Service 처리 결과 회신 (Unit ID Mapping 등)
    return IngestResult.WithUnitId(domainEvent.UnitId);
}
```

**중요**: Adapter는 도메인 변환만. **InMemory 상태 변경 없음** — State Service 책임. 이 구조는 SAD 본체 6장 "공통 Adapter 책임 원칙" 구조.

### 6.3 State Service 입력 Channel 구조

상세는 `Core_구현참고_Adapter_입력처리_구조` 문서 권위 위치. 이 문서는 이 흐름의 한 부분만 참조:

- Adapter가 도메인 객체를 `IStateService.IngestAsync`로 전달
- 내부적으로 `Channel<DomainEvent>` 구조에 Write
- State Service 단일 Consumer가 FIFO 순으로 Pull
- 각 입력은 atomic 단위 처리 (InMemory 갱신 + 정합성 검사 + DB 영속화 요청 + Event 발행 + SignalR Push)

→ 이 문서 영역 밖. 다음 단계에 서비스 핸드오프.

## 7. 동시성 구조

### 7.1 동시 발생 지점

같은 Adapter 인스턴스를 두 곳이 동시 접근:

| 곳 | 작업 |
|---|---|
| Controller (HTTP 요청 처리 스레드) | `_lastHeartbeatTicks = 현재 시각` 쓰기 |
| HealthMonitor (백그라운드 스레드) | `_lastHeartbeatTicks` 읽기 + 비교 |

ASP.NET Core는 ThreadPool 위에서 동작 — 이 두 곳이 **다른 스레드**. 동시 read/write.

`long`은 64비트 — 32비트 머신에서는 read·write가 두 단계로 나뉘어 **torn read** 발생 가능. 64비트 머신에서도 컴파일러 최적화로 캐싱·재정렬 발생 가능.

→ 동시성 보호 필요.

### 7.2 Interlocked 구조

`System.Threading.Interlocked` Class — atomic 연산 lock-free 방식.

```csharp
private long _lastHeartbeatTicks = DateTime.UtcNow.Ticks;

// 쓰기
Interlocked.Exchange(ref _lastHeartbeatTicks, DateTime.UtcNow.Ticks);
//   _lastHeartbeatTicks를 새 값으로 교체. atomic 처리.

// 읽기
var ticks = Interlocked.Read(ref _lastHeartbeatTicks);
//   _lastHeartbeatTicks를 atomic하게 읽음. torn read 방지.
```

**핵심**: CPU 명령어 수준 atomic이라 lock 없음. 빠름.

### 7.3 lock 구조와 비교

`lock` 구조도 가능:

```csharp
private readonly object _lock = new();
private long _lastHeartbeatTicks;

// 쓰기
lock (_lock) { _lastHeartbeatTicks = DateTime.UtcNow.Ticks; }

// 읽기
long ticks;
lock (_lock) { ticks = _lastHeartbeatTicks; }
```

| 구조 | Interlocked | lock |
|---|---|---|
| 성능 | 빠름 (CPU 명령어) | 느림 (커널 구조 가능) |
| 쓰임 | 단일 변수 atomic | 여러 줄 atomic |
| 가독성 | atomic 단일 호출 | 명시적이라 자연 |

**이 경우**: 단일 `long` 변수 atomic → `Interlocked` 자연.

여러 변수 묶음 atomic이 필요한 곳(예: `_lastHeartbeatTicks` + `_isDisconnected` 두 변수 한 묶음 갱신)는 `lock` 자연. 다만 이 경우는 `_isDisconnected`가 HealthMonitor 단일 스레드라 동시성 문제 없음 → `Interlocked`로 충분.

## 8. 두절 표면화 흐름

### 8.1 HealthMonitor → State Service

```csharp
protected override async Task ExecuteAsync(CancellationToken stoppingToken)
{
    using var timer = new PeriodicTimer(TimeSpan.FromSeconds(5));
    while (await timer.WaitForNextTickAsync(stoppingToken))
    {
        var elapsed = DateTime.UtcNow - _adapter.LastHeartbeatUtc;
        var isTimedOut = elapsed > TimeSpan.FromSeconds(30);

        if (isTimedOut && !_isDisconnected)
        {
            // 두절 감지 시점
            _isDisconnected = true;
            await _stateService.NotifyAdapterDisconnectAsync(
                AdapterType.GeneralProcessWip);
        }
        else if (!isTimedOut && _isDisconnected)
        {
            // 자연 복귀 (재연결 표면화는 별도)
            _isDisconnected = false;
        }
    }
}
```

**중복 알림 방지**: `_isDisconnected` 플래그로 한 번만 알림. 두절이 30초 지속되어도 NotifyAdapterDisconnect는 한 번.

### 8.2 State Service 수신 atomic 처리

`NotifyAdapterDisconnectAsync` 호출 시 State Service는 입력 Channel에 두절 도메인 객체 Write. 단일 Consumer가 이 입력을 atomic 단위로 처리:

```csharp
// State Service 안 (단순화)
public async Task NotifyAdapterDisconnectAsync(AdapterType adapterType)
{
    await _inputChannel.Writer.WriteAsync(new AdapterDisconnectedEvent(adapterType));
}

// 단일 Consumer 루프
private async Task ConsumeLoopAsync()
{
    await foreach (var input in _inputChannel.Reader.ReadAllAsync())
    {
        await ProcessAtomicAsync(input);
    }
}

private async Task ProcessAtomicAsync(IDomainInput input)
{
    switch (input)
    {
        case AdapterDisconnectedEvent ev:
            await HandleAdapterDisconnectAsync(ev);
            break;
        // 다른 입력 처리...
    }
}

private async Task HandleAdapterDisconnectAsync(AdapterDisconnectedEvent ev)
{
    // atomic 순서 (한 묶음):
    // 1. InMemory 권위 측 Null 초기화
    NullifyAdapterAuthorityState(ev.AdapterType);

    // 2. Slot Block 마킹 (AdapterDisconnect)
    MarkSlotsBlocked(ev.AdapterType, BlockReason.AdapterDisconnect);

    // 3. 진행 중 Job/Transfer 종료
    TerminateInFlightJobsAndTransfers(ev.AdapterType);

    // 4. DB 영속화 요청 (비동기 Worker 위임)
    await _persistenceWorker.QueueAsync(/* ... */);

    // 5. 도메인 Event 발행 (다른 서비스 구독)
    await _eventBus.PublishAsync(new AdapterDisconnectedDomainEvent(ev));

    // 6. SignalR Push (Dashboard 자연 반영)
    await _signalRHub.BroadcastAsync(/* ... */);
}
```

**중요**: 이 6단계가 **단일 atomic 단위** — 다음 입력은 이 처리 종료 후 Pull.

### 8.3 SAD §5.11 구조와 정합

SAD §5.11 Adapter 두절 표면화 단락 구조:

> Adapter가 두절 사실을 State Service에 전달하면, State Service는 [수신] atomic 처리로 해당 Adapter 권위 측 InMemory를 통째 Null 초기화하고 Block 마킹(AdapterDisconnect)·진행 중 Job/Transfer 종료를 일괄 수행한다(권위 일원화 구조 — 단일 atomic 안에서 처리). ... Block된 Slot은 §5.6 Job 직전 검증을 통과하지 못하고 §5.8 Fallback 판단에서 가용 Slot에서 제외된다.

→ 위 코드 구조와 정합. SAD 본체에 담긴 구조의 한 적용 사례.

### 10.1 인터페이스 (Port)

```csharp
// Port 인터페이스 — Hexagonal의 외부 경계
// 비즈니스 로직(State Service 등)이 의존하는 추상 계약
public interface IGeneralProcessWipAdapter
```

```csharp
// 외부 Payload DTO
public record WipEventDto
{
```

### 10.2 Adapter

```csharp
public class GeneralProcessWipAdapter : IGeneralProcessWipAdapter
{
    // State Service 의존성 — DI로 Singleton 주입
    private readonly IStateService _stateService;

    // ILogger — ASP.NET Core 표준 Logging
    private readonly ILogger<GeneralProcessWipAdapter> _logger;

    // 마지막 헬스 신호 수신 시각 (UTC tick)
    // long 타입 — 64비트, Interlocked로 atomic 처리
    // 초기값: 인스턴스 생성 시점 (Host 시작 시점)
    private long _lastHeartbeatTicks = DateTime.UtcNow.Ticks;

    // 생성자 주입 — DI Container가 의존성 자동 주입
    public GeneralProcessWipAdapter(
        IStateService stateService,
        ILogger<GeneralProcessWipAdapter> logger)
    {
        _stateService = stateService;
        _logger = logger;
    }

    // 마지막 수신 시각 — Interlocked로 atomic 읽기
    // DateTimeKind.Utc 명시로 시각대 어긋남 방지
    public DateTime LastHeartbeatUtc =>
        new(Interlocked.Read(ref _lastHeartbeatTicks), DateTimeKind.Utc);

    public async Task<IngestResult> IngestEventAsync(WipEventDto dto)
    {
        // 1. 마지막 수신 시각 갱신 — Interlocked atomic 쓰기
        //    어떤 호출이든 "수신 자체"가 헬스 신호
        Interlocked.Exchange(ref _lastHeartbeatTicks, DateTime.UtcNow.Ticks);

        // 2. HealthCheck only면 도메인 변환 skip + Ack 회신
        if (dto.IsHeartbeatOnly)
        {
            _logger.LogDebug("Heartbeat received");
            return IngestResult.HeartbeatAck;
        }

        // 3. 외부 DTO → 도메인 객체 변환 (Mapper 처리)
        var domainEvent = WipEventMapper.ToDomain(dto);

        // 4. State Service 입력 Channel에 전달
        //    내부적으로 ChannelWriter로 Write — 비동기 방식
        //    State Service 단일 Consumer가 FIFO 순으로 Pull
        await _stateService.IngestAsync(domainEvent);

        // 5. 처리 결과 회신 (Unit ID 등)
        //    State Service 측 Unit Mapping 결과를 외부에 알림
        return IngestResult.WithUnitId(domainEvent.UnitId);
    }
}
```

### 10.3 HealthMonitor

```csharp
// BackgroundService 상속 — Host 자동 시작·종료
public class GeneralProcessWipHealthMonitor : BackgroundService
{
    // Adapter 의존성 — 같은 Singleton 인스턴스 주입
    // (Controller가 받는 인스턴스와 동일 — DI 주입)
    private readonly IGeneralProcessWipAdapter _adapter;

    // State Service 의존성 — 두절 알림 대상
    private readonly IStateService _stateService;

    // ILogger
    private readonly ILogger<GeneralProcessWipHealthMonitor> _logger;

    // 점검 주기 — 5초마다
    // 이 값을 IConfiguration에서 읽는 방식도 자연 (운영 환경별로)
    private readonly TimeSpan _checkInterval = TimeSpan.FromSeconds(5);

    // 두절 판정 임계값 — 30초
    // SRS 권위 값 (외부 측 가용성 요구) — 운영 환경별 IConfiguration
    private readonly TimeSpan _heartbeatTimeout = TimeSpan.FromSeconds(30);

    // 두절 상태 플래그 — 중복 알림 방지
    // 이 변수는 ExecuteAsync 단일 스레드라 동시성 문제 없음
    private bool _isDisconnected = false;

    // 생성자 주입
    public GeneralProcessWipHealthMonitor(
        IGeneralProcessWipAdapter adapter,
        IStateService stateService,
        ILogger<GeneralProcessWipHealthMonitor> logger)
    {
        _adapter = adapter;
        _stateService = stateService;
        _logger = logger;
    }

    // Host가 시작 시점에 한 번 호출
    // stoppingToken: Host 종료 시점에 cancel 신호
    protected override async Task ExecuteAsync(CancellationToken stoppingToken)
    {
        _logger.LogInformation("HealthMonitor started, interval={Interval}, timeout={Timeout}",
            _checkInterval, _heartbeatTimeout);

        // PeriodicTimer — 정확한 주기 보장
        // using으로 Host 종료 시 자연 Dispose
        using var timer = new PeriodicTimer(_checkInterval);

        // while 루프 — 반복
        // WaitForNextTickAsync — 다음 주기까지 비동기 대기 (block 없음 — ThreadPool 자유)
        // stoppingToken cancel 시 OperationCanceledException 발생 → 루프 자연 종료
        try
        {
            while (await timer.WaitForNextTickAsync(stoppingToken))
            {
                // 1. 마지막 수신 시각 읽기 — Adapter Singleton 인스턴스
                var elapsed = DateTime.UtcNow - _adapter.LastHeartbeatUtc;

                // 2. 주기 초과 판정
                var isTimedOut = elapsed > _heartbeatTimeout;

                // 3. 두절 감지 — 첫 발화
                if (isTimedOut && !_isDisconnected)
                {
                    _isDisconnected = true;
                    _logger.LogWarning(
                        "WIP adapter disconnected — elapsed={Elapsed}s",
                        elapsed.TotalSeconds);

                    // 두절 사실을 State Service에 전달
                    // State Service [수신] atomic 안에서 InMemory Null 초기화 +
                    // Slot Block 마킹 + 진행 중 Job/Transfer 종료 일괄 처리
                    await _stateService.NotifyAdapterDisconnectAsync(
                        AdapterType.GeneralProcessWip);
                }
                // 4. 자연 복귀 — 두절 상태였는데 다시 정상 수신
                else if (!isTimedOut && _isDisconnected)
                {
                    _isDisconnected = false;
                    _logger.LogInformation("WIP adapter reconnected");
                    // 재연결 별도 알림 없음 — 재수신 자체가 §5.7 정합성 검사
                    // 자동 작동으로 자연 표면화
                }
            }
        }
        catch (OperationCanceledException)
        {
            // Host 종료 신호 — graceful 종료
            _logger.LogInformation("HealthMonitor stopping");
        }
        catch (Exception ex)
        {
            // 예상 외 예외 — logging + Host에 전파
            _logger.LogError(ex, "HealthMonitor crashed");
            throw;
        }
    }
}
```

### 10.5 Program.cs

```csharp
// 진입점 — Host 빌드·실행

var builder = WebApplication.CreateBuilder(args);

// === DI 등록 ===

// 1. Adapter — Singleton (마지막 수신 시각 상태 공유)
builder.Services.AddSingleton<IGeneralProcessWipAdapter, GeneralProcessWipAdapter>();
builder.Services.AddSingleton<IRestackProcessWipAdapter, RestackProcessWipAdapter>();
// (다른 Adapter — GM·SM·MQTT·SignalR Hub 등)

// 2. HealthMonitor — Hosted Service (Host 자동 시작·종료)
//    AddHostedService는 내부적으로 Singleton 등록 + Host 시작 결합
builder.Services.AddHostedService<GeneralProcessWipHealthMonitor>();
builder.Services.AddHostedService<RestackProcessWipHealthMonitor>();

// 3. State Service — Singleton (Core 권위 일원화)
builder.Services.AddSingleton<IStateService, StateService>();

// 4. Transfer/AMMR Service — Singleton
builder.Services.AddSingleton<ITransferService, TransferService>();
builder.Services.AddSingleton<IAmmrService, AmmrService>();

// 5. Controller — Scoped (자동 등록)
builder.Services.AddControllers();

// 6. Logging — ASP.NET Core 기본 Logging
builder.Services.AddLogging();

// === 빌드·실행 ===

var app = builder.Build();

// HTTP 라우팅
app.MapControllers();

// Host 실행 — 이 호출이 block됨
//   Kestrel HTTP 서버 시작
//   모든 HostedService.StartAsync 호출 → ExecuteAsync 시작
//   Ctrl+C·SIGTERM 수신 시까지 살아 있음
//   종료 신호 수신 시:
//     모든 HostedService.StopAsync(token) 호출
//     진행 중 HTTP 요청 graceful 종료
//     프로세스 종료
app.Run();
```

이 문서로 Adapter 헬스 모니터 구조의 구현 방식을 한자리에. 다른 Adapter(GM·SM·MQTT)도 이 구조의 변형 — 외부 측 특성에 따라 자연 분기. MQTT는 라이브러리 disconnect Callback으로 단일 Class, GM/SM은 REST Polling 응답 Timeout 방식으로 이 문서와 비슷한 구조.
