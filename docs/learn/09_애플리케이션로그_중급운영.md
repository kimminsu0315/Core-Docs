## 1. 개요 — Log의 3계층 분류

Core 시스템에서 다루는 Log는 목적과 보존 정책에 따라 3계층으로 나뉩니다.

| 계층 | 이름                                       | 위치  | 보존    | 누가 보나               | 예시                                                                |
|------|--------------------------------------------|-------|---------|-------------------------|---------------------------------------------------------------------|
| 1    | **비즈니스 이력** (Transfer·Job·Unit 위치) | DB    | 1년 이상 | 운영자, 생산관리 담당자  | "U-045 / 14:20 / 세척 WIP 출고 / J-789"                              |
| 2    | **보안 Log** (Block 요약·사용자 변경)     | DB    | 1년 이상 | 관리자, 감사            | "user_3 / 14:25 / Slot Block 해제 / WIP-A.S2"                        |
| 3    | **애플리케이션 Log** (시스템 동작 이력)    | 파일  | 100일   | 개발자, 운영팀          | "[14:32:15.234] State Service: SlotStateChanged WIP-A.S2 occupied→empty" |

이 문서는 **계층 3 애플리케이션 Log**를 다룹니다. 계층 1·2의 세부 내용(Transfer 이력·Job 이력·Unit 위치 변경 이력·Block 요약 Event·보안 Log)은 본체 SRS·SAD에서 정의됩니다.

## 2. 애플리케이션 Log의 목적과 범위

### ※ 목적

Core가 운영 중 InMemory에서 어떤 Event가 어떤 순서로 처리됐는지 추적하기 위한 Log입니다. Core의 핵심 InMemory 권위(TransferList·AmmrList·Slot 상태·Unit 위치 등)는 State Service에 모이고 일부는 DB에 영속화되지 않으므로, 사후 디버깅이 가능하려면 이 Log가 필수입니다.

### ※ 범위

다음 변화가 발생할 때마다 Log를 남깁니다:

- InMemory 상태 변화 (Slot 상태·Unit 위치·AMMR 상태 등)
- 서비스 간 Event 발행/구독 (State Service Event 발행 → Transfer·AMMR Service 구독)
- 외부 시스템과의 통신 (GM/SM 폴링, MQTT 송수신, WIP Event 수신, SignalR Push)
- 의사결정 결과 (Transfer 생성·skip, Transfer↔AMMR 매칭, Job 발행, 동적 Fallback 진입)
- 예외와 오류

### ※ 비즈니스 이력 / 보안 Log와의 차이

| 구분          | 비즈니스 이력             | 보안 Log                  | 애플리케이션 Log   |
|---------------|---------------------------|----------------------------|---------------------|
| 무엇을 기록   | 비즈니스 결과 (Unit 이동) | 사람 변경 + Block 요약     | 시스템 내부 동작    |
| 누가 본다     | 운영자                    | 관리자/감사                | 개발자/운영팀       |
| 보존 기간     | 1년 이상                  | 1년 이상                   | 100일               |
| 저장소        | PostgreSQL                | PostgreSQL                 | 파일 (Serilog)      |
| 형식          | 정규화된 Table           | 정규화된 Table            | JSON 라인           |

## 3. 기술 선택 — Serilog + 구조화 Logging

### ※ 선택 이유

ASP.NET Core 표준 `ILogger` 인터페이스를 그대로 쓰면서 출력만 Serilog로 받습니다. 이렇게 하면 다음 이점이 있습니다:

- 표준 인터페이스 사용 → 코드 변경 없이 향후 다른 Logging 라이브러리로 교체 가능
- Serilog의 구조화 출력 (JSON 라인) 자동 적용
- 파일 Rolling(일자별 분할) 기본 지원
- 향후 외부 저장소(ELK, Loki, Seq 등) 연동 시 sink 추가만으로 가능

### ※ 핵심 패키지

- `Serilog.AspNetCore`
- `Serilog.Sinks.File` (파일 출력)
- `Serilog.Formatting.Compact` (JSON 라인 형식)

## 4. Log 형식 표준

### 4.1 공통 필드

모든 Log 라인은 다음 4개 필드를 반드시 포함합니다.

| 필드      | 타입                   | 설명                                                                            |
|-----------|------------------------|---------------------------------------------------------------------------------|
| `ts`      | ISO 8601 (밀리초 포함) | Event 발생 시각                                                                |
| `level`   | string                 | Log 레벨 (debug / info / warn / error)                                         |
| `source`  | string                 | 발생 Component (StateService / TransferService / AmmrService / GmAdapter 등)     |
| `event`   | string                 | Event 유형 식별자 (PascalCase)                                                  |

### 4.2 Context 필드

Event 종류에 따라 다음 Context 필드를 추가합니다. 디버깅 추적의 4가지 출발점(Unit / Job / AMMR / 시점)이 잘 작동하도록 일관되게 사용합니다.

| 필드           | 타입    | 사용 시점                              |
|----------------|---------|----------------------------------------|
| `unit_id`      | string  | Unit 관련 Event (예: U-045)            |
| `transfer_id`  | string  | Transfer 관련 Event (예: T-456)        |
| `job_id`       | string  | Job 관련 Event (예: J-789)             |
| `ammr_id`      | int     | AMMR 관련 Event (1~N)                  |
| `wip`          | string  | WIP 식별자 (예: WIP-A)                  |
| `slot`         | int     | Slot 번호                               |
| `from` / `to`  | string  | 상태 변화 (예: occupied → empty)        |
| `reason`       | string  | 의사결정 근거 또는 오류 사유            |
| `score`        | float   | 매칭 점수 등 수치 결과                  |

### 4.3 예시 Log 라인

```json
{"ts":"2026-04-10T14:32:15.234Z","level":"info","source":"StateService","event":"SlotStateChanged","wip":"WIP-A","slot":2,"from":"occupied","to":"empty","trigger":"slot_sensor_off","unit_id":"U-045"}
{"ts":"2026-04-10T14:32:15.241Z","level":"info","source":"TransferService","event":"TransferCreated","transfer_id":"T-456","origin":"WIP-A.S2","dest":"WIP-B.S1","unit_id":"U-045"}
{"ts":"2026-04-10T14:32:15.456Z","level":"info","source":"AmmrService","event":"TransferMatched","transfer_id":"T-456","ammr_id":1,"score":0.87}
{"ts":"2026-04-10T14:32:15.460Z","level":"info","source":"AmmrService","event":"JobDispatched","job_id":"J-789","job_type":"pickup","ammr_id":1,"transfer_id":"T-456"}
{"ts":"2026-04-10T14:32:16.012Z","level":"warn","source":"AmmrService","event":"JobResultReceived","job_id":"J-789","job_result":"failure","reason":"ammr_hw_vision_fault","ammr_id":1}
{"ts":"2026-04-10T14:32:16.018Z","level":"warn","source":"StateService","event":"BlockMarked","target":"AMMR1.S3","reason":"AmmrHwError","unit_id":"U-045"}
```

## 5. 서비스별 Log Event 목록

각 서비스가 발생시켜야 할 Event의 권장 목록입니다. 구현 시 이 목록을 참고하되, 필요에 따라 추가/조정 가능합니다.

본체 §5.5·§5.6에 따라 권위·게이트웨이 구조는 다음과 같습니다:
- TransferList·AmmrList = State Service InMemory 권위 단일
- SignalR Hub Push·MQTT 발신 = State Service 단일 게이트웨이
- AMMR 상태 변화·Job 수행 결과 = 통합 Event (각 타입별 별도 Event 없음)

### 5.1 State Service Event

| Event                     | 레벨    | 발생 시점                                  | 주요 필드                                 |
|----------------------------|---------|--------------------------------------------|-------------------------------------------|
| `SlotStateChanged`         | info    | Slot 점유/비어있음 변화 감지                | wip, slot, from, to, trigger, unit_id      |
| `BlockMarked`              | warn    | Slot Block 발생 (사유 enum 별도)            | target, reason, unit_id                    |
| `BlockResolved`            | info    | Slot Block 해제 (사유 enum 별도)            | target, by (user_id 또는 system)           |
| `UnitLocationUpdated`      | info    | Unit 위치 InMemory 갱신                      | unit_id, from_location, to_location        |
| `AmmrStateChanged`         | info    | AMMR 상태 변화 통합 Event (3축 새 값 일괄)  | ammr_id, status, position, battery         |
| `StateChangeEventPublished`| debug   | 다른 서비스로 Event 발행                    | event_type, subscribers                    |

**Block 사유 enum** (BlockMarked/BlockResolved의 `reason` 필드 값):
- `AdapterDisconnect` / `AdapterRestored`
- `AmmrHwDisconnect` / `AmmrHwReconnected`
- `AmmrHwError` / `AmmrHwRecovered`
- `IntegrityMismatch` / `IntegrityRestored`
- `ArrivalValidationFail` / `ArrivalRetryResolved`
- `RestackRecipeMismatch` / `RestackRecipeRestored`

## 6. 파일 저장 정책

### 6.1 파일 위치 및 Rolling

```
/var/log/core/
├── core-20260408.log
├── core-20260409.log
├── core-20260410.log     ← 오늘 (현재 쓰여지는 중)
└── ...
```

- 일자별 Rolling: 매일 자정에 새 파일 생성
- 한 파일이 지나치게 커지는 경우 추가 분할 가능 (Serilog `rollOnFileSizeLimit` 옵션)
- 로컬 디스크 또는 Mount된 Log 볼륨에 저장

### 6.2 보존 기간

- 기본 보존: 100일
- Serilog `retainedFileCountLimit` 옵션으로 자동 삭제 설정
- 오래된 Log가 필요한 경우 별도 백업 정책 수립 (현재는 없음)

### 6.3 한 파일 통합 정책

서비스별로 파일을 분리하지 않고 **모든 서비스가 한 파일에 시간 순으로 기록**합니다.

이유:
- 디버깅 시 시간 순 흐름이 가장 중요
- 서비스 간 Event 인과 관계 추적이 한 파일 내에서 가능
- 분리하면 두 파일을 번갈아 보면서 시각 맞추는 부담

서비스 구분은 `source` 필드로 충분합니다. 필요 시 `grep '"source":"AmmrService"'`로 특정 서비스만 추출 가능합니다.

## 7. 디버깅 추적 패턴 — 4가지 출발점

디버깅은 항상 다음 4가지 중 하나에서 출발합니다. Log 필드 설계가 이 4가지 패턴을 모두 지원하도록 일관되게 유지합니다.

### 7.1 Unit 추적

특정 Unit이 어디로 갔는지, 무슨 일이 있었는지 시간 순으로 본다.

```bash
grep '"unit_id":"U-045"' /var/log/core/core-20260410.log
```

### 7.2 Job 추적

특정 Job의 생명주기를 본다 (발행 → 수행 → 결과 보고).

```bash
grep '"job_id":"J-789"' /var/log/core/core-20260410.log
```

Transfer 추적도 같은 구조:

```bash
grep '"transfer_id":"T-456"' /var/log/core/core-20260410.log
```

### 7.3 AMMR 추적

특정 AMMR의 행동을 시간 순으로 본다.

```bash
grep '"ammr_id":1' /var/log/core/core-20260410.log
```

### 7.4 시점 추적

특정 시간대에 시스템 전체에서 무슨 일이 있었는지 본다.

```bash
sed -n '/"ts":"2026-04-10T14:00/,/"ts":"2026-04-10T14:30/p' /var/log/core/core-20260410.log
```

## 8. 디버깅 시나리오 예시

### 8.1 Unit 소실 추적

**상황**: 운영자가 Dashboard에서 "U-045 위치: 미상" 경고를 봤다. 마지막 위치를 찾아야 한다.

**1단계 — 비즈니스 이력 조회**

```sql
SELECT * FROM transfer_history WHERE unit_id = 'U-045' ORDER BY ts DESC LIMIT 5;
```

결과: "14:25 / AMMR 1호기 Pickup 완료 / 출발지 WIP-A.S2"

→ 14:25에 AMMR 1호기가 가져갔음. 이후 행적은 비즈니스 이력에 없음.

**2단계 — 14:25 전후 애플리케이션 Log 조회**

```bash
grep '"unit_id":"U-045"' /var/log/core/core-20260410.log
```

결과:
```json
{"ts":"...14:25:01...","source":"AmmrService","event":"JobResultReceived","job_id":"J-789","job_result":"success","job_type":"pickup","ammr_id":1,"unit_id":"U-045"}
{"ts":"...14:30:15...","source":"AmmrService","event":"JobDispatched","job_id":"J-790","job_type":"dropoff","ammr_id":1,"transfer_id":"T-456","work_location":"WIP-B.S1","expected_unit":"U-045"}
{"ts":"...14:30:16...","source":"AmmrService","event":"JobResultReceived","job_id":"J-790","job_result":"failure","reason":"ammr_hw_vision_fault","ammr_id":1,"slot":3}
{"ts":"...14:30:16...","source":"StateService","event":"BlockMarked","target":"AMMR1.S3","reason":"AmmrHwError","unit_id":"U-045"}
```

**해석**:
- 14:25:01 Pickup 정상 완료
- 14:30:15 Dropoff 시도 시작
- 14:30:16 AMMR HW 비전 센서가 자기 적재 Slot에 Unit 없음을 통합 보고로 알림 (`reason=ammr_hw_vision_fault`)
- State Service가 통합 보고 수신 후 `BlockMarked` 발행 (사유 = AmmrHwError)
- → Pickup과 Dropoff 사이 약 5분 동안 누군가 Unit을 꺼냈다는 결론

**3단계 — 보안 Log 확인**

```sql
SELECT * FROM security_log
WHERE target LIKE '%U-045%' OR target LIKE '%AMMR1.S3%'
ORDER BY changed_at DESC;
```

자동 기록된 Block 요약 Event 확인. 누가 꺼냈는지는 Log로는 알 수 없음 (사람이 QR Scan 안 했음).

**4단계 — 현장 조사**

Log로 추적 가능한 범위는 여기까지. 현장에서 14:25~14:30 시간대 AMMR 1호기 경로 주변 확인.

### 8.4 특정 시간대 전체 흐름 추적

**상황**: 어제 14시쯤 어긋난 동작이 있었다는 막연한 제보. 시간대 전체 흐름을 살펴봐야 함.

**1단계 — 시간 범위로 잘라내기**

```bash
sed -n '/"ts":"2026-04-09T14:00/,/"ts":"2026-04-09T14:30/p' /var/log/core/core-20260409.log > yesterday-14h.log
```

**2단계 — 특정 source만 필터링**

```bash
grep '"source":"StateService"' yesterday-14h.log
```

State Service 발생 Event만 추려서 그 시간대의 상태 변화를 한눈에 파악.

**3단계 — warn/error Event만 추출**

```bash
grep -E '"level":"(warn|error)"' yesterday-14h.log
```

이상 징후가 있었던 Event만 빠르게 확인.

**4단계 — 의심 Event 찾으면 해당 unit_id/transfer_id/job_id/ammr_id로 다시 추적**

`grep '"unit_id":"X"'` 등으로 좁혀서 인과 관계 파악.

## 9. Serilog 설정 예시

### 9.1 패키지 설치

```bash
dotnet add package Serilog.AspNetCore
dotnet add package Serilog.Sinks.File
dotnet add package Serilog.Formatting.Compact
```

### 9.2 Program.cs 설정

```csharp
using Serilog;
using Serilog.Formatting.Compact;

var builder = WebApplication.CreateBuilder(args);

builder.Host.UseSerilog((context, services, configuration) => configuration
    .MinimumLevel.Information()
    .MinimumLevel.Override("Microsoft", Serilog.Events.LogEventLevel.Warning)
    .Enrich.FromLogContext()
    .WriteTo.File(
        formatter: new CompactJsonFormatter(),
        path: "/var/log/core/core-.log",
        rollingInterval: RollingInterval.Day,
        retainedFileCountLimit: 100,
        rollOnFileSizeLimit: true,
        fileSizeLimitBytes: 100 * 1024 * 1024  // 100MB
    )
);

// ... 나머지 설정
```

### 9.3 Logging 사용 예시

```csharp
public class StateService
{
    private readonly ILogger<StateService> _logger;

    public StateService(ILogger<StateService> logger)
    {
        _logger = logger;
    }

    public void HandleSlotStateChange(string wip, int slot, string from, string to, string trigger, string? unitId)
    {
        _logger.LogInformation(
            "SlotStateChanged {Wip}.{Slot} {From}→{To} trigger={Trigger} unit={UnitId}",
            wip, slot, from, to, trigger, unitId
        );
    }
}
```

위 코드가 다음 JSON 라인으로 출력됩니다:

```json
{"ts":"...","level":"info","source":"StateService","event":"SlotStateChanged","wip":"WIP-A","slot":2,"from":"occupied","to":"empty","trigger":"slot_sensor_off","unit_id":"U-045"}
```

### 9.4 source / event 필드 자동화

`source`는 `ILogger<T>`의 타입명에서 자동 추출되도록 설정 가능합니다. `event`는 메시지 템플릿의 첫 단어를 사용하는 Convention을 정해두면 일관성 있게 관리됩니다.

자세한 Mapping 규칙은 구현 시점에 정합니다.

## 10. 향후 확장 — 분석 도구 도입 검토 여지

초기에는 파일 + grep으로 충분합니다. 운영 규모가 커지거나 운영팀 요청이 있으면 다음 도구 도입을 검토합니다.

### 10.1 Seq

- .NET 친화적, Serilog와 가장 자연스러운 연동
- 작은 팀에 적합
- 웹 UI로 검색/시각화
- 로컬 설치 또는 Docker Container로 운영

### 10.2 Loki + Grafana

- 가벼운 대안. ELK보다 운영 부담 적음
- Grafana Dashboard와 통합되어 메트릭/Log 한 화면에서 보기 가능
- Container 환경에 적합

### 10.3 ELK (Elasticsearch + Logstash + Kibana)

- 대규모 운영에 적합
- 강력한 검색/시각화 기능
- 인프라 부담 큼 (메모리/디스크/운영)

### 10.4 도입 시점 판단 기준

- 일일 Log 양이 1GB를 넘기 시작
- grep으로 검색하는 데 수십 초 이상 걸림
- 운영팀이 Log 검색에 자주 시간을 쓰게 됨
- 시각화/Dashboard 요구가 명확해짐

위 조건 중 2개 이상 충족 시 도입을 본격 검토합니다.
