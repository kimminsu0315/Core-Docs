## 1. 이 자료의 차원

본체 §4.5에 담긴 **권위 일원화 구조**의 코드 패턴을 세 서비스 코드 예시로 보여준다.

> 모든 InMemory 권위가 State Service 한 곳에 모이면서 동시성·정합성 부담이 단일 atomic 경계에 집약된다. (§4.5)

핵심 질문: TransferList·AmmrList 권위가 State Service에 있는데, Transfer Service·AMMR Service는 어떻게 그 List를 정렬·매칭 결정하고 갱신을 요청하는가?

자연스러운 오해 — "Transfer Service가 List를 받아와서 정렬한 뒤 다시 리턴." 이 방식은 본체와 다르다. 실제 방식은 **읽기·쓰기가 분리된 권위 일원화** 패턴이다.

---

## 2. 권위 일원화 구조의 핵심

세 서비스 협업의 코드 차원 구조:

1. **읽기 = direct lookup** — Transfer Service·AMMR Service가 State Service InMemory에 read-only 참조(`IReadOnlyDictionary` / `IReadOnlyList` 등)로 접근. 같은 프로세스 안 같은 객체에 대한 참조 — 중간 네트워크·복사 경유 없음. List 자체가 손에 들어오는 게 아니라 read-only 창으로 들여다본다.
2. **도메인 결정** — 두 서비스가 각자 책임 영역에서 결정 수행 (Transfer Service: 우선순위 정렬 / AMMR Service: 매칭·Job Sequence 전개).
3. **쓰기 = State Service 메서드 경유** — 결정 결과 도메인 객체를 State Service의 쓰기 메서드 인자로 전달. 두 서비스가 List를 직접 수정하지 않는다.
4. **atomic 일괄** — State Service가 atomic 안에서 InMemory 갱신 + 이력 영속화 요청 + Event 발행 + SignalR Push를 일괄 수행 (§5.5·§5.6 atomic 처리 방식).

이 방식으로 (a) 읽기 성능(direct lookup) (b) 쓰기 권위 일원화 (c) atomic 단순성이 양립한다. 본체 §5.6에서 명시:

> '직접 조회'는 두 가지를 의미한다 — (a) 시점 측면: Event Cache가 아닌 권위 저장소에서 현 시점 값을 즉각 Pull, (b) 접근 측면: State Service가 노출하는 읽기 전용 참조를 통한 direct lookup(중간 네트워크·복사 경유 없음). 쓰기 경로는 이와 분리되며 State Service가 제공하는 쓰기 메서드를 통해서만 가능하다.

## 3. State Service — 권위 측

### 3.1 자료구조 보유

State Service가 모든 InMemory 권위 자료구조를 단일 보유. 외부에는 직접 노출하지 않는다.

```csharp
public class StateService
{
    // InMemory 권위 자료구조 — State Service만 직접 접근
    private readonly SortedList<int, Transfer> _transferList = new();
    private readonly Dictionary<AmmrSlotId, AmmrSlot> _ammrSlots = new();
    private readonly Dictionary<SlotId, Slot> _wipSlots = new();

    // 외부 의존성
    private readonly IHistoryWorker _historyWorker;
    private readonly IEventBus _eventBus;
    private readonly ISignalRHub _signalRHub;

    // 입력 Channel — 단일 Consumer atomic 경계 (§4.5)
    private readonly Channel<IDomainCommand> _inputChannel;

    public StateService(
        IHistoryWorker historyWorker,
        IEventBus eventBus,
        ISignalRHub signalRHub)
    {
        _historyWorker = historyWorker;
        _eventBus = eventBus;
        _signalRHub = signalRHub;
        _inputChannel = Channel.CreateUnbounded<IDomainCommand>();
    }
}
```

### 3.2 읽기 전용 참조 노출

다른 서비스가 들여다볼 수 있게 read-only 참조만 외부 공개. value 타입은 immutable record로 두어 외부 서비스가 내부 필드를 직접 변경하지 못하게 한다 (§4.6).

```csharp
public class StateService
{
    // 읽기 전용 View — direct lookup 경로
    public IReadOnlyList<Transfer> TransferListView
        => _transferList.Values.ToList();

    public IReadOnlyDictionary<AmmrSlotId, AmmrSlot> AmmrSlotsView
        => _ammrSlots;

    public IReadOnlyDictionary<SlotId, Slot> WipSlotsView
        => _wipSlots;

    // 단일 항목 조회 헬퍼
    public Slot? GetSlot(SlotId id)
        => _wipSlots.TryGetValue(id, out var s) ? s : null;

    public AmmrSlot? GetAmmrSlot(AmmrSlotId id)
        => _ammrSlots.TryGetValue(id, out var s) ? s : null;
}

// 권위 데이터는 immutable record로 — 외부 서비스가 내부 필드 변경 불가
public record Slot(
    SlotId Id,
    SensorState SensorState,
    UnitId? UnitIdValue,
    UnitIdGrade Grade,        // Confirmed / Estimated
    BlockReason? BlockReason);

public record AmmrSlot(
    AmmrSlotId Id,
    SensorState SensorState,
    UnitId? UnitIdValue,
    UnitIdGrade Grade,
    bool IsFull,              // 만재 라벨
    BlockReason? BlockReason);

public record Transfer(
    TransferId Id,
    UnitId UnitId,
    SlotId SourceSlot,
    SlotId Destination,
    int Priority,
    TransferStatus Status);
```

### 3.3 쓰기 메서드 — atomic 처리

쓰기 메서드는 모두 입력 Channel을 거쳐 단일 Consumer가 atomic 안에서 처리. InMemory 갱신·정합성 검사·DB 영속화 요청·Event 발행·SignalR Push가 일괄 (§5.5 [수신] atomic 방식).

```csharp
public class StateService
{
    // 쓰기 메서드 — Transfer Service가 호출
    public async Task RegisterTransferAsync(Transfer transfer, int sortKey)
    {
        await _inputChannel.Writer.WriteAsync(
            new RegisterTransferCommand(transfer, sortKey));
    }

    public async Task RegisterTransferSkipAsync(UnitId unitId)
    {
        await _inputChannel.Writer.WriteAsync(
            new RegisterTransferSkipCommand(unitId));
    }

    public async Task InvalidateTransferAsync(TransferId id)
    {
        await _inputChannel.Writer.WriteAsync(
            new InvalidateTransferCommand(id));
    }

    // 쓰기 메서드 — AMMR Service가 호출
    public async Task ConsumeTransferForAmmrAsync(
        TransferId transferId, AmmrSlotId ammrSlotId)
    {
        await _inputChannel.Writer.WriteAsync(
            new ConsumeTransferCommand(transferId, ammrSlotId));
    }

    public async Task MarkAmmrSlotFullAsync(AmmrSlotId id)
    {
        await _inputChannel.Writer.WriteAsync(
            new MarkAmmrSlotFullCommand(id));
    }

    public async Task DispatchJobAsync(Job job)
    {
        await _inputChannel.Writer.WriteAsync(
            new DispatchJobCommand(job));
    }

    // 단일 Consumer — atomic 경계
    private async Task ConsumeLoopAsync(CancellationToken ct)
    {
        await foreach (var cmd in _inputChannel.Reader.ReadAllAsync(ct))
        {
            // atomic 안에서 일괄
            switch (cmd)
            {
                case RegisterTransferCommand c:
                    await HandleRegisterTransferAsync(c);
                    break;
                case ConsumeTransferCommand c:
                    await HandleConsumeTransferAsync(c);
                    break;
                case MarkAmmrSlotFullCommand c:
                    await HandleMarkAmmrSlotFullAsync(c);
                    break;
                case DispatchJobCommand c:
                    await HandleDispatchJobAsync(c);
                    break;
                // ...
            }
        }
    }

    private async Task HandleRegisterTransferAsync(RegisterTransferCommand c)
    {
        // ① InMemory 갱신
        _transferList.Add(c.SortKey, c.Transfer);

        // ② 정합성 검사 — 이 사건은 자연 정합 (등록은 신규)

        // ③ DB 영속화 요청 (비동기 Worker 위임 — §4.6)
        await _historyWorker.EnqueueAsync(
            new TransferCreatedHistory(c.Transfer));

        // ④ Event 발행 (구독자 = AMMR Service)
        await _eventBus.PublishAsync(
            new TransferRegisteredEvent(c.Transfer));

        // ⑤ SignalR Push (Dashboard 실시간 반영)
        await _signalRHub.PushTransferListChangedAsync();
    }

    private async Task HandleDispatchJobAsync(DispatchJobCommand c)
    {
        // 발신 경로 atomic (§5.6 [발신] 코드블록)
        // ① Core 논리 AMMR 상태 갱신 (JobInProgress 부착)
        var ammr = _ammrs[c.Job.AmmrId];
        _ammrs[c.Job.AmmrId] = ammr with
        {
            CoreLogicState = CoreLogicState.JobInProgress(c.Job.JobType)
        };

        // ② Job 발행 이력 기록
        await _historyWorker.EnqueueAsync(new JobDispatchHistory(c.Job));

        // ③ SignalR Hub Push
        await _signalRHub.PushAmmrChangedAsync(c.Job.AmmrId);

        // ④ AMMR Adapter 호출 (도메인 객체 → MQTT payload 변환·발신)
        await _ammrAdapter.PublishJobAsync(c.Job);
    }
}
```

## 4. Transfer Service — 도메인 결정 측 (1)

State Service가 발행한 Event를 구독해 Transfer 생성·우선순위 결정 수행. List 자체에는 손대지 않는다.

```csharp
public class TransferService
{
    private readonly StateService _state;
    private readonly IEventBus _eventBus;

    public TransferService(StateService state, IEventBus eventBus)
    {
        _state = state;
        _eventBus = eventBus;

        // State Service가 발행한 Event 구독
        _eventBus.Subscribe<SlotQrRecognizedEvent>(OnSlotQrRecognizedAsync);
        _eventBus.Subscribe<SlotReleasedEvent>(OnSlotReleasedAsync);
    }

    // 출고 Slot QR 인식 → Transfer 생성·정렬 (§5.3)
    public async Task OnSlotQrRecognizedAsync(SlotQrRecognizedEvent evt)
    {
        if (evt.SlotKind != SlotKind.Outbound) return;

        // ① 읽기 — read-only 참조로 direct lookup
        var existing = _state.TransferListView;

        // ② 멱등 검사 — 동일 Unit 미수행 Transfer 존재? (§5.3)
        if (existing.Any(t => t.UnitId == evt.UnitId
                           && t.Status != TransferStatus.Completed))
        {
            // skip 플래그 True 레코드 기록 요청
            await _state.RegisterTransferSkipAsync(evt.UnitId);
            return;
        }

        // ③ 도메인 결정 — 신규 Transfer + 정렬 위치 산출
        //    (이 계산이 Transfer Service의 비즈니스 로직)
        var destination = ResolveNextDestination(evt.UnitId);
        var transfer = new Transfer(
            Id: TransferId.New(),
            UnitId: evt.UnitId,
            SourceSlot: evt.SlotId,
            Destination: destination,
            Priority: 0,
            Status: TransferStatus.Pending);

        var sortKey = CalculateSortKey(transfer, existing);

        // ④ 쓰기 요청 — 결과 도메인 객체를 State Service에 넘김
        await _state.RegisterTransferAsync(transfer, sortKey);
    }

    // Slot 해제 → 자기 대기 Transfer 무효화 판정 (§5.3)
    public async Task OnSlotReleasedAsync(SlotReleasedEvent evt)
    {
        // ① 읽기
        var pending = _state.TransferListView
            .Where(t => t.SourceSlot == evt.SlotId
                     && t.Status == TransferStatus.Pending)
            .ToList();

        // ② 도메인 결정 — 출발 Slot이 해제됐으면 무효화
        // ③ 쓰기 요청
        foreach (var t in pending)
        {
            await _state.InvalidateTransferAsync(t.Id);
        }
    }

    // 도메인 로직 — Transfer Service 책임
    private SlotId ResolveNextDestination(UnitId unitId) { /* Recipe 기반 다음 공정 산출 */ throw new NotImplementedException(); }
    private int CalculateSortKey(Transfer t, IReadOnlyList<Transfer> existing) { /* 우선순위 계산 */ throw new NotImplementedException(); }
}
```

## 5. AMMR Service — 도메인 결정 측 (2)

매칭 능동 주체로 두 List를 동시 참조. 결정 후 갱신 요청 + Job 발신 요청 모두 State Service 단일 게이트웨이 경유.

```csharp
public class AmmrService
{
    private readonly StateService _state;
    private readonly IEventBus _eventBus;

    public AmmrService(StateService state, IEventBus eventBus)
    {
        _state = state;
        _eventBus = eventBus;

        // 두 갈래 Trigger (§5.6)
        _eventBus.Subscribe<TransferRegisteredEvent>(OnTransferRegisteredAsync);
        _eventBus.Subscribe<AmmrSlotReleasedEvent>(OnAmmrSlotReleasedAsync);
        _eventBus.Subscribe<JobResultEvent>(OnJobResultAsync);
        _eventBus.Subscribe<AmmrStateChangedEvent>(OnAmmrStateChangedAsync);
    }

    // Trigger ① — Transfer 등록 시 자기 AmmrList 확인
    public async Task OnTransferRegisteredAsync(TransferRegisteredEvent evt)
    {
        // ① 두 List 동시 read-only 참조
        var transferList = _state.TransferListView;
        var ammrSlots = _state.AmmrSlotsView;

        // ② 매칭 결정 — AMMR Service 책임
        var match = FindMatch(transferList, ammrSlots);
        if (match is null) return;

        var (transfer, ammrSlot) = match.Value;

        // ③ Job Sequence 전개 — Move → Pickup → Move → Dropoff
        var firstJob = BuildFirstJob(transfer, ammrSlot);

        // ④ Job 직전 검증 — Pickup이면 출발 Slot 점유·Block 아님 검사 (§5.6)
        if (!ValidateJobPreconditions(firstJob))
        {
            await _state.InvalidateTransferAsync(transfer.Id);
            return;
        }

        // ⑤ 쓰기 요청 — Transfer 매칭 확정 + Core 논리 상태 갱신
        await _state.ConsumeTransferForAmmrAsync(transfer.Id, ammrSlot.Id);

        // ⑥ Job 발신 요청 — State Service 발신 게이트웨이 경유
        //    (§5.6·§5.10 — MQTT 양방향 모두 State Service 단일 게이트웨이)
        await _state.DispatchJobAsync(firstJob);
    }

    // Trigger ② — AMMR Slot 해제 시 만재 대기 깨우기 등 (§5.6 + §5.8)
    public async Task OnAmmrSlotReleasedAsync(AmmrSlotReleasedEvent evt)
    {
        var transferList = _state.TransferListView;
        var ammrSlots = _state.AmmrSlotsView;

        // 만재 라벨 해소 우선 평가 (§5.8)
        var fullLabelMatch = FindFullLabelResolution(ammrSlots, transferList);
        if (fullLabelMatch is not null)
        {
            var (transferId, ammrSlotId) = fullLabelMatch.Value;
            await _state.UnmarkAmmrSlotFullAsync(ammrSlotId);
            await _state.ResumeTransferAsync(transferId);
            return;
        }

        // 일반 매칭 — 위와 동일 흐름
        var match = FindMatch(transferList, ammrSlots);
        if (match is null) return;
        // ... (생략 — OnTransferRegisteredAsync와 동일)
    }

    // Job 결과 수신 — payload 분기 Trigger 3필드 + Slot 센서값 갱신 입력
    public async Task OnJobResultAsync(JobResultEvent evt)
    {
        // 분기 Trigger: AMMR HW 상태 + Job 결과 + Reason (§5.5·§5.6)
        // payload 4필드 중 3필드가 분기 키, Slot 센서값은 사후 InMemory 갱신용
        var branch = ClassifyJobResult(evt.HwState, evt.JobResult, evt.Reason);

        switch (branch)
        {
            case JobResultBranch.Success:
                await ProgressToNextJobAsync(evt);
                break;

            case JobResultBranch.PickupFailSlotSide:
                // §5.7 정합성 검사 자연 마킹 + Transfer 실패 종료
                await _state.FailTransferAsync(evt.TransferId);
                break;

            case JobResultBranch.DropoffFailSlotSide:
                // §5.8 동적 Fallback 진입
                await EnterDynamicFallbackAsync(evt);
                break;

            // 분기 1·2 (HW 장애 / HW 측 Reason) 케이스는 State Service 단독 처리
            // — payload 전체 신뢰 없어 도메인 정보 없음 (§5.6 처리 주체 분기)
        }
    }

    // 동적 Fallback (§5.8)
    private async Task EnterDynamicFallbackAsync(JobResultEvent evt)
    {
        // ① Block 마킹 요청
        await _state.MarkSlotBlockAsync(evt.TargetSlotId, BlockReason.IntegrityMismatch);

        // ② 같은 WIP 내 가용 Slot 재탐색
        var alternative = FindAlternativeSlotInSameWip(evt.TargetSlotId);
        if (alternative is not null)
        {
            await _state.RetargetJobAsync(evt.JobId, alternative.Value);
            return;
        }

        // ③ 대체 경로 탐색 (§5.8)
        var fallback = FindFallbackPath(evt);
        if (fallback is not null)
        {
            await _state.RetargetJobAsync(evt.JobId, fallback.Value);
            return;
        }

        // ④ 만재 라벨 부착 요청
        await _state.MarkAmmrSlotFullAsync(evt.AmmrSlotId);
    }

    // Charge Job 자체 생성 — Transfer 미경유 (§5.8)
    public async Task OnAmmrStateChangedAsync(AmmrStateChangedEvent evt)
    {
        // Battery 저전력 분류 진입 인지 → Charge Job 자체 생성
        if (evt.AxisChanged == AxisKind.BatteryClassification &&
            evt.NewValue == BatteryClassification.Low)
        {
            await GenerateChargeJobAsync(evt.AmmrId);
        }
    }

    private async Task GenerateChargeJobAsync(AmmrId ammrId)
    {
        var ammr = _state.GetAmmr(ammrId);
        if (ammr is null) return;

        // 도메인 결정 — Charge Job 도메인 객체 생성
        var chargeJob = new Job(
            JobId.New(),
            JobType.Charge,
            ammrId,
            chargeStation: ResolveChargeStation(ammr.Position));

        // State Service 단일 게이트웨이 경유 — Job 이력 기록 + Core 논리 상태 갱신
        await _state.DispatchJobAsync(chargeJob);
    }

    // 도메인 로직 — AMMR Service 책임
    private (Transfer, AmmrSlot)? FindMatch(
        IReadOnlyList<Transfer> transfers,
        IReadOnlyDictionary<AmmrSlotId, AmmrSlot> ammrSlots) { throw new NotImplementedException(); }
    private Job BuildFirstJob(Transfer t, AmmrSlot s) { throw new NotImplementedException(); }
    private bool ValidateJobPreconditions(Job job) { throw new NotImplementedException(); }
    private (TransferId, AmmrSlotId)? FindFullLabelResolution(
        IReadOnlyDictionary<AmmrSlotId, AmmrSlot> ammrSlots,
        IReadOnlyList<Transfer> transfers) { throw new NotImplementedException(); }
    private SlotId? FindAlternativeSlotInSameWip(SlotId failedSlot) { throw new NotImplementedException(); }
    private SlotId? FindFallbackPath(JobResultEvent evt) { throw new NotImplementedException(); }
    private SlotId ResolveChargeStation(Position pos) { throw new NotImplementedException(); }
    private async Task ProgressToNextJobAsync(JobResultEvent evt) { throw new NotImplementedException(); }
}
```

## 6. 핵심 구조 정리

세 서비스 협업의 구조를 한 표로:

| 서비스 | 책임 | List 접근 방식 | 쓰기 방식 |
|---|---|---|---|
| State Service | 권위 측 | 자료구조 직접 보유 | 단일 atomic 경계 안에서 처리 |
| Transfer Service | Transfer 생성·정렬 결정 | read-only 참조 (direct lookup) | State Service 메서드 경유 갱신 요청 |
| AMMR Service | 매칭·Job Sequence 결정 | read-only 참조 (direct lookup) | State Service 메서드 경유 갱신 요청 + Job 발신 요청 |

**"받아와서 처리 후 리턴"이 아니라 "들여다보고 → 결정 → 권위자에게 갱신 요청" 방식.**

이 방식의 효과:
- **읽기 성능** — direct lookup으로 같은 프로세스 안 같은 객체 참조 (네트워크·복사 없음)
- **쓰기 권위 일원화** — List 수정 코드는 State Service 한 곳에만 존재 → 동시성·정합성 부담 단일 atomic 경계 집약
- **atomic 단순성** — InMemory 갱신·이력 영속화·Event 발행·SignalR Push가 한 단위
- **Push 시차 회피** — 시점 검증은 직접 Pull (Push Event는 권위 갱신과 구독자 수신 사이 시차 존재해 시점 검증에 부적합 — §5.6)

## 7. 다른 학습자료와의 차원 분리

이 자료의 차원이 기존 학습자료 3종과 어떻게 분리되는가:

| 자료 | 차원 |
|---|---|
