## 1. InMemory vs DB 권위 위치 결정

> 이 장은 데이터를 어디에 둘지 결정할 때의 기본 원칙을 정리합니다.

### 1.1 권위 위치(authority) 개념

데이터가 두 곳 이상에 존재할 때, 그 중 하나를 "진실의 근원"으로 정해야 합니다. 이를 권위 위치(source of truth)라고 합니다. 권위 위치가 명확하지 않으면 두 곳의 데이터가 어긋났을 때 어느 쪽을 믿어야 할지 모호해집니다.

원칙: **한 데이터의 권위 위치는 한 곳이어야 한다.**

### 1.2 Core 시스템의 InMemory/DB 분리 원칙

Core는 데이터의 성격에 따라 권위 위치를 명확히 분리합니다.

| 권위 위치              | 다루는 데이터                | 예시                                                                                                              |
|------------------------|------------------------------|-------------------------------------------------------------------------------------------------------------------|
| **InMemory**           | 실시간으로 변하는 현재 상태   | Slot 점유, AMMR 위치/Battery/상태, TransferList·AmmrList (State Service InMemory 권위), Slot Block                   |
| **DB (PostgreSQL)**    | 메타 정보 + 이력 + 영구 설정  | Unit 메타 정보, Tray 타입, 목표일, 보안 Log, Transfer 이력·Job 이력·Unit 위치 변경 이력, Recipe                     |

원칙:

- **이중화 금지**: 같은 데이터를 InMemory와 DB 양쪽에 권위로 두지 않습니다. 한쪽이 권위이면 다른 쪽은 사본(Cache)일 뿐입니다.
- **재시작 시**: InMemory 데이터는 백지 상태에서 시작하여 Event로 자연 재생성됩니다. DB 데이터는 영속적이므로 그대로 유지됩니다.

### 1.3 분류 기준

새로운 데이터의 권위 위치를 정할 때 다음 질문에 답해보면 분류가 명확해집니다.

- **얼마나 자주 변하는가?**
  - 초당 수 회 이상 변함 → InMemory 후보
  - 시간 단위 또는 Event성 변경 → DB 후보
- **재시작 후에도 보존되어야 하는가?**
  - 보존 필요 → DB
  - Event로 자연 재생성 가능 → InMemory
- **이력 추적이 필요한가?**
  - 필요 → DB (Transfer 이력·Job 이력·Unit 위치 변경 이력 또는 보안 Log)
  - 불필요 → InMemory
- **현재 시점의 한 값만 필요한가, 과거 값도 필요한가?**
  - 현재 값만 → InMemory 후보
  - 과거 값 추적 필요 → DB

이 4가지 질문에 답한 결과가 한쪽으로 명확하게 쏠리면 그쪽으로 가면 됩니다. 애매하면 보통 DB가 안전한 기본값입니다.

### 1.4 Cache는 권위가 아닌 사본

Cache는 "DB의 데이터를 자주 쓰니까 빠른 접근을 위해 InMemory에도 복사해두는 것"입니다. 이중화처럼 보이지만 다릅니다.

- **이중화**: 두 쪽이 각자 권위를 가진 상태에서 서로 동기화. 어느 쪽이 진실인지 모호.
- **Cache**: 한쪽(DB)이 권위이고 다른 쪽(InMemory)은 그저 빠른 사본. 진실은 항상 DB.

Core의 "이중화 금지" 원칙은 권위가 분산된 이중화를 금지하는 것이지, Cache를 금지하는 것은 아닙니다. 다만 Cache는 비용이 있으므로 신중하게 도입합니다 (2장 참고).

## 2. Cache의 의미와 비용

> 이 장은 Cache 도입을 결정할 때 알아야 할 비용과 흔한 함정을 정리합니다.

### 2.1 Cache의 본질

Cache의 목적은 단순합니다 — **DB 조회 비용을 줄이는 것**. 자주 조회되는 데이터를 InMemory에 사본으로 두어, DB까지 가지 않고 InMemory에서 답을 줍니다.

### 2.2 Cache 도입의 4가지 비용

Cache는 공짜가 아닙니다. 다음 4가지 비용이 따라옵니다.

**※ 비용 1 — 무효화 로직**

원본 DB 데이터가 변경되면 Cache도 같이 갱신해야 합니다. 이걸 빼먹으면 Cache가 stale 상태가 되고, 사용자는 옛날 데이터를 봅니다.

```csharp
// DB 갱신 시 Cache도 같이 무효화
public async Task UpdateOrderAsync(Order order)
{
    await _db.Orders.UpdateAsync(order);
    _cache.Remove($"order:{order.Id}");  // 잊으면 stale
}
```

**※ 비용 2 — 일관성 검증**

Cache와 DB가 어긋나는 케이스를 모니터링해야 합니다. 무효화 로직에 버그가 있으면 어긋남이 누적됩니다.

**※ 비용 3 — 재시작 시 복구**

Core 재시작 후 Cache는 비어 있습니다. 첫 조회들이 모두 Cache 미스로 DB에 가야 하므로, 재시작 직후 일시적인 부하 스파이크가 발생합니다.

**※ 비용 4 — 메모리 관리**

Cache 크기 제한, 만료 정책(LRU, TTL 등), 메모리 압박 대응 등을 설계해야 합니다. 무한히 Cache를 쌓으면 메모리 부족이 됩니다.

### 2.3 premature optimization 안티패턴

부하가 작은데 미리 Cache를 만들어두면 비용만 발생하고 이득은 거의 없습니다. 이것을 premature optimization(조기 최적화)이라고 부릅니다. 두 가지 함정을 야기합니다.

- **함정 1**: 부담 없는 부분을 미리 최적화해서 코드만 복잡해짐. 시간 낭비 + 버그 위험.
- **함정 2**: 진짜 부담의 원인을 모르고 엉뚱한 곳을 최적화함. 노력 많이 했는데 효과 없음.

원칙: **최적화는 측정 후에**. 먼저 단순하게 만들고, 측정으로 부담의 원인을 확인한 다음, 그 원인에만 최적화를 적용합니다.

## 3. 부하 측정 방법론

> 이 장은 "측정 후 최적화" 원칙을 실제로 적용하기 위한 측정 방법을 정리합니다.

### 3.1 두 차원의 측정

부하 측정은 크게 두 종류로 나뉩니다.

**※ 차원 1 — 시스템 전체의 부하 (블랙박스 측정)**

DB 서버 자체의 자원 사용량을 외부에서 관찰합니다. 코드 변경 없이 측정 가능.

- DB 서버의 CPU/메모리/디스크 IO 사용률 (top, htop, iostat 등)
- PostgreSQL 자체 통계 View (5장 참고)
- 활성 연결 수 / 대기 트랜잭션

이 차원은 "DB 서버가 전반적으로 얼마나 바쁜가"를 보여줍니다.

**※ 차원 2 — 특정 조회 경로의 부하 (코드 차원 측정)**

특정 쿼리나 동작이 얼마나 자주, 얼마나 오래 걸리는지 측정합니다.

- 코드에서 쿼리 실행 시간 측정 (Stopwatch, EF Core interceptor)
- 호출 빈도 카운터
- APM 도구 (Application Insights, OpenTelemetry 등)
- 구조화 Logging 기반 사후 분석 (Serilog)

이 차원은 "어떤 쿼리가 진짜 부담의 원인인가"를 보여줍니다.

### 3.2 단계별 측정 방법

도구를 다 도입할 필요는 없습니다. 단계별로 점진적으로 도입합니다.

**※ 1단계 — 가벼운 측정 (운영 시작 직후)**

가장 적은 비용으로 시작합니다.

- DB 서버의 CPU/메모리 사용률을 가끔 확인 (10% 이내면 거의 신경 쓸 필요 없음)
- PostgreSQL `pg_stat_statements` 확장 활성화 → 한 달에 한 번 정도 Top 10 쿼리 확인
- 17장 애플리케이션 Log(Serilog)에 DB 호출 시간 자동 기록 → grep으로 사후 분석

이 정도가 가장 가벼운 측정입니다. 도구 설치나 인프라 구축 거의 없음.

**※ 2단계 — 구조적 측정 (안정화 후)**

운영이 익숙해지고 부담이 느껴질 때 도입.

- EF Core logging interceptor로 모든 쿼리의 실행 시간을 자동 기록
- 너무 느린 쿼리는 별도 경고 Log로 분리
- OpenTelemetry 도입으로 분산 추적 (요청 → 메서드 → 쿼리 흐름 시각화)

**※ 3단계 — 정밀 측정 (부담 명백할 때)**

특정 쿼리가 진짜 문제로 드러났을 때.

- PostgreSQL `EXPLAIN ANALYZE`로 실행 계획 분석 (인덱스 사용 여부, 풀Scan 여부)
- 부하 테스트 도구 (k6, NBomber 등)로 별도 환경에서 한계점 측정

### 3.3 측정의 본질

"측정 후 최적화"의 진짜 가치는 단순히 "최적화의 근거"가 아니라 **최적화 우선순위 결정의 도구**라는 점입니다.

- 측정 없이 직관에 따라 최적화 → 엉뚱한 곳에 시간 쓸 위험
- 측정 결과에 따라 최적화 → 한정된 시간을 가장 부담 큰 곳에 집중

원칙: **측정 데이터가 가리키는 곳을 최적화한다.**

## 4. PostgreSQL 인스턴스/데이터베이스/스키마 구조

> 이 장은 5장(통계 View)을 이해하기 위한 기초 구조를 정리합니다.

### 4.1 4계층 구조

PostgreSQL은 위에서 아래로 4계층 구조를 가집니다.

```
PostgreSQL Instance (서버, 포트 1개)
├── Database (한 인스턴스에 여러 개 가능)
│   ├── Schema (한 데이터베이스에 여러 개, 기본은 public)
│   │   ├── Table
│   │   ├── View
│   │   ├── Function
│   │   └── ...
│   └── (다른 스키마)
├── Database 2
└── ...
```

예시:

```
PostgreSQL Instance (localhost:5432)
├── core_db
│   └── public
│       ├── unit
│       ├── security_log
│       ├── transfer_history
│       ├── job_history
│       └── unit_location_history
├── analytics_db
│   └── public
│       └── ...
└── postgres (관리용 기본 DB)
```

### 4.2 데이터베이스 격리 원칙

데이터베이스끼리는 서로 격리되어 있습니다.

- 한 데이터베이스에 접속한 Session은 다른 데이터베이스의 Table을 직접 SELECT 할 수 없음
- 다른 데이터베이스 접근이 필요하면 별도 확장 모듈 필요 (`dblink`, `postgres_fdw`)
- 권한 관리도 데이터베이스 단위로 분리

이 격리는 보안과 권한 관리 측면에서 의도된 설계입니다.

### 4.3 일반적인 패턴

- **단일 데이터베이스**: 한 시스템이 하나의 데이터베이스를 사용. Core는 이 패턴 (`core_db` 하나)
- **여러 데이터베이스 공유**: 같은 PostgreSQL 인스턴스를 여러 시스템이 공유할 때 각자 데이터베이스 분리
- **환경 분리**: 개발/스테이징/운영은 보통 인스턴스 자체를 분리 (같은 인스턴스에서 데이터베이스만 분리하지 않음)

## 5. PostgreSQL 통계 View

> 이 장은 PostgreSQL이 제공하는 부하 측정 통계 View를 정리합니다.

### 5.1 통계 View의 범위 차이

PostgreSQL의 통계 View는 두 종류로 나뉩니다.

| 종류 | 범위 | 어디서 조회해도 |
|---|---|---|
| **인스턴스 전체 차원** | 모든 데이터베이스 정보 | 같은 결과 |
| **데이터베이스 내부 차원** | 접속한 데이터베이스 정보만 | 접속에 따라 다른 결과 |

### 5.2 `pg_stat_database` (인스턴스 전체 차원)

데이터베이스 단위 통계. 어느 데이터베이스에 접속해서 조회해도 모든 데이터베이스 정보가 나옵니다.

```sql
SELECT
    datname,
    xact_commit,
    xact_rollback,
    blks_read,
    blks_hit,
    deadlocks,
    numbackends
FROM pg_stat_database
WHERE datname IS NOT NULL;
```

주요 컬럼:

- `datname`: 데이터베이스 이름
- `xact_commit`/`xact_rollback`: Commit/롤백 누적 횟수
- `blks_read`: 디스크에서 읽은 블록 수
- `blks_hit`: Cache(shared buffers)에서 가져온 블록 수 → `blks_hit / (blks_read + blks_hit)`이 Cache 적중률
- `deadlocks`: Deadlock 발생 누적 횟수
- `numbackends`: 현재 활성 연결 수

### 5.3 `pg_stat_user_tables` (데이터베이스 내부 차원)

Table별 조회/수정/삽입/삭제 횟수. **접속한 데이터베이스의 Table만** 보입니다.

```sql
SELECT
    relname,
    seq_scan,
    idx_scan,
    n_tup_ins,
    n_tup_upd,
    n_tup_del,
    n_live_tup
FROM pg_stat_user_tables
ORDER BY (seq_scan + idx_scan) DESC;
```

주요 컬럼:

- `relname`: Table 이름
- `seq_scan`: 풀Scan(시퀀셜 Scan) 횟수 → 많으면 인덱스 누락 의심
- `idx_scan`: 인덱스 Scan 횟수
- `n_tup_ins`/`n_tup_upd`/`n_tup_del`: INSERT/UPDATE/DELETE 누적 행 수
- `n_live_tup`: 현재 활성 행 수 (대략적인 행 카운트)

다른 데이터베이스의 정보를 보려면 그 데이터베이스로 접속을 바꿔야 합니다.

```bash
psql -d core_db -c "SELECT relname, seq_scan FROM pg_stat_user_tables;"
psql -d analytics_db -c "SELECT relname, seq_scan FROM pg_stat_user_tables;"
```

### 5.4 `pg_stat_statements` (확장 모듈, 예외적으로 인스턴스 전체)

쿼리별 누적 통계. 가장 강력한 부하 분석 도구입니다.

**※ 활성화 방법**

`postgresql.conf`에 한 줄 추가하고 재시작:

```
shared_preload_libraries = 'pg_stat_statements'
```

그 후 한 데이터베이스에서 확장 설치:

```sql
CREATE EXTENSION pg_stat_statements;
```

**※ 특이점 — 데이터베이스 범위**

`pg_stat_statements`는 다른 통계 View와 다르게, **확장은 데이터베이스 단위로 설치되지만 저장되는 통계는 인스턴스 전체의 모든 쿼리**입니다.

- `core_db`에 확장을 설치하면, `core_db`에 접속해서 `pg_stat_statements` View를 조회할 수 있음
- 그 View에는 `core_db` 쿼리뿐 아니라 `analytics_db` 등 다른 데이터베이스에서 실행된 쿼리도 모두 포함됨
- `dbid` 컬럼으로 어느 데이터베이스의 쿼리인지 구분 가능

**※ 활용 예시 — 가장 시간 많이 쓴 쿼리 Top 10 (데이터베이스 이름과 함께)**

```sql
SELECT
    d.datname,
    s.query,
    s.calls,
    s.total_exec_time,
    s.mean_exec_time,
    s.rows
FROM pg_stat_statements s
JOIN pg_database d ON s.dbid = d.oid
ORDER BY s.total_exec_time DESC
LIMIT 10;
```

**※ 활용 예시 — 가장 자주 호출된 쿼리 Top 10**

```sql
SELECT
    d.datname,
    s.query,
    s.calls,
    s.mean_exec_time
FROM pg_stat_statements s
JOIN pg_database d ON s.dbid = d.oid
ORDER BY s.calls DESC
LIMIT 10;
```

이 두 쿼리 결과를 한 달에 한 번 정도 확인하면 1단계 측정으로 충분합니다.

### 5.5 통계 초기화

통계는 누적값이라 의미 있는 분석을 위해서는 가끔 초기화가 필요합니다.

```sql
-- pg_stat_statements 통계 초기화
SELECT pg_stat_statements_reset();

-- pg_stat_database, pg_stat_user_tables 등 초기화
SELECT pg_stat_reset();
```

운영 변화가 큰 시점(배포, 설정 변경 등) 전후로 초기화하면 변화 영향을 측정하기 쉽습니다.

## 6. Core 시스템 적용 결정

> 이 장은 위 원칙을 Core 시스템에 어떻게 적용할지에 대한 구체적 결정을 정리합니다.

### 6.1 데이터 항목별 권위 위치 결정

| 데이터                                                | 권위 위치                       | Cache           | 근거                                                                            |
|-------------------------------------------------------|---------------------------------|----------------|---------------------------------------------------------------------------------|
| Slot 점유 상태                                         | InMemory                        | —              | 초당 다수 변경, 재시작 시 Event로 재생성                                         |
| AMMR 위치/Battery/상태                                  | InMemory                        | —              | 실시간 변동, Event로 재생성                                                      |
| TransferList·AmmrList                                  | InMemory (State Service 권위)    | —              | 실시간 변동, 우선순위 정렬·매칭 결정 때문에 InMemory 필수                            |
| Slot Block 현재 상태                                   | InMemory                        | —              | 실시간 변동. 이력은 보안 Log (Block 요약 Event)                                  |
| Unit 메타 정보                                         | DB                              | —              | 영속, 이력 추적 필요                                                              |

| 보안 Log                                              | DB                              | —              | 영속 이력 (Block 요약·사용자 변경 행위·자동 처리 행위)                              |
| Transfer 이력·Job 이력·Unit 위치 변경 이력              | DB                              | —              | 영속 이력                                                                         |

### 6.2 GM 폴링 데이터 처리

**※ 결정 (본체 SAD v1.0_d118 §5.1·§4.6 반영)**

- **권위 매체 분담**: 마스터 데이터 영속·이력 = **DB 권위** / 운영 시점 실시간 조회 = **State Service InMemory 적재**
- **흐름**: Core 기동 시 State Service가 DB로부터 활성 제품·발행 Unit 일체를 InMemory에 적재. 이후 GM 1시간 주기 폴링 atomic 안에서 **InMemory와 DB 양쪽 갱신** (State Service 단일 게이트웨이 방식).

**※ 근거**

- 실시간 운영 (QR 발행·우선순위 계산·Transfer 매칭 등) 조회는 InMemory → 1ms 미만 응답 보장
- 영속·이력은 DB → 재시작 시 자연 복구 가능
- 두 매체를 같은 atomic 안에서 갱신하는 방식으로 정합 유지 (State Service 단일 게이트웨이 → 이중 권위 구조 없음)
- Cache 구조 아님 — "이중 권위가 아니라 권위 매체 분담" 구조

**※ 폴링 시점 흐름 예시 (본체 기준)**

```
[1시간 타이머 Trigger — State Service 입력 Channel 경유]
1. GM 수주/제품 마스터 Endpoint 호출 → 응답 받음
2. 응답 파싱 → 마스터 목록 추출
3. State Service atomic 안에서:
   3-1. InMemory 적재본과 Diff 비교
   3-2. DB UPDATE / INSERT / DELETE 수행 (영속·이력)
   3-3. InMemory 적재본 갱신 (운영 시점 조회용)
   3-4. 도메인 Event 발행 (구독 서비스에 변경 알림)
4. 새 제품에 대해 GM Recipe Endpoint 호출 → 같은 atomic 처리로 InMemory·DB 일괄 넣음
5. 끝
```

위 흐름에서 두 매체 (InMemory·DB)가 같은 atomic 안에서 갱신되며, State Service가 단일 게이트웨이로 정합을 보장합니다. 다른 서비스(Transfer Service·AMMR Service 등)는 State Service가 노출하는 read-only 참조로 InMemory 마스터 데이터를 직접 조회합니다.

### 6.3 1단계 측정 토대 (운영 시작 전 준비)

운영 시작 전에 다음 두 가지를 코드/설정에 넣어둡니다.

**※ 토대 1 — EF Core interceptor로 DB 호출 시간 자동 Logging**

EF Core의 `DbCommandInterceptor`를 구현하여 모든 쿼리 실행 시간을 Serilog에 구조화 Logging. 17장 애플리케이션 Log에 자동 기록되어 100일 보존.

**※ 토대 2 — `pg_stat_statements` 활성화**

`postgresql.conf`에 `shared_preload_libraries = 'pg_stat_statements'` 추가 후 PostgreSQL 재시작. `core_db`에서 `CREATE EXTENSION pg_stat_statements;` 실행.

이후 운영 중 한 달에 한 번 5.4의 Top 10 쿼리를 확인하여 부담의 원인을 점검합니다.

### 6.4 향후 Cache 도입 판단 기준

다음 조건이 모두 충족되면 Cache 도입을 검토합니다.

- 1단계 측정 결과 특정 쿼리가 명백한 부담으로 드러남 (CPU 사용률 또는 응답 지연)
- 그 쿼리가 GM 폴링 데이터처럼 변경 빈도가 낮은 데이터를 대상으로 함
- Cache 도입의 4가지 비용(2.2 참고)을 감수할 가치가 있다는 판단

Cache는 가장 부담 큰 조회 패턴 한두 개에만 적용하고, 전면적인 Cache는 피합니다.
